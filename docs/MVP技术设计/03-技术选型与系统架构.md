# 03 · 技术选型与系统架构

## 1. 部署与产品优先级

| 优先级 | 目标 | 说明 |
|--------|------|------|
| P0 | **独立站**（Vercel） | 先上线可玩 + 简易后台 |
| P1 | 经济/广告闭环 | 碎片、皮肤、激励 |
| P2 | 门户 H5 瘦包 | 无后台、无自建远程遥测 |
| 预留 | App（Capacitor） | 一期只预留端口，不上架 |

玩家账号登录：**本期不做**。简易后台可用环境开关 + 可选访问密钥保护。

## 2. 技术选型（定稿）

| 层 | 选型 | 理由 | 禁止 |
|----|------|------|------|
| 语言 | TypeScript strict | 规则/AI 可测 | — |
| Web | Next.js App Router | 贴 Vercel；路由分玩家/admin | 对局 AI 放 Server Route |
| UI | React + Tailwind | 对齐界面规格 | 在组件内写吃子判定 |
| 棋盘 | SVG（线 + 交点 hit） | 36 点轻量 | Phaser / Unity / Godot |
| 逻辑 | 纯 TS `packages/game-core` | 多端复用 | core 依赖 DOM/Next |
| UI 状态 | zustand（仅壳） | 会话视图 | core 依赖 zustand |
| 存档 | localStorage + schemaVersion | 无账号 | 无迁移的破坏性改结构 |
| 后台 | `/admin` 客户端读配置 + 调 core | 与线上同一 AI | 后台写库改配置（本期） |
| 测试 | Vitest 测 core | 回归 | 无单测合并规则改动 |
| 包管理 | pnpm | Vercel 友好 | — |

## 3. 分层架构

```
┌─────────────────────────────────────────┐
│  Shells: 独立站 Next / 门户瘦包 / App预留 │
├─────────────────────────────────────────┤
│  Adapters: IStorage / IAds / IAdminGate   │
├─────────────────────────────────────────┤
│  App UI: 玩家页 + /admin 工具台           │
├─────────────────────────────────────────┤
│  game-core（纯 TS，零 DOM）               │
│   rules | fsm | ai | content(只读配置)    │
└─────────────────────────────────────────┘
```

**原则**

1. 单一真相源：玩法见 `docs/游戏创意/`；实现见本目录 04–05。  
2. 依赖倒置：壳依赖端口；core 不依赖壳。  
3. 后台与玩家共用 `pickSheepAction` / `applyAction`，禁止 admin 另写一套 AI。

## 5. 模块职责

## 6. 软件工程 DoD（编码期）

- 规则变更必须带 Vitest  
- 新增皮肤必须通过 Catalog 校验（狼套含羊资源）  
- portal 构建不含 `/admin`  
- 禁止 core 出现 `window` / `document` / `fetch`（存档除外走 adapter）
