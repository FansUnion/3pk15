# Fangrush 工程边界总览

> 本文只维护长期有效的模块职责、依赖方向和平台架构。具体字段、页面、环境变量和构建参数由代码、测试和仓库脚本维护。

## 一、最终结构

```text
packages/game-core                 规则、AI、关卡、奖励、SaveGame
packages/platform-capabilities     账号、存储、广告、支付、生命周期契约
packages/api-contracts             自有 API 请求/响应 schema
packages/player-runtime            跨 Player/Admin/Mobile 的玩家运行时
packages/web-shared                Web 专用 locale、样式和基础契约
packages/web-assets                品牌、皮肤、棋盘、声音和资源门禁

apps/player-web                    独立站与 Poki/CG H5 壳和 adapter
apps/mobile                        Capacitor、Android/iOS 壳和原生 adapter
apps/fangrush-api                  自建认证、云存档、权益、购买、隐私 API
apps/admin                         关卡、AI、验收和内部管理工具

supabase/                          PostgreSQL migrations/config/seed
distribution/                      渠道提交材料与合规证据
```

所有载体共享同一 `game-core`。平台差异只能进入应用外壳或平台适配层，不能复制或分叉规则、AI和关卡配置。

## 二、模块职责

| 区域 | 唯一职责 | 不应包含 |
|---|---|---|
| `packages/game-core` | 规则、AI、关卡、对局状态和奖励模型 | 浏览器API、页面、平台SDK |
| `packages/platform-capabilities` | 跨终端能力接口、结果语义、target矩阵与Noop/测试实现 | 具体平台SDK和UI |
| `packages/api-contracts` | 版本化API输入、输出、错误码与校验schema | 数据库连接和服务实现 |
| `packages/player-runtime` | 玩家对局UI、store、进度协调、音频和通用玩家文案 | Next路由、原生SDK、具体渠道adapter |
| `packages/web-shared` | Web专用locale、共享基础样式和Web契约 | 移动平台契约、游戏规则和具体页面流程 |
| `packages/web-assets` | 共享运行资源及资源检查 | 页面、平台密钥、第二份资源目录 |
| `apps/player-web` | Next独立站、Poki/CG静态壳、Web账号与平台adapter | Admin实现、移动原生SDK、第二套运行时 |
| `apps/mobile` | Capacitor壳、Android/iOS工程、原生登录/存储/广告/IAPadapter | Next路由、门户SDK、第二套规则/UI |
| `apps/fangrush-api` | 自建身份会话、云存档、权益、购买校验和隐私API | 玩家页面、Supabase Auth、客户端规则判定 |
| `apps/admin` | 关卡台、AI诊断、Admin试玩和验收 | 玩家正式进度、第二套玩家运行时 |
| `supabase/` | PostgreSQL schema、migration、seed与本地配置 | Supabase Auth业务、TypeScript服务逻辑 |
| `distribution/` | 构建说明、合规、表单和提交素材 | 运行时游戏逻辑 |

## 三、依赖方向

```text
player-runtime ──> game-core / platform-capabilities / api-contracts
player-web     ──> player-runtime / web-shared / web-assets
mobile         ──> player-runtime / platform-capabilities / api-contracts / web-assets
admin          ──> player-runtime / game-core / web-shared / web-assets
fangrush-api   ──> api-contracts / PostgreSQL

任意 apps/A -X-> apps/B 源码
任意客户端  -X-> Supabase数据库
```

Admin通过`player-runtime`复用棋盘、对局store、帮助和声音，并通过Admin模式关闭奖励、解锁和正式玩家进度写入。应用之间禁止源码import；公共能力达到复用门槛时进入package，不通过webpack alias长期维持隐式依赖。

## 四、部署与平台边界

- 玩家端与Admin来自同一monorepo，但作为两个独立项目构建、部署、配置域名和访问权限。
- 独立站是使用自有账号/云存档、广告固定为空实现的玩家入口。
- Poki与CrazyGames共享玩家产品，但分别通过平台适配器处理加载、暂停、静音、广告和平台数据。
- `build:poki`、`build:crazygames`只表示目标构建边界，不代表真实SDK、平台审核或广告已经接入。
- Admin不进入玩家、H5或App的公开构建产物。
- App使用同一`apps/mobile`维护Android/iOS，共享React壳，差异留在原生目录和adapter。
- 自有API独立部署；Supabase只作为PostgreSQL，客户端不使用Supabase Auth或直接数据库访问。
- 平台构建、提交字段、素材和合规证据统一维护在 `distribution/`。

## 五、平台服务适配

广告、生命周期、分享、支付、登录和云存档都必须通过能力接口接入：

- 不支持的能力明确返回不可用，界面隐藏或友好降级；
- 广告失败不能吞掉基础奖励或破坏对局；
- 平台暂停和静音必须同时影响输入、计时和音频；
- 密钥、签名、合同和支付凭据不进入Git；
- Mock只用于开发和验收，不能伪装成真实平台通过。

## 六、App技术方向

当前推荐在数据证明值得投入后采用 **Capacitor + 独立移动外壳**：

- 最大化复用现有React界面和纯 `game-core`；
- 原生广告、IAP、推送、登录和生命周期通过插件隔离；
- Web、H5和App继续共享规则与内容；
- 若未来出现明确性能或复杂原生体验瓶颈，再评估React Native、Flutter或原生重写，不提前维护第二套UI。

账号与云存档不是商店审核硬前置，但本项目已经确定独立站、Android、iOS共享自有账号和云存档。先在独立站验证服务端，再复用到App；首个移动技术样机可以暂用Noop账号/云以验证壳和生命周期。

## 七、变更原则

1. 先修改代码与自动测试，再同步本文中的长期职责变化。
2. 页面、字段、环境变量和命令变化不要求逐项写入本文。
3. 新平台接入不得修改核心规则来迎合SDK。
4. 每次发布检查核心测试、资源、应用边界、生产构建和公开产物。
5. 架构文档不维护当前任务状态；发行目录不维护运行时逻辑；产品文档不复制代码实现。
