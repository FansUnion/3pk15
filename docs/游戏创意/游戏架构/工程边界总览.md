# Fangrush 工程边界总览

> 本文只维护长期有效的模块职责、依赖方向和平台架构。具体字段、页面、环境变量和构建参数由代码、测试和仓库脚本维护。

## 一、最终结构

```text
packages/game-core
  规则、合法行动、终局、AI、关卡、奖励结算
          │
          ├── packages/web-shared
          │     跨应用契约和共享基础样式
          ├── packages/web-assets
          │     品牌、皮肤、棋盘、声音和资源门禁
          ├── apps/player-web
          │     独立站与H5玩家产品
          ├── apps/admin
          │     关卡、AI、验收和内部管理工具
          └── future apps/mobile
                Android/iOS外壳与原生平台能力
```

所有载体共享同一 `game-core`。平台差异只能进入应用外壳或平台适配层，不能复制或分叉规则、AI和关卡配置。

## 二、模块职责

| 区域 | 唯一职责 | 不应包含 |
|---|---|---|
| `packages/game-core` | 规则、AI、关卡、对局状态和奖励模型 | 浏览器API、页面、平台SDK |
| `packages/web-shared` | 跨应用契约与共享基础样式 | 游戏规则和具体页面流程 |
| `packages/web-assets` | 共享运行资源及资源检查 | 页面、平台密钥、第二份资源目录 |
| `apps/player-web` | 玩家页面、输入、会话、存档和平台适配 | Admin实现、第二套规则 |
| `apps/admin` | 关卡台、AI诊断、Admin试玩和验收 | 玩家正式进度、第二套棋盘规则 |
| `distribution/` | 构建说明、合规、表单和提交素材 | 运行时游戏逻辑 |
| 未来 `apps/mobile` | 移动壳、原生生命周期、广告、IAP和登录桥接 | 复制Web或核心规则 |
| 未来服务端 | 账号、云存档、统计和运营数据 | 客户端本地规则判定 |

## 三、依赖方向

```text
player-web ──> game-core / web-shared / web-assets
admin      ──> game-core / web-shared / web-assets
admin      ──> player-web中经过验证的对局运行时（受控单向复用）
player-web -X-> admin
```

Admin可以复用成熟的棋盘、对局store、帮助和声音能力，并通过Admin模式关闭奖励、解锁和正式玩家进度写入。Player不得引用Admin。只有共享运行时持续扩大并造成实际维护问题时，才抽出新的公共运行包。

## 四、部署与平台边界

- 玩家端与Admin来自同一monorepo，但作为两个独立项目构建、部署、配置域名和访问权限。
- 独立站是无平台广告的公开演示和玩家入口。
- Poki与CrazyGames共享玩家产品，但分别通过平台适配器处理加载、暂停、静音、广告和平台数据。
- `build:poki`、`build:crazygames`只表示目标构建边界，不代表真实SDK、平台审核或广告已经接入。
- Admin不进入玩家、H5或未来App的公开构建产物。
- 平台构建、提交字段、素材和合规证据统一维护在 `distribution/`。

## 五、平台服务适配

广告、生命周期、分享、支付、登录和云存档都必须通过能力接口接入：

- 不支持的能力明确返回不可用，界面隐藏或友好降级；
- 广告失败不能吞掉基础奖励或破坏对局；
- 平台暂停和静音必须同时影响输入、计时和音频；
- 密钥、签名、合同和支付凭据不进入Git；
- Mock只用于开发和验收，不能伪装成真实平台通过。

## 六、App技术方向

当前推荐在数据证明值得投入后采用 **Capacitor + 独立移动外壳**：

- 最大化复用现有React界面和纯 `game-core`；
- 原生广告、IAP、推送、登录和生命周期通过插件隔离；
- Web、H5和App继续共享规则与内容；
- 若未来出现明确性能或复杂原生体验瓶颈，再评估React Native、Flutter或原生重写，不提前维护第二套UI。

账号与云存档不是App上架硬前置。商店版本可以先使用本地存档，只有跨设备或多渠道身份需求成立时才引入服务端。

## 七、变更原则

1. 先修改代码与自动测试，再同步本文中的长期职责变化。
2. 页面、字段、环境变量和命令变化不要求逐项写入本文。
3. 新平台接入不得修改核心规则来迎合SDK。
4. 每次发布检查核心测试、资源、应用边界、生产构建和公开产物。
5. 架构文档不维护当前任务状态；发行目录不维护运行时逻辑；产品文档不复制代码实现。
