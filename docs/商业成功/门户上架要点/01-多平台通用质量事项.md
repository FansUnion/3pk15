# 01 · 多平台通用质量事项

> **分层 A。** 借 Poki 严标准提炼；独立站 / CrazyGames / 其他 Web 渠道通常同样受益。

每条结构：**为什么重要 → 适用 / 不适用 → 要准备什么 / 关键口径 → 常见误区（如有）→ 细则链接**。

## 1. 内容与打磨（原创、Web-first、完成度）

### 为什么重要

策展型门户与玩家都会先看「值不值得点、点开后爽不爽」。原创差异、少 Bug、反馈到位，比堆功能更能过审、留住时长。粗糙移植感会直接伤害点击后转化与口碑。

### 适用 / 不适用

- **适用：** 所有要上 Web 门户或公开演示的游戏。  
- **对 Fangrush：** 棋盘策略不是热门「趋势跟风」品类，更要靠**清晰核心循环**（落子—吃子—胜负）与**可辨识主题**取胜；避免缩略图/角色看起来像现有狼羊类 IP 的克隆。

### 要准备什么 / 关键口径

- 核心循环一句话说清；前几分钟能「玩明白并想再来一局」。  
- **Web-first：** 为浏览器交互与短会话设计，而非未改的手游壳。  
- 视觉统一、操作有反馈、少致命 Bug。  
- 家庭向、全年龄友好；避开暴力色情赌博等不安全主题。  
- 借鉴品类可以，须有**可辨识的原创贡献**（规则、主题、UX），不是模板换皮。

### 常见误区

- 「数据好就能豁免质量/IP 问题」——策展站通常不行。  
- 「和某爆款很像更容易过」——高度克隆反而更容易拒。

## 2. 跨设备与控制（桌面 / 手机 / 平板）

### 为什么重要

Web 流量大量来自手机与平板。只在桌面能玩，等于主动丢掉大半受众，也难过多数门户的「全设备」基线。

### 适用 / 不适用

- **适用：** 所有 H5 / iframe 嵌入游戏。  
- **对 Fangrush：** 棋盘必须在触控下可读可点；桌面可用键鼠增强（快捷键、悬停提示），但**不能只有键鼠才能完成对局**。

### 要准备什么 / 关键口径

- 桌面、手机、平板均可完成一局。  
- 移动端：竖屏或横屏铺满；两者都支持更佳。  
- 触控与键鼠各自合理：触控目标够大；键盘游戏提供暂停习惯（如 ESC）。  
- 平板按**移动操控方案**对待，勿假设一定有鼠标。  
- 尽量提供替代输入（纯点选、可改键等），扩大无障碍受众。

### 常见误区

- 「平板当桌面测一下就行」——平板更接近触控布局。  
- 小屏上棋盘挤成一团、按钮叠层导致点错。

## 3. 进入与引导（少菜单、图示教程、可跳过）

### 为什么重要

门户玩家耐心极短。菜单墙、不可跳过的长说明、纯文字教程，会直接拉低「点进页 → 真正开始玩」的转化，并在试玩录像里表现为一分钟内流失。

### 适用 / 不适用

- **适用：** 所有短会话 Web 游戏。  
- **对 Fangrush：** 规则有一定门槛，更应用**图示 + 短句**；避免开场长剧情或必须读完的规则墙。赛季/文案可以后置。

### 要准备什么 / 关键口径

- 尽量少屏直接进入可玩状态。  
- 教程偏视觉（高亮棋盘、示范一步），少大段文字。  
- 过场、长说明、开场动画**均可跳过**。  
- 桌面可显示键位提示；移动显示触控相关提示，且互不干扰。

### 常见误区

- 把「完整规则手册」塞进首次会话。  
- 强制观看过场才能点第一子。

## 4. 存档与隐私稳健（无痕可玩、有进度则存）

### 为什么重要

隐私模式、存储配额失败、第三方 cookie 策略都会让 `localStorage` 不可用。若因此白屏或无法开玩，玩家会立刻离开。有进度却不保存、又不说明，会伤害信任。

### 适用 / 不适用

- **适用：** 任何依赖浏览器本地存储的 Web 游戏。  
- **对 Fangrush：** 关卡进度、设置、皮肤偏好若依赖本地存储，须在存储失败时仍能**开一局**；若某模式不存档，须在 UI 说清楚。

### 要准备什么 / 关键口径

- 无痕 / 存储失败时仍能进入核心玩法（包装读写、失败降级）。  
- **有进度则保存**；否则明确告知「退出不保留」。  
- 不要把「必须登录才能玩」当成默认（门户包尤其如此；账号能力见 Poki 专属 B5）。

### 常见误区

- 未捕获存储异常 → 整页崩溃。  
- 假装有云存档，实际一清缓存就没。

### 细则链接

## 5. 广告产品规则（抽象，跨门户通用纪律）

### 为什么重要

广告是多数门户的主变现。糟糕的插播时机、锁死核心玩法、恐吓式「关广告拦截」提示，会伤害体验，也违反多数平台政策。把纪律做成产品规则，换平台时只需换 SDK，不必重做体验。

### 适用 / 不适用

- **适用：** 任何走平台广告分成的包（Poki、CrazyGames 等）。  
- **不适用 / 弱化：** 独立站若明确做成无广告 Demo，可无插播，但仍应保留「将来可插自然断点」的产品结构。  
- **对 Fangrush：** 自然断点优先在**局间、重开、回菜单再继续**；勿在思考落子中途硬插。激励（若有）必须可跳过、失败不发奖。

### 要准备什么 / 关键口径

- **插屏：** 放在自然断点；回到玩法前触发；**频控交给平台**，勿自建「每 N 秒必出」计时器。  
- **激励：** 先说清奖励；**仅成功观看后发奖**；始终提供**普通继续**路径；激励与继续同时可见；继续按钮视觉层级不低于激励（常见要求：继续偏绿、激励带影片图标且非绿色）。  
- **广告播放期：** 游戏静音、停输入；不在此期间打「玩法中」类分析事件（平台 SDK 各有规定）。  
- **不锁核心玩法**在广告或去广告付费之后。  
- **不做**自定义「检测到广告拦截」恐吓 UI；有拦截时静默处理、激励不发奖即可。  
- 门户包：**不要第二套广告网**、不要自建去广告内购入口。

### 常见误区

- 一奖励叠多条激励视频。  
- 只有激励按钮、没有普通继续。  
- 广告中仍响应空格/方向键，导致父页面滚动或误操作。

### 细则链接

- [双语 03 · Advertisement integration](../Poki官方文档双语整理/03-技术要求与外部请求.md)  
- 具体 API 名与时机 → [02 · B2](./02-Poki专属事项.md)

## 6. 门户包干净（无调试、无乱外链、无第二套变现）

### 为什么重要

门户默认会拦截或审查外部请求；调试入口、Admin、第二广告网、未批准追踪，轻则 Inspector 警告，重则拒审或运行时被拦导致白屏。

### 适用 / 不适用

- **适用：** 每一个「上传给平台」的构建产物。  
- **对 Fangrush：** Poki/CrazyGames 包与独立站构建分离心态——门户包无 Admin、无站内广告实验代码、无指向外部商业站的硬推外链。

### 要准备什么 / 关键口径

- 发布构建去掉开发工具、调试叠加层、测试关卡作弊入口。  
- 无第二广告网；无未批准的分析/字体/CDN 外链（平台规则以各站为准；Poki 默认禁几乎所有外链，且 **GA 一律禁**）。  
- 工作室 Logo 可在加载页；勿做启动闪屏外链跳转。  
- 若确有批准的外链/分析：游戏内可打开的隐私说明 + **自托管**隐私政策 URL（勿用临时网盘/未托管文档链）。

### 常见误区

- 开发包直接上传「也能玩」。  
- 用 Google Fonts / 公共 CDN「图省事」——在严平台上会踩雷。

### 细则链接

- [双语 03 · Privacy、Clean build、No branding/external ads](../Poki官方文档双语整理/03-技术要求与外部请求.md)

## 7. 性能与体积意识

### 为什么重要

加载慢、卡顿、首包过大，会直接打低「转化到真正开玩」和时长。门户玩家从分类页点进来，几秒无响应就会退出。

### 适用 / 不适用

- **适用：** 所有 Web 游戏。  
- **对 Fangrush：** 棋盘资源相对可控，仍要管首屏脚本/图集体积与交互帧率；合同向常见基线是中端机约 **≥30 FPS**（目标 60）——以各站合同为准。

### 要准备什么 / 关键口径

- 首屏与可玩之间的等待可控；大资源可后载，但勿阻塞「能点第一下」。  
- 去掉调试残留与未压缩资源。  
- 各门户可能有更硬的初始包目标（Poki 常见目标 **初始下载 &lt;8MB**，见专属篇）。  
- 图片与音频有优化意识（Inspector 类工具常会警告未优化图）。

### 常见误区

- 用「以后再压」把调试资源留进上架包。  
- 只在高配桌面测流畅度。

## 8. 文案与本地化（标杆细度）

### 为什么重要

语言越多，全球受众越广。对**文本型**游戏（剧情、说明、问答、挂机数值文案），本地化往往直接决定能不能在非英语市场玩下去。即便不是重度文本游戏，**英文本地化质量**仍影响门户默认评审与首印象。

### 适用 / 不适用

**适合大力本地化（ROI 通常更高）：**

- 剧情向、点击/挂机、问答、机制主要靠文字说明的游戏。

**其他类型：** 本地化仍有价值，但边际 ROI 可能较低，可按阶段加语言，不必一次上齐。

**对 Fangrush：**

- 棋盘策略以**图示 + 短文案**为主，不是典型「长剧情文本游」，但菜单、规则短句、赛季/活动文案仍需要专业英语。  
- **英文本地化优先**（门户与海外评审默认语言）。  
- 是否立刻上 EFIGS 全套或 CJK，按预算与 ROI 决策；结构上应**随时可加语言**，而不是写死在组件里。

### 要准备什么 / 关键口径

**准备**

- 可翻译文案**集中管理**（例如统一 JSON/文案表），勿散落在各组件硬编码。  
- UI 布局按**更长译文**预留（德/法/俄等常比英文长），避免截断与重叠。

**语言优先级（Poki 文档口径，可作多平台参考）**

1. **EFIGS**：English, French, Italian, German, Spanish（**可考虑土耳其语**）。  
2. **CJK**：简体中文、日语、韩语（成本高；有把握 ROI 再上）。  
3. **巴西葡萄牙语 & 俄语**：补全球覆盖。

**展示与默认行为**

- 设置/菜单中提供**语言切换**。  
- **默认**按浏览器语言选择，并在运行时做检测；用户手动选择后应记住偏好（在存储可用时）。

**与门户包的关系**

- 本地化对文本密集游戏在 Requirements 里常标为**建议**；不是「零语言也能过」的借口——至少英文要通顺、完整。

### 常见误区

- 只做机器翻译塞进 UI，不改版式 → 长语言爆版。  
- 字符串散落，导致漏翻、中英混杂。  
- 默认写死中文或写死英文、忽略浏览器语言。

## 9. 第一印象素材（缩略图 / 封面）

### 为什么重要

分类页上玩家先看图再决定点不点。图不清晰、文字堆满、与实机不符，会伤害点击率；点进去发现「货不对板」会伤害停留与转化。

### 适用 / 不适用

- **适用：** 一切商店/门户/商店页有瓷砖图的渠道。  
- **对 Fangrush：** 小尺寸仍能认出棋盘/三狼主题；表现真实玩法，避免纯 AI 拼贴风与实机割裂。

### 要准备什么 / 关键口径

- 小尺寸可读：主体大、背景干净、少字或无字。  
- 与实机视觉一致（默认皮肤/典型局面）。  
- 强对比，避免融进门户背景色。  
- 各站还有硬规格（Poki：满版方形、≥628px 等）——见专属篇；**「真实玩法 + 小图可读」是跨站通用纪律**。

### 常见误区

- 缩略图写满游戏名/口号（小图不可读）。  
- 用夸张宣传图吸引点击，进游却是另一套美术。

## 10. AI 素材纪律

### 为什么重要

门户鼓励审慎用 AI，但拒「捷径感」：水印、提示词残字、风格拼盘、未试玩的 AI 关卡/文案，会在质量审核与 Fit 测试中暴露。纯 AI、无打磨还可能带来 IP 归属风险。

### 适用 / 不适用

- **适用：** 使用 AI 辅助美术、文案、关卡、代码的所有项目。  
- **对 Fangrush：** AI 可用于草稿与提速，但上架前须人工统一风格、去水印、试玩关卡；狼羊主题尤其注意不要提示出接近知名 IP 的外形。

### 要准备什么 / 关键口径

- **黄金法则：** AI 赋能，不取代打磨与原创。  
- 提交前：视觉干净一致；无水印/提示词；对话经人改；**玩法必须测过**。  
- 能说明工具与迭代流程（若被问到 IP/流程）。  
- 进严肃流量测试前，须有可辨识的独特视角或实质改进，而非模板直出。

### 常见误区

- 缩略图一种 AI 风、局内另一种风。  
- 「生成了就上架」未做设计向 QA。
