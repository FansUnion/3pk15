#### 为什么重要

Poki for Developers（P4D）与试玩工具处于**限量准入**。没有后台，就无法上传版本、跑 Playtest / Fit、进正式漏斗。

#### 要准备什么 / 关键口径

- 通过官方 early access / 提交入口申请。  
- 审查看重：质量与手感、是否契合 Poki 受众、跨端技术扎实度。  
- **并非每份申请都有回复**；合适才会联系。  
- 获准后硬门槛：满足 Requirements + 集成 Poki SDK。

#### 常见误区

### B1.3 Playtesting → Player Fit → Web Fit → Final Review

#### 为什么重要

这是 Poki 的**数据关**：没有录像与样本数据，就不知道引导/难度是否成立；不过 Fit **不能**进终审与正式发布。

#### 适用 / 不适用

- **适用：** 所有要在 Poki 发布的游戏。  
- **对 Fangrush：** 棋类节奏偏慢，更要盯「前几分钟是否懂规则、是否想再开一局」——直方图前两柱堆积通常是引导问题。

#### 要准备什么 / 关键口径

**五关概览**

1. 添加游戏  
2. Feedback & **Playtesting**（约 10 条真实玩家录像/轮）  
3. **Player Fit Test**（约 500 人时长分布）  
4. **Web Fit Test**（约 1 万曝光量级的分类页测试）  
5. **Final Poki Review**

**Playtesting**

- 用于查 Bug、测引导、调难度、验证核心乐趣；各阶段都可反复申请。  
- 申请 Player Fit 前须在 Level 2 **观看至少 5 条**录像。

**Player Fit → 进入 Web Fit 的门槛（须同时满足）**

1. **平均总游玩时长 &gt; 3 分钟**  
2. **500 次会话中至少 25% 超过 3 分钟**

- 每日请求次数有上限（文档口径约 **2 次/开发者/日**）。  
- 右侧柱（尤其 5m+）越高越好；人前两柱 → 改引导/前期。

**Web Fit**

- 解锁：看过 ≥5 条 Playtest、**通过 Player Fit**、已上传**静态缩略图**。  
- 约 **3–5 天**；**开始后不可中止**；中途可申请换包。  
- 此阶段广告事件**非强制**；若已接，多为占位、**无收入分成**。  
- **三项同等重要指标**（相对所测分类均值评 0–5 分，**≥3** 约等于达均值）：  
  - **CTR** — 分类页点击缩略图比例  
  - **Average Time on Page** — 游戏页停留  
  - **C2P（Conversion to Play）** — 到达首次 `gameplayStart()` 的比例（加载+引导）

**未过 Web Fit**

- 不代表游戏「彻底不行」，而是尚未适配该平台；**必须过 Web Fit 才能终审**。  
- CTR 低 → 缩略图；停留短 → 回 Playtest/Player Fit；C2P 低 → 加载/体积/精简进入。

**Final Review**

- 约 **1–2 周**；看测试结果 + 质量指南 + 分类内独特性。  
- 通过 → 签约与发布流程；未通过 → 当前不能在 Poki 发布（可日后再提其他作品）。

#### 常见误区

- 没看录像就刷 Player Fit。  
- 把 Web Fit「看起来像上架」当成已全球发布并开始算广告收入。

## B2 · SDK 与运行时硬约束

### B2.1 生命周期事件：`init` / `gameLoadingFinished` / `gameplayStart|Stop`

#### 为什么重要

后台的参与度、C2P、广告机会都依赖事件时机。时机错了，Fit 数据失真，发布后指标也会坏。

#### 适用 / 不适用

- **适用：** 所有 Poki HTML5 集成。  
- **对 Fangrush：** `gameplayStart` 应对「玩家开始交互/对局进行中」，不是页面 load 完。

#### 要准备什么 / 关键口径

- `PokiSDK.init()`：成功或失败都**继续加载游戏**。  
- `gameLoadingFinished()`：加载完成时打一次（服务 C2P）。  
- `gameplayStart()`：**首次玩家输入**/进入可玩交互时；禁止 load 时误打。  
- `gameplayStop()`：暂停、菜单、关卡结束、过场、死亡等玩法中断。  
- **禁止**同类事件连发（Start 后立刻再 Start 等）。  
- **广告播放期间禁止**任何 SDK 游戏事件。

### B2.2 `commercialBreak` / `rewardedBreak` 与广告期行为

#### 为什么重要

这是 Poki 变现与政策的核心 API。时机错误会被 QA 打回；广告期不静音/不禁输入会破坏体验与嵌入页安全。

#### 适用 / 不适用

- **适用：** 正式发布路径上的变现（Web Fit 阶段可不接，但最终发行需要正确集成）。  
- **对 Fangrush：** 插播放在局间/重开等自然断点；激励若做「提示/撤销」类奖励，必须保留免费继续。

#### 要准备什么 / 关键口径

- `commercialBreak()`：自然断点、**退出暂停回到玩法**时；去选关等「离开玩法」路径不要当插播点（换装类有官方边缘特例，棋类一般不适用）。  
- 玩家继续玩前，实践上常在每次 `gameplayStart()` 前提供插播机会；**是否出广告由平台决定**，勿自建频控计时器。  
- `rewardedBreak()`：玩家主动选择；一奖励一视频；失败/拦截不发奖。  
- 广告期：**静音 + 禁用输入**；防空格/方向键把**父页面**滚走。  
- 激励 UI 层级见 [01 §5](./01-多平台通用质量事项.md)（绿继续、影片图标等）。

### B2.3 画布 16:9、初始包体、防父页滚动、平板移动操控

#### 为什么重要

嵌入页与分类流量假设游戏是**统一宽屏画布**；包体过大拖垮 C2P；滚动穿透与错误操控方案会在 Inspector/真机上直接暴露。

#### 适用 / 不适用

- **适用：** Poki Requirements 硬性项。  
- **对 Fangrush：** 棋盘 UI 需在 16:9 画布内完成布局；竖屏手机可能是 letterbox/适配，但文档要求按 16:9 规模化铺满画布——实现细节见工程侧，本处只记**平台口径**。

#### 要准备什么 / 关键口径

- 全设备 **16:9** 铺满游戏画布（参考比例尺寸如 640×360 等）。  
- 初始下载目标 **&lt; 8MB**。  
- 游戏内滚动/按键**不得带动父页面**滚动。  
- **平板强制移动端操控方案**。  
- 响应式在不同尺寸下一致可用。

### B2.4 外链默认禁止、GA 禁止、隐私声明例外

#### 为什么重要

#### 适用 / 不适用

- **适用：** 所有 Poki 上架包。  
- **对 Fangrush：** 门户包不要依赖 Google Fonts、公开 CDN、GA；需要分析走平台能力或获批例外。

#### 要准备什么 / 关键口径

- 发布前移除非必要外链资源。  
- 例外（多人外部服、个案批准的第三方分析）须：**向 Poki 申请 + 最新隐私声明 + 游戏内可访问 + 托管政策 URL**。

## B3 · 素材与发布运营

### B3.1 静态缩略图硬规格

#### 为什么重要

**开始 Web Fit 前必须上传**静态缩略图；CTR 主要看它。

#### 适用 / 不适用

- **适用：** 进 Web Fit 及之后所有阶段。  
- **对 Fangrush：** 满版方形、少字、表现真实棋盘对局瞬间。

#### 要准备什么 / 关键口径

- 全出血正方形，**≥ 628×628 px**；无额外边框/信箱黑边。  
- **避免文字/标题**；大小图块都清晰。  
- 强对比；注意勿贴近 Playground 背景色 **#83FFE7** 而「消失」。  
- 与实机一致；主体清晰、静态也要有动感。  
- **Web Fit 进行中**勿随意换图；通过后改图须支持团队审核；全球发布后仅重大内容更新时改，避免货不对板。

### B3.2 动画缩略图与 Soft Release A/B

#### 为什么重要

### B3.3 Technical / Soft / Global 与发布后

#### 为什么重要

过终审不等于「丢上首页不管」；Poki 用分阶段放量控风险，并持续用数据决定曝光。

#### 适用 / 不适用

- **适用：** 签约后的发行路径。  
- **对 Fangrush：** 软发布**不可跳过**；此阶段就要盯 C2P、错误、广告机会（文档口径关注约 **≥1 广告/DAU** 的插播+激励组合，以当时后台与对接人为准）。

#### 要准备什么 / 关键口径

- **Technical Test：** 可选；多人或新奇特技术可能被要求；在 New Games 等有限流量压测。  
- **Soft Release：** QA 后约 1–2 周排期；相关分类页；约 2–3 周；监控指标与反馈；可扩分类。  
- **Global Release：** 软发布优化后全量；本地化首页/分类/游戏页推广；约 2 周新游加权 + 缩略图火焰标。  
- **发布后：** 修 Bug/性能、加内容、加语言；指标变好约一周内反映在推荐里——**不是买量承诺**。

#### 适用 / 不适用

## B4 · 商务与收款

### B4.1 Web Exclusive vs Non-Exclusive；Discord / YouTube Playables

#### 为什么重要

独家换营销与分成包；非独家多为一次性授权费。签错模式会卡住双门户或社区渠道策略。

#### 适用 / 不适用

- **对 Fangrush / 本品策略：** 渠道路线通常强调**双门户、先非独家**——签约前必须和商务条款对齐，勿默认 Web Exclusive。  
- Steam / 移动商店 / 主机：网页独家**一般不限制**（以合同为准）。

#### 要准备什么 / 关键口径

- **Web Exclusive：** 开放网页仅 Poki；**Discord、YouTube Playables 算网页**，另行发布会冲突。  
- **年限待核：** 官方不同页出现 **5 年 / 7 年** 表述——**以合同与现行 Deal Types 为准**。  
- **Non-Exclusive：** 一次性 flat fee；无独家那套分成营销包。  
- 决策前对照 [渠道发展路线](../../渠道发展路线.md)。

#### 常见误区

- 以为「不上别的门户就行」，却忘了 Discord/YT Playables。  
- 未读合同就按营销页年限做排期。

### B4.2 分成口径摘要

#### 为什么重要

影响「自带流量 vs 平台导流」的预期与是否值得做社区导流。

#### 适用 / 不适用

- 主要与 **Web Exclusive** 分成模型相关；非独家以授权费为准。  
- **一律以合同为准。**

#### 要准备什么 / 关键口径（文档常见摘要）

- 玩家因**你**而来（书签、直接搜索、自有社区等）：开发者约 **100%**。  
- 经 **Poki.com / Poki 营销**而来：约 **50/50**。  
- 具体定义、排除项、结算周期以合同与财务条款为准。

### B4.4 合同向质量条款摘要

#### 为什么重要

除 SDK 文档外，合同可能写明 FPS、修 Bug 时限、设备覆盖等——违约风险在商务而非 Inspector。

- 近 3 年中端机、近 12–18 个月主流桌面浏览器等覆盖预期；高比例用户无严重问题。
