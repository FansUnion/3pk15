# Fangrush 渠道能力与广告商业化总方案

> 本文是独立站、H5门户、App与Admin在登录、存档、广告、IAP、统计、外链和打包边界上的产品规划唯一来源。渠道先后顺序见 [渠道发展路线](渠道发展路线.md)，当前执行状态见 [任务清单](../任务/任务清单.md)，平台提交规格与证据见 [`distribution`](../../distribution/README.md)。
>
> 平台SDK方法、广告频控、收入分成、隐私和审核规则会变化。实现和提交前必须重新核对官方英文文档、Developer Portal与合同；本文不承诺固定分成或eCPM。

## 1. 决策摘要

1. 独立站长期作为无广告Demo、品牌与试玩入口，不以首发收入为目标。
2. Poki与CrazyGames使用各自平台SDK和广告分成，不混入AdSense、移动广告SDK或自建支付。
3. H5首发只采用结算后的自愿激励广告；插屏必须等待真实留存与广告数据再决定。
4. App只在H5数据达到投入门槛后启动，采用“激励广告 + 低频局间插屏 + 去广告IAP + 少量皮肤IAP”的混合模式。
5. 广告永远不能替代基础通关奖励、阻断主线、改变战力或掩盖坏地图。
6. 游戏决定何时请求广告及奖励内容；平台或广告网络决定是否填充、是否实际展示和回调结果。
7. 规则、AI、关卡和奖励基础逻辑跨渠道统一，渠道差异只通过平台Adapter、构建配置和原生壳表达。

### 2.1 能力解释

- “可以有”不等于首发必须开发；账号、云存档、复杂统计和App能力仍受阶段门槛控制。
- 门户包应尽量瘦，不包含Admin、独立站统计、其他广告网络、未批准外链和不必要外部请求。
- 独立站Admin属于开发/运营工具，不应被理解为面向普通玩家的产品能力。
- 平台登录或云存档不能反向改变 `game-core` 的关卡、规则和奖励语义。

## 4. 广告控制权

广告曝光由游戏和平台共同决定：

```text
游戏到达自然停顿点
  -> 游戏请求商业广告或激励广告
  -> 平台SDK/广告网络判断资格、填充、频控和阻断状态
  -> 平台实际展示或返回失败/无填充/冷却
  -> 游戏可靠恢复输入与音频
  -> 只有激励广告成功完成才发放额外奖励
```

### 4.1 游戏负责

- 选择请求时机，不在核心操作中请求。
- 广告前明确说明奖励。
- 暂停输入、AI时序和音频，防止后台继续行动。
- 处理成功、失败、取消、无填充、AdBlock、冷却和重复回调。
- 激励成功后幂等发奖；失败时不吞基础奖励。
- 为不看广告或广告不可用的玩家保留正常继续路径。

### 4.2 平台或广告网络负责

- 当前是否允许展示及是否有广告填充。
- 平台级频控、地区、广告库存、AdBlock和隐私条件。
- 广告素材、播放流程和完成结果。
- 结算口径、收入分成和无效流量判断。

因此，Poki/CrazyGames不是“SDK自动在任何合适时机替我们投放”，也不是“我们调用一次就一定曝光”。游戏通常需要在合适节点调用SDK，平台保留最终展示决定权。具体方法、回调和频控以接入时官方文档为准。

## 5. 广告位置总表

| 游戏环节 | 激励广告 | 插屏/商业广告 | 决策 |
|---|---:|---:|---|
| 首次打开、品牌页 | 否 | 否 | 先让玩家看见产品和开玩入口 |
| 首次教学前后 | 否 | 否 | 不打断规则理解和首局转化 |
| 关卡说明页 | 否 | 否 | 保持目标、策略和开局信息清晰 |
| 选择狼、移动、捕食 | 否 | 禁止 | 核心操作不可中断 |
| 连吃决策 | 否 | 禁止 | 这是本品最重要的策略节点 |
| 羊AI行动 | 否 | 禁止 | 避免时序、恢复与误解问题 |
| 暂停/帮助/设置 | 否 | 否 | 玩家主动离开玩法，不应被二次打断 |
| 胜利结算 | 推荐 | 后续可评估 | 基础奖励先到账，再提供自愿加成 |
| 失败结算 | 后续评估 | 后续可评估 | 只允许提示/辅助型自愿奖励，不卖必胜 |
| 下一关之前 | 不重复 | 后续可评估 | 只有真实数据支持时加入低频局间广告 |
| 重复游玩若干局后 | 可用 | 后续可评估 | 必须受平台频控和留存保护约束 |
| 独立站Demo | 关闭 | 关闭 | 品牌与试玩优先 |

### 5.1 明确禁止

- 开局即广告、首局未完成前强制广告。
- 棋盘操作、连吃、AI行动中插入广告。
- 广告解锁基础主线或广告失败导致无法继续。
- 把基础通关奖励包装成“看广告才能领取”。
- 伪造关闭按钮、诱导误触或把广告按钮做成默认下一步。
- 同一结算连续展示激励与插屏。
- 用广告收益掩盖首局完成率、留存或玩家满意度下降。

## 6. 首发频控方案

### 6.1 独立站

- 正式环境广告次数固定为0。
- 开发环境保留Mock和Admin实验台。

### 6.2 Poki/CrazyGames

- 首发只开放胜利结算后的自愿激励广告。
- 同一局最多领取一次广告奖励，重复回调必须幂等。
- 当前不主动加入插屏；平台若要求商业节点，优先放在完整结算后、进入下一关前。
- 后续插屏候选基线：首局不展示、连续两局不重复、至少间隔2至3个完整对局；最终以平台频控和真实数据修订，不能作为写死政策。

### 6.3 App

- 第一版保留结算激励广告。
- 插屏只允许完整对局之间，首局和连续失败后不立即展示。
- 购买“去广告”后移除强制插屏，但是否保留玩家主动选择的激励广告需要在商品说明中明确。

## 7. 玩家为什么愿意看广告

玩家不会为了项目收入看广告，只会为了当前目标中可感知的价值看广告。有效激励需要满足：

1. 自愿：不看也能继续和获得基础奖励。
2. 清楚：广告前知道获得什么、持续多久。
3. 及时：成功完成后立即生效。
4. 有目标：奖励能明显推进下一件皮肤或近期收藏目标。
5. 有差额：玩家看得见当前持有、目标价格和还差多少。
6. 不破坏平衡：广告不能提供棋盘战力或必胜能力。

### 7.1 当前激励

当前实现为胜利后主动观看广告，激活30分钟碎片双倍。它的价值是减少重复刷关时间、加快皮肤解锁，不改变对局能力。

当前优势：

- 放在自然结算点。
- 只有成功播放才激活。
- 广告失败不影响基础奖励。
- 结算页已展示当前碎片、下一件皮肤成本和差额。

待验证假设：

- 30分钟是否符合玩家实际会话长度。
- 新玩家是否理解双倍影响后续对局，而不是当前奖励立即翻倍。
- 50/80通用碎片皮肤是否形成足够近的目标。
- 玩家是否愿意为了外观而重复游玩。

### 7.2 候选优化，不提前实施

- 将文案改得更具体，例如“未来30分钟通关碎片双倍”。
- 显示预计再通关多少次可兑换目标皮肤。
- 若数据证明30分钟难理解，再测试“本局奖励加倍”这种即时方案。
- 失败后广告提示仅在多次失败且现有三级指导不足时评估，不允许直接提供必胜或跳关。

## 8. App广告平台与IAP组合

App Store和Google Play不是广告SDK。App需要通过移动壳接入广告网络，并通过Google Play Billing或Apple StoreKit处理数字商品。

### 8.1 广告平台候选

| 方案 | 适用阶段 | 优点 | 成本/风险 |
|---|---|---|---|
| Google AdMob | App首个广告方案候选 | Android/iOS覆盖、资料多、适合先验证 | 账号审核、隐私、同意管理、SDK与填充差异 |
| AppLovin MAX | 需要聚合与优化后 | 聚合、竞价和运营工具较完整 | SDK更重、配置与运营复杂度更高 |
| Unity LevelPlay | 进入多网络聚合后 | 多网络与游戏场景经验 | 集成、隐私与依赖管理成本上升 |
| 单一其他广告网络 | 有明确地区/合同优势时 | 可能补充特定地区库存 | 不适合作为无数据时的首选 |

首个App建议先选择一种主方案，不同时接多套聚合。达到稳定DAU且单一网络填充或收益受限后，再评估MAX/LevelPlay等聚合。

### 8.2 推荐收入组合

| 方式 | 定位 | 第一版 |
|---|---|---:|
| 自愿激励广告 | 免费玩家主要变现 | 做 |
| 低频局间插屏 | 广覆盖补充 | 谨慎做 |
| 去广告IAP | 长期玩家体验与收入补充 | 做 |
| 少量付费皮肤/IAP皮肤包 | 收藏玩家补充 | 小规模做 |
| 订阅、Battle Pass | 需要持续运营内容 | 不做 |
| 付费解锁主线 | 与免费策略闯关定位冲突 | 不做 |

本品不适合IAP为唯一收入：轻量棋盘游戏初期付费意愿未知；也不适合广告为唯一收入：高频广告会直接破坏策略节奏。App应采用克制的混合模式。

### 8.3 App接入门槛

- Capacitor或原生壳、Android/iOS工程和签名。
- 广告SDK初始化、预加载、生命周期、后台恢复和回调幂等。
- Google Play Billing/StoreKit、恢复购买、交易校验和退款边界。
- 隐私政策、同意管理、数据安全/隐私标签、年龄分级和目标受众判断。
- iOS追踪与广告标识相关流程必须按当期Apple与广告SDK要求处理。
- 崩溃监控、版本升级、商店审核、回滚与长期SDK维护。

因此App不是“把H5压成安装包”即可完成；H5数据达标后再启动仍是正确顺序。

## 9. 登录、云存档与统计策略

### 9.1 登录与云存档

- 独立站首发维持无账号、本地存档，降低隐私和后端成本。
- H5门户优先使用平台允许的身份/存储能力，不要求玩家再注册Fangrush账号。
- App只有在跨设备、付费恢复或多渠道统一身份需求明确后才建设自有账号和云存档。
- 云存档必须定义版本、冲突合并、离线写入、回滚和平台迁移，不能只上传一个JSON。

### 9.2 统计与监测

独立站可以在隐私合规前提下统计页面访问、开玩、首局完成和渠道来源；H5门户默认使用平台数据，不自行嵌入未经批准的统计脚本；App可在明确披露和同意边界下接入分析与崩溃工具。

建议只采集支持决策的最小事件：

- 首次开玩、首次有效移动、首次捕食、首局终局。
- 每关开始/结束、胜负、终局原因、plies、重试。
- 提示使用、奖励理解入口、皮肤查看与兑换。
- 广告机会、请求、开始、完成、失败原因、奖励发放。
- 会话长度、每会话局数、次日/七日回访（平台可提供时）。

不采集棋盘策略无关的敏感个人信息；不为了“数据完整”引入未经平台允许的外部请求。

## 10. 广告状态与奖励契约

统一广告结果至少包括：成功、取消、失败、不可用、无填充和冷却。任何平台Adapter都必须映射到同一结果，不让业务层依赖平台私有回调。

激励广告契约：

```text
基础奖励先确定并幂等保存
  -> 玩家主动请求广告
  -> 广告成功完成
  -> 广告奖励幂等发放

取消/失败/不可用/无填充/冷却
  -> 不发广告奖励
  -> 不回滚基础奖励
  -> 恢复输入与音频
  -> 保留再来一局和下一关
```

重复回调、页面恢复、App前后台切换和SDK异常都不能重复发奖。

## 11. 盈利测算与数据门槛

广告收入模型：

```text
有效广告收入 = 有效展示数 / 1000 * 实际eCPM
门户到手收入 = 有效广告收入 * 合同分成比例
```

规划可使用低/中/高情景，不把它写成报价：

| 每日有效展示 | eCPM $1 | eCPM $5 | eCPM $15 |
|---:|---:|---:|---:|
| 1,000 | $1 | $5 | $15 |
| 10,000 | $10 | $50 | $150 |
| 100,000 | $100 | $500 | $1,500 |

实际结果取决于地区、终端、广告类型、完成率、填充率、AdBlock、隐私选择、季节、平台分成和无效流量。没有稳定玩家量时，广告收入可以忽略。

### 11.1 H5继续投入指标

- 平台点击到开玩转化。
- 首局开始率、首局完成率和首局胜败后继续率。
- 平均会话时长、每会话局数、回访。
- 激励广告机会率、主动点击率、开始率、完成率和广告后继续率。
- 实际eCPM、填充率、每DAU广告展示和每DAU收入。
- 广告组与不看广告玩家的留存差异。

### 11.2 App启动条件

至少一个H5平台证明：核心玩法有人持续玩、结算激励有真实需求、收入或留存值得承担移动端工程和商店运营成本。未达到门槛时，不因“App可能单价更高”提前投入。

## 12. 分阶段实施

### 阶段A：当前双H5首发

- 独立站无广告。
- 完成Poki/CrazyGames真实Adapter、生命周期和广告回调。
- 只上线结算激励广告。
- 使用平台数据验证玩家、关卡和广告价值。

### 阶段B：H5数据优化

- 调整激励文案、目标差额和奖励方案。
- 只有数据支持时测试低频局间广告。
- 不以短期广告收入牺牲首局完成和回访。

### 阶段C：App验证版

- 建立Capacitor/原生壳和一种移动广告方案。
- 接入激励广告、低频插屏、去广告IAP和少量皮肤IAP。
- 完成隐私、恢复购买、崩溃和商店审核闭环。

### 阶段D：规模化

- 根据DAU、地区和填充评估广告聚合。
- 根据付费和收藏数据扩展皮肤，而不是先扩货币。
- 账号、云存档、推送和长期运营按真实需求启动。

## 13. 验收标准

- 所有渠道复用同一规则、AI和24关配置。
- 独立站生产环境无广告；Admin仅存在于允许的开发环境。
- 门户包只加载对应平台SDK，不加载其他广告、统计和无关外链。
- 广告只在允许节点请求；核心操作期间不会触发。
- 广告失败、无填充、AdBlock和冷却不影响基础游戏。
- 激励成功才发奖，重复回调不重复发奖。
- 广告前后输入、AI时序和音频可靠暂停与恢复。
- 玩家能看懂奖励、目标皮肤和差额，不看广告也能继续。
- App购买具备恢复、幂等、退款与去广告边界。
- 收入结论使用真实平台数据、合同和可复现公式，不使用未经证实的固定单价。
