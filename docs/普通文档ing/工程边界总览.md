# 工程边界与最终架构准备

> 本文档是跨模块、跨部署平台的工程架构入口。它只维护长期有效的边界、决策和实施阶段；函数、字段、页面清单、环境变量名称和构建参数以代码、测试和仓库脚本为唯一来源。

## 1. 总体架构

项目采用“共享游戏核心 + 平台外壳 + 平台服务适配”的结构：

```text
packages/game-core
  规则、合法行动、终局、AI、关卡、奖励结算
          │
          ├── packages/web-shared
          │     跨应用契约、共享样式和平台边界
          │
          ├── apps/player-web
          │     独立网站、H5 玩家入口、浏览器存储
          │
          ├── apps/admin
          │     Admin、验收和管理 API
          │
          └── future apps/mobile
                Android/iOS 外壳、原生生命周期、广告、IAP、登录
```

共享核心不得因为网站、H5 或 App 的展示方式不同而复制规则。平台差异只能进入外壳或适配层。

## 2. 代码职责

| 区域 | 唯一职责 | 不应维护的内容 |
|---|---|---|
| `packages/game-core` | 规则、AI、关卡配置、对局状态、奖励结算 | 浏览器 API、平台 SDK、页面文案 |
| `apps/player-web` | 玩家页面、输入、展示、浏览器生命周期和平台适配 | 第二套游戏规则、Admin 实现 |
| `apps/player-web/lib/platform.ts` | 平台生命周期和平台能力抽象 | 具体平台业务流程 |
| `apps/player-web/lib/ads.ts` | 广告能力接口、Mock 和不可用降级 | 真实平台密钥和平台业务规则 |
| `apps/admin` | Admin 页面、验收、AI诊断和管理 API | 玩家正式入口和第二套规则 |
| `packages/web-assets` | 共享运行资源和资源生成/检查工具 | 页面、规则和平台密钥 |
| `distribution/` | 构建说明、提交资料、合规和素材 | 运行时游戏逻辑 |
| 未来 `apps/mobile` | 移动端外壳、原生桥接、商店能力 | 复制 `game-core` 规则 |
| 数据库/服务端（未来） | 账号、云存档、统计、后台运营数据 | 客户端本地规则判定 |

代码中的具体目录和接口可能演进；发生变化时先更新代码和测试，再同步本文档的职责边界。

## 2.1 公共能力与应用依赖

前台 Player 与后台 Admin 不是两套独立规则，也不是所有代码都放进同一个公共包。当前共享关系如下：

```text
packages/game-core    规则、AI、关卡、奖励
packages/web-shared   跨应用类型、契约、共享基础样式
packages/web-assets   图片、皮肤、音效和资源检查工具
        ▲                    ▲
        │                    │
apps/player-web       apps/admin
玩家产品与运行时       Admin、验收、AI诊断、管理 API
        ▲
        │ Admin 按需复用部分成熟的玩家运行时
```

### 共享层职责

| 层 | 维护内容 | Player | Admin |
|---|---|---|---|
| `packages/game-core` | 合法行动、终局、AI、24关、奖励和存档模型 | 使用 | 使用 |
| `packages/web-shared` | 平台/广告/存储/验收契约、共享基础样式 | 使用 | 使用 |
| `packages/web-assets` | 皮肤、棋盘、音效、品牌资源和检查工具 | 构建同步 | 构建同步 |
| `apps/player-web` | 玩家页面、棋盘、对局会话、玩家存档、平台适配 | 所有者 | Admin 按需复用部分运行时 |
| `apps/admin` | 关卡台、Admin 试玩、AI 诊断、验收记录和管理 API | 不依赖 | 所有者 |

Admin 当前允许单向复用 `apps/player-web` 的成熟运行时，例如 `PlayScreen`、`BoardSvg`、帮助内容、对局 store、音效和部分适配器；通过 `adminMode` 关闭玩家奖励、解锁、任务和正式进度写入。Admin 不得复制规则、AI 或棋盘交互，也不得被 Player 反向引用。

依赖方向约束为：

```text
apps/player-web  ──> game-core / web-shared / web-assets
apps/admin       ──> game-core / web-shared / web-assets / player-web部分运行时
player-web       -X-> admin
```

只有当公共运行时持续扩大、Player 改动频繁导致 Admin 构建回归，或两端需要独立演进时，才考虑把棋盘运行时抽成新的 `packages/web-runtime`；在此之前不为抽象而抽象。

## 3. 部署目标与当前状态

本地 `5003` 和 `5002` 只是开发端口，不代表生产部署方式。生产目标是“一个 monorepo、两个独立应用、两个部署项目”：玩家端与 Admin 分开构建、分开发布、分开配置域名和访问权限；两者仍共享 `game-core` 与 `web-shared`。

| 部署目标 | 当前实现 | 最终定位 | 当前阶段 |
|---|---|---|---|
| 独立网站 | `apps/player-web` 的 standalone 构建 | 无广告演示、玩家体验和项目申请入口 | 现在优先部署 |
| Poki | Web 平台构建变量和接口骨架 | 平台 SDK、广告和生命周期接入 | 平台条件明确后开发 |
| CrazyGames | Web 平台构建变量和接口骨架 | 平台 SDK、广告和生命周期接入 | 平台条件明确后开发 |
| Admin | Web 内部工具 | 关卡、AI、平台和验收辅助 | 随核心版本维护 |
| Android/iOS | 目前只有预留桥接，不是可发布 App | 移动端游戏外壳和商店版本 | 数据达标后启动 |

当前 `build:poki` 和 `build:crazygames` 是环境区分后的 Web 构建，不等于真实 SDK 已接入，也不等于已经生成平台最终提交包。真实 SDK 的接口、密钥和审核要求必须以平台资料为准。

### 3.1 本地端口与生产部署

| 应用 | 本地开发 | 推荐生产项目 | 推荐生产域名 | 是否公开 |
|---|---|---|---|---|
| `apps/player-web` | `http://localhost:5003` | 独立 Vercel Project | `fangrush.com` | 是 |
| `apps/admin` | `http://localhost:5002` | 独立 Vercel Project | `admin.fangrush.com`（取得远程访问需要时再启用） | 否/受保护 |

Vercel 项目仍来自同一个 Git 仓库，但分别设置 Root Directory：玩家项目指向 `apps/player-web`，Admin 项目指向 `apps/admin`。不要把两个应用合并为一个公开项目，也不要把 `5002`、`5003` 写入生产访问说明。

Vercel 维护原则：两个 Project 分别配置构建命令、Preview/Production 环境变量、域名、访问权限和回滚记录；Player 先发布，Admin 按需发布。Root Directory、环境变量和部署状态属于 Vercel 运维配置，代码仓库中的构建脚本、发布门禁和产物审计仍是工程事实来源。详细操作见 [`distribution/standalone/README.md`](../../distribution/standalone/README.md)。

环境变量按应用维护：`apps/player-web/.env.example`、`apps/player-web/.env.local`、`apps/admin/.env.example` 和 `apps/admin/.env.local` 纳入 Git，文件只保存安全的本地初始值；真实 Vercel Secret、生产凭据和个人数据不纳入 Git。Player 维护站点、平台和广告构建变量；Admin 当前只维护 `NEXT_PUBLIC_APP_SHELL`、`ADMIN_ENABLED` 和 `ADMIN_ACCESS_KEY`。根目录环境变量文件已废弃。

## 4. 构建和发布原则

1. 独立网站先使用生产 Web 构建，关闭平台广告和 Admin 对外入口。
2. 平台申请前只维护接口、Mock、构建边界和提交材料，不凭猜测写真实 SDK。
3. 平台审核或项目条件明确后，分别实现 Poki、CrazyGames 的适配器和平台验收。
4. App 不从 Web 页面直接“伪装打包”为最终产品；需要独立移动端壳、原生生命周期、签名和商店发布配置。
5. 所有平台必须复用同一份 `game-core`，平台层只能处理加载、暂停、恢复、广告、分享、支付和账号等外部能力。
6. 生产构建、资源检查、核心测试和 Web 回归门禁必须在平台发布前通过。

## 5. App 技术方向

当前推荐：先保持 `game-core` 与 Web 独立，未来采用 **Capacitor + 独立 `apps/mobile` 外壳**，必要时通过原生插件接入广告、IAP、推送和登录。

选择依据：

- 现有前端是 Next/React，棋盘交互已经在 Web 层完成，Capacitor 可以最大化复用 UI 和交互代码。
- `game-core` 是纯共享核心，适合在移动 WebView 中继续运行，不需要重写规则引擎。
- 广告、IAP、登录和商店生命周期必须通过原生桥接隔离，避免把 App 专属依赖污染 Web 和 H5。
- 先做 Capacitor 方案的成本低于立即维护 React Native 或 Flutter 的第二套 UI；在数据尚未证明需要原生性能时，不应提前承担双 UI 的维护成本。
- 如果未来出现明显的原生性能、复杂动画或平台体验要求，再评估 React Native/Flutter，而不是现在就锁死更重的方案。

## 6. 今天开始的基础准备

现在做：

- 确认共享核心、Web、平台适配、未来 App 的目录边界。
- 确认独立网站的生产构建和部署入口。
- 确认广告、平台生命周期、分享、支付和账号都通过适配接口进入。
- 保留真实 SDK 的接入点和 Mock，不实现尚未确定的平台 SDK。
- 维护平台材料、构建产物和密钥的隔离规则。
- 为玩家端和 Admin 分别维护部署项目、域名、环境变量和回滚记录；Admin 不进入玩家端产物。

现在不做：

- 不实现 Poki/CrazyGames 真实 SDK。
- 不创建可误认为已完成的 Android/iOS 发布包。
- 不提前接入广告网络、IAP、账号和云存档。
- 不因为平台差异复制规则、AI 或关卡配置。

## 7. 阶段顺序

| 阶段 | 目标 | 完成条件 |
|---|---|---|

## 9. 玩家入口与 Admin 拆分状态

当前玩家和 Admin 已由两个独立 Next 应用拥有实现，旧兼容应用已退役。当前职责为：

- `packages/game-core`：规则、关卡、AI、奖励。
- `apps/player-web/components`：棋盘、对局、页面导航和帮助内容。
- `apps/player-web/i18n`：服务端与客户端文本。
- `apps/player-web/lib/save-store.ts`、`storage.ts`、`active-game.ts`：本地存档和恢复。
- `apps/player-web/lib/platform.ts`、`ads.ts`：平台和广告能力接口。
- `apps/admin`：Admin 页面、API、验收和 AI 诊断。
- `packages/web-assets`：共享图片、音效、图标及其生成和检查工具。

源码边界检查确保 Player 与 Admin 不反向耦合；玩家和 Admin 产物审计分别由 `pnpm audit:player-artifact` 和 `pnpm audit:admin-artifact` 提供证据。

物理拆分顺序固定为：

1. 先保持单应用运行，抽取并验证共享依赖边界。
2. 建立玩家 Web 构建入口，复用 `game-core` 和共享 Web 能力。
3. 建立独立 Admin 内部构建入口，保留质量工具和 Admin API。
4. 让平台包严格审计通过后，才把平台提交包标记为不含 Admin。

玩家和 Admin 的实现、测试、资源及构建入口迁移已完成。部署拆分不等于仓库拆分：两个 Vercel Project 继续使用同一个 monorepo 和共享包。
