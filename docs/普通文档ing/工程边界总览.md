# 工程边界与最终架构准备

> 本文档是跨模块、跨部署平台的工程架构入口。它只维护长期有效的边界、决策和实施阶段；函数、字段、页面清单、环境变量名称和构建参数以代码、测试和仓库脚本为唯一来源。

## 1. 总体架构

项目采用“共享游戏核心 + 平台外壳 + 平台服务适配”的结构：

```text
packages/game-core
  规则、合法行动、终局、AI、关卡、奖励结算
          │
          ├── apps/web
          │     独立网站、H5 平台入口、Admin、浏览器存储
          │
          └── future apps/mobile
                Android/iOS 外壳、原生生命周期、广告、IAP、登录
```

共享核心不得因为网站、H5 或 App 的展示方式不同而复制规则。平台差异只能进入外壳或适配层。

## 2. 代码职责

| 区域 | 唯一职责 | 不应维护的内容 |
|---|---|---|
| `packages/game-core` | 规则、AI、关卡配置、对局状态、奖励结算 | 浏览器 API、平台 SDK、页面文案 |
| `apps/web` | Web 页面、Admin、输入、展示、浏览器生命周期 | 第二套游戏规则、平台专属原生实现 |
| `apps/web/src/lib/platform.ts` | 平台生命周期和平台能力抽象 | 具体平台业务流程 |
| `apps/web/src/lib/ads.ts` | 广告能力接口、Mock 和不可用降级 | 真实平台密钥和平台业务规则 |
| `distribution/` | 构建说明、提交资料、合规和素材 | 运行时游戏逻辑 |
| 未来 `apps/mobile` | 移动端外壳、原生桥接、商店能力 | 复制 `game-core` 规则 |
| 数据库/服务端（未来） | 账号、云存档、统计、后台运营数据 | 客户端本地规则判定 |

代码中的具体目录和接口可能演进；发生变化时先更新代码和测试，再同步本文档的职责边界。

## 3. 部署目标与当前状态

| 部署目标 | 当前实现 | 最终定位 | 当前阶段 |
|---|---|---|---|
| 独立网站 | `apps/web` 的 standalone 构建 | 无广告演示、玩家体验和项目申请入口 | 现在优先部署 |
| Poki | Web 平台构建变量和接口骨架 | 平台 SDK、广告和生命周期接入 | 平台条件明确后开发 |
| CrazyGames | Web 平台构建变量和接口骨架 | 平台 SDK、广告和生命周期接入 | 平台条件明确后开发 |
| Admin | Web 内部工具 | 关卡、AI、平台和验收辅助 | 随核心版本维护 |
| Android/iOS | 目前只有预留桥接，不是可发布 App | 移动端游戏外壳和商店版本 | 数据达标后启动 |

当前 `build:poki` 和 `build:crazygames` 是环境区分后的 Web 构建，不等于真实 SDK 已接入，也不等于已经生成平台最终提交包。真实 SDK 的接口、密钥和审核要求必须以平台资料为准。

## 4. 构建和发布原则

1. 独立网站先使用生产 Web 构建，关闭平台广告和 Admin 对外入口。
2. 平台申请前只维护接口、Mock、构建边界和提交材料，不凭猜测写真实 SDK。
3. 平台审核或项目条件明确后，分别实现 Poki、CrazyGames 的适配器和平台验收。
4. App 不从 Web 页面直接“伪装打包”为最终产品；需要独立移动端壳、原生生命周期、签名和商店发布配置。
5. 所有平台必须复用同一份 `game-core`，平台层只能处理加载、暂停、恢复、广告、分享、支付和账号等外部能力。
6. 生产构建、资源检查、核心测试和 Web 回归门禁必须在平台发布前通过。

## 5. App 技术方向

当前推荐：先保持 `game-core` 与 Web 独立，未来采用 **Capacitor + 独立 `apps/mobile` 外壳**，必要时通过原生插件接入广告、IAP、推送和登录。

选择依据：

- 现有前端是 Next/React，棋盘交互已经在 Web 层完成，Capacitor 可以最大化复用 UI 和交互代码。
- `game-core` 是纯共享核心，适合在移动 WebView 中继续运行，不需要重写规则引擎。
- 广告、IAP、登录和商店生命周期必须通过原生桥接隔离，避免把 App 专属依赖污染 Web 和 H5。
- 先做 Capacitor 方案的成本低于立即维护 React Native 或 Flutter 的第二套 UI；在数据尚未证明需要原生性能时，不应提前承担双 UI 的维护成本。
- 如果未来出现明显的原生性能、复杂动画或平台体验要求，再评估 React Native/Flutter，而不是现在就锁死更重的方案。

## 6. 今天开始的基础准备

现在做：

- 确认共享核心、Web、平台适配、未来 App 的目录边界。
- 确认独立网站的生产构建和部署入口。
- 确认广告、平台生命周期、分享、支付和账号都通过适配接口进入。
- 保留真实 SDK 的接入点和 Mock，不实现尚未确定的平台 SDK。
- 维护平台材料、构建产物和密钥的隔离规则。

现在不做：

- 不实现 Poki/CrazyGames 真实 SDK。
- 不创建可误认为已完成的 Android/iOS 发布包。
- 不提前接入广告网络、IAP、账号和云存档。
- 不因为平台差异复制规则、AI 或关卡配置。

## 7. 阶段顺序

| 阶段 | 目标 | 完成条件 |
|---|---|---|
| A | 独立网站生产准备 | 构建、环境、安全、移动浏览器和核心回归通过 |
| B | 首批项目/平台申请 | 材料齐全、账号和申请完成，等待平台条件 |
| C | 平台 SDK 接入 | 以官方规格实现适配、广告回调和生命周期 |
| D | 真实平台验收 | Inspector/Preview/Portal 和提交检查通过 |
| E | App 商业化 | H5 有真实数据后，再启动移动壳、广告、IAP 和账号 |
