# Fangrush 最终诊断与执行清单

> 目的：在已有“基本可玩”和第一轮体验升级的基础上，集中诊断所有剩余产品问题，并形成确认后可直接执行的完整清单。
>
> 范围：前台产品体验、棋局、关卡、皮肤展示、音效动画接入、页面质量、工程质量和商业化准备。
>
> 排除项：专业美术/音频团队后续生产的完整资源 backlog。本清单只处理资源规范、接入质量、替换能力和当前可用体验。

## 一、当前状态判断

### 已完成的基础升级

- 首页桌面端由窄容器升级为宽屏布局。
- 主要页面已统一页面宽度和基础视觉结构。
- 皮肤页、设置页具备开始/继续游戏入口。
- 棋局顶部返回和底部退出语义已区分。
- 对局结束不再自动弹出插屏广告，胜利奖励先结算。
- 皮肤页图片使用 `next/image`。
- 棋盘支持键盘选择可操作狼和目标位置。
- 新手引导支持 Escape 关闭，并具备对话框语义。
- 全局增加键盘焦点样式。
- Hard AI 计时逻辑已跨环境兼容。
- 核心测试 23/23 通过，皮肤资源检查通过，Web 构建和 lint 通过。

### 当前产品阶段

项目已经从“功能原型”进入“产品质量和商业化准备阶段”。剩余问题不再主要是单个按钮或单个页面，而是以下系统性问题：

1. 第一局是否足够清晰、完整、有趣。
2. 关卡是否经过可解释的平衡验证。
3. 角色和棋盘资源是否在真实观看距离下清晰可辨。
4. 长期内容和奖励是否能形成持续游玩理由。
5. 页面、存档、广告失败和异常状态是否有稳定兜底。
6. 是否具备真实用户测试和商业化实验的基础。

## 二、最终诊断

### P0：必须先解决的问题

P0 指无法开始、无法完成、状态丢失、奖励错误或会阻断核心玩法的问题。

- [ ] 验证所有关键路由在首次访问、刷新、浏览器返回和直接输入地址时均可用。
- [ ] 验证本地存档尚未生成、存档损坏、存档字段缺失时能回退到默认状态。
- [ ] 验证胜利、失败、重试、重置、退出和继续游戏不会重复发奖或丢失进度。
- [ ] 验证广告 SDK 不可用、加载失败、用户拒绝或网络中断时不影响基本玩法和奖励结算。
- [ ] 验证 AI 思考超时、无合法行动和异常关卡配置时有可恢复表现。
- [ ] 验证棋局在桌面端、移动端和低性能设备上不会出现无法点击、点击穿透或永久等待。

### P1：核心体验必须解决的问题

P1 指玩家能进入游戏，但会明显困惑、挫败、误解或无法顺畅继续。

- [ ] 第一局需要同时完成教学和留存，必须让玩家看见并理解狼、羊、移动、吃子、回合和胜负。
- [ ] 春日第一关需要重新确认岩石的教学价值，不能为了“简单”而让玩家看不到核心差异，也不能第一次就堆过多规则。
- [ ] 关卡必须有布局意图、难度目标、预计回合数和测试结果，不能只凭开发者感觉判断平衡。
- [ ] 狼、羊、岩石在约 30cm 桌面观看距离和 375px 移动端上必须能迅速区分。
- [ ] 无效点击、选择错误、AI 思考、连吃、胜利和失败必须提供清晰且不冲突的反馈。
- [ ] 所有页面必须给出明确下一步，不能让玩家停在没有行动方向的结果或展示页。
- [ ] 中英文文案不能出现硬编码遗漏、混用、长度溢出或按钮含义不一致。

### P2：产品品质需要解决的问题

P2 指不阻断玩法，但会影响高级感、理解成本、信任感和长期留存。

- [ ] 皮肤需要从“素材列表”升级为“主题收藏”：角色、羊、棋盘、名称、解锁目标和装备效果一致。
- [ ] 皮肤解锁成本、掉落频率和章节推进需要形成可解释的奖励曲线。
- [ ] 首页、玩法、章节、关卡、猎场、皮肤、任务、设置和棋局需要形成稳定导航闭环。
- [ ] 动画时长、音效音量、连吃反馈和胜负反馈需要分层，不应全部使用同等强度。
- [ ] 支持 `prefers-reduced-motion`，并确保减少动效后玩家仍能理解状态变化。
- [ ] 图片、按钮、表单、弹层和棋盘需要完成无障碍语义、焦点和替代文本检查。
- [ ] 桌面宽屏需要有合理的信息密度，不能只是把窄页面拉宽后留下大面积空白。

### P3：商业化和高级体验问题

- [ ] 建立激励广告抽象层，SDK 只负责播放，游戏负责资格、频次、奖励和失败兜底。
- [ ] 明确撤回、双倍奖励、额外碎片和皮肤解锁的边界，不能让广告成为通关必需。
- [ ] 建立事件记录和实验开关，在有用户量后再调整广告频率和奖励数量。
- [ ] 为皮肤、章节、任务和重复游玩建立可扩展内容模型。
- [ ] 建立发布前质量门禁、回滚策略和线上问题记录机制。

## 三、完整解决方案

### 方案 A：首局体验重构

1. 用短引导说明当前动作，不一次解释全部规则。
2. 引导必须可跳过、可重新查看、可用键盘操作，并在首次关键操作后自动收束。
3. 第一关负责让玩家完成一次有意义的策略选择。
4. 首次胜利展示奖励、下一关和再次挑战；首次失败展示原因、重试和玩法说明。
5. 第一局不展示强制广告、不打断结算、不要求理解皮肤和任务系统。

### 方案 B：关卡平衡体系

每个关卡增加以下设计记录和验证维度：

- 教学点和玩家预期。
- 狼、羊、岩石数量和初始布局意图。
- 狼方可行动空间、羊方逃生空间和首回合影响。
- 关键策略、常见错误和失败原因。
- 预计最短、平均和最长对局长度。
- AI 模拟结果、熟练玩家结果和新手可理解性结果。

固定布局用于教学和复盘；受控变体用于重复游玩。任何变体都必须经过规则引擎合法性检查，不能用随机性掩盖不平衡。

### 方案 C：资源和皮肤体系

- 为狼、羊、岩石建立最小识别尺寸和明暗对比标准。
- 每套皮肤必须有完整主题，不允许只更换单个角色而破坏整体一致性。
- 预览必须真实反映棋局效果，装备后应能立即看到变化。
- 皮肤页面显示解锁条件、当前碎片、装备状态和进入游戏入口。
- 资源生产 backlog 后续只需替换 Catalog 资源，不应再次改动核心棋局组件。

### 方案 D：棋局反馈体系

统一处理以下状态：

- 选择狼：焦点、选中环、轻微放大、选择音效。
- 可走位置：普通移动和吃子位置使用不同视觉等级。
- 落子：短动画和明确落点。
- 吃子：路径、目标消失、冲击音效和连吃计数。
- AI 思考：遮挡输入并提示当前回合。
- 无效操作：短促反馈，不改变棋局状态。
- 胜利/失败：结果、奖励、重试、下一步行动和退出路径。
- 静音/减少动效：所有状态仍可理解。

### 方案 E：页面和异常恢复

- 首页提供开始和继续两个明确入口。
- 章节页说明未解锁原因和下一步行动。
- 关卡页显示难度、目标、奖励和玩法入口。
- 皮肤页显示预览、装备、解锁和继续游戏。
- 设置页显示静音、玩法说明、继续游戏和隐私入口。
- 棋局页区分返回关卡列表、退出首页、重试当前关卡。
- 刷新和浏览器返回不应造成重复奖励、空白状态或错误路由。

### 方案 F：商业化准备

执行顺序必须是：

1. 核心玩法稳定。
2. 新手首局稳定。
3. 奖励和皮肤循环稳定。
4. 再接入激励广告。
5. 最后根据数据决定是否增加低频插屏。

激励广告只提供可选收益，例如额外碎片、一次撤回或奖励翻倍。广告不可用时，用户仍能完成关卡并获得基础奖励。

## 四、执行顺序

### 第一批：发布阻断和状态安全

- [ ] 路由冒烟：所有公开页面、所有主要关卡、直接地址访问。
- [ ] 存档异常：空存档、旧存档、损坏 JSON、缺字段。
- [ ] 奖励幂等：重复渲染、刷新、返回、重试后的发奖行为。
- [ ] 广告失败：失败、取消、不可用、超时。
- [ ] AI 异常：超时、无行动、异常布局。

### 第二批：首局和新手引导

- [ ] 确定春日第一关教学目标和岩石策略。
- [ ] 验证引导弹层、跳过、继续、Escape、键盘和触摸。
- [ ] 验证第一步有效操作、第一次吃子、第一次胜利和第一次失败。
- [ ] 验证首胜奖励、下一关、重试和退出路径。

### 第三批：关卡平衡

- [ ] 为春日全部关卡建立布局分析。
- [ ] 运行 AI 批量模拟并记录狼胜率、羊胜率、超步和平均步数。
- [ ] 对极端布局、必胜路线、无意义岩石和重复路线进行人工复盘。
- [ ] 调整关卡顺序和教学点，而不是只调整数量。

### 第四批：资源和棋局表达

- [ ] 完成狼、羊、岩石缩小识别检查。
- [ ] 完成所有皮肤主题的成套预览和装备验证。
- [ ] 验证棋盘、角色、选择、可走、吃子和胜负状态的对比度。
- [ ] 验证声音分层、静音和减少动效。

### 第五批：长期内容和奖励

- [ ] 梳理章节、关卡、任务、碎片和皮肤的推进关系。
- [ ] 检查第一套可解锁皮肤是否足够有吸引力。
- [ ] 检查奖励获得速度是否让玩家有近期目标和长期目标。
- [ ] 检查重复通关是否仍有合理收益但不会无限刷取破坏经济。

### 第六批：商业化试验

- [ ] 完成广告资格、冷却、每日上限和失败兜底。
- [ ] 只在结果页或玩家主动点击处提供激励广告。
- [ ] 记录展示、完成、奖励发放、取消和失败事件。
- [ ] 设计小流量实验，不在没有数据时强行提高频率。

### 第七批：发布前验收

- [ ] 375、390、414、1024、1280、1440、1920 视口。
- [ ] 中文、英文、刷新、返回、静音、减少动效。
- [ ] 新玩家、回访玩家、皮肤玩家、静音玩家、异常玩家。
- [ ] 核心测试、资源检查、lint、生产构建和端到端冒烟。
- [ ] 记录 P0/P1/P2/P3 结果并决定是否发布。

## 五、验收标准

### 玩家体验

- 新玩家不看外部说明即可完成第一步有效操作。
- 玩家能分辨狼、羊、岩石和可走位置。
- 玩家能理解胜利、失败和下一步行动。
- 失败不会让玩家丢失进度或奖励。
- 皮肤、任务和关卡页面不会形成死路。

### 关卡质量

- 每关有设计意图和测试记录。
- 没有未经解释的必胜、必败或无意义路线。
- 难度递进和新增元素有明确理由。

### 工程质量

- 核心测试通过。
- 资源检查通过。
- lint 无错误。
- Web 生产构建通过。
- 广告和网络失败可恢复。
- 存档和奖励逻辑具备幂等性。

### 商业化质量

- 无广告时游戏完整可玩。
- 新手首局不被强制广告打断。
- 激励广告是可选收益，不是通关门槛。
- 广告失败不会吞奖励、卡住界面或污染存档。

## 六、执行决策

确认后将严格按“第一批到第七批”执行，每完成一批都进行对应验收，不跨过 P0/P1 问题直接扩展商业化。

优先原则只有一个：**先让玩家清楚、顺畅、有兴趣地完成一局，再让玩家愿意重复玩，最后让商业化自然发生。**
