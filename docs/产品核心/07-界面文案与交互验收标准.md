# 界面文案与交互验收标准
本文定义 Fangrush 玩家端与 Admin 的长期界面表达和交互验收口径。它负责“信息怎样让人理解和操作”，不重复维护规则数值、关卡配置、视觉资源细节或当前任务进度。

## 一、基本原则

1. 玩家先理解“现在发生了什么、要做什么、下一步去哪”，再看到奖励、分享和广告。
2. 一个页面只保留一个明确主任务；次要操作降级到文字按钮、折叠区或更多菜单。
3. 文案描述玩家能观察到的行为，不暴露函数名、字段名、算法名和测试术语。
4. 数字必须同时说明对象和口径。“3/5”应写成“玩家操作难度 3/5”，“150”应写成“150 次单方行动”。
5. 自动代理数据用于发现风险，不得称为玩家胜率，不得替代人工试玩结论。
6. 中文先保证自然、准确和简短，再写含义一致的自然英文；英文不是逐字翻译。

## 二、三种语言边界

### 玩家语言

使用“捕获、行动、羊方回合、被困、下一关”等直接表达。局内不出现 `ply`、`seed`、`mixed`、`pickSheepAction`、P95 或内部状态名。

### Admin 产品语言

可以显示“固定种子、自动代理、P95、规则终局”等证据概念，但第一次出现时必须解释用途。Admin 同时明确区分：

- 玩家操作难度：找出并执行获胜路线的难度；
- 羊方防守强度：正式游戏使用的 AI 档位；
- 自动代理结果：固定策略和种子的自动排雷数据；
- 人工验收：真实玩家对理解度、难度和体验的判断。

### 代码与测试语言

函数、类型、字段、完整配置和脚本参数只由代码、测试和生成报告维护。产品文档只保留定义、边界和解释，不抄写可从代码生成的明细。

## 三、关键页面信息顺序

### 关卡说明

关卡名称与名称含义 → 本关目标 → 盘面特点 → 主要策略提示 → 开始挑战。未被真实数据支持的精确预计步数不面向玩家展示。

### 局内

当前捕获数/正式目标 → 剩余羊数 → 当前行动方 → 棋盘 → 当前必要操作。帮助、重开、静音和退出不得挤压棋盘主任务。

### 终局

胜负 → 一句话原因或建议 → 唯一主动作 → 可选详情。胜局主动作是下一关；失败或和局主动作是重新挑战。分享、再玩一次、奖励广告和关卡列表属于次要操作。

终局允许关闭结果层查看只读棋盘，但必须保留稳定的“查看结果”和继续/重试入口。关闭不等于恢复对局。

### Admin 关卡页

关卡定位 → 玩家操作难度与羊防守强度 → 正式目标和行动上限 → 自动代理证据 → 人工验收 → 详细设计资料。摘要没有样本、策略、种子和棋谱时，不展示为完整胜率结论。

## 四、状态与异常

每个关键页面至少检查：首次进入、加载、空数据、正常、失败、重试、恢复、终局和无下一项。局内额外检查 AI 思考、连续捕食、无合法行动跳过、达到行动上限和重复局面。

错误文案应包含“发生了什么”和“用户现在能做什么”。不得只显示 `unexpected`、错误码或内部异常文本。

## 五、中英文与可读性

- 中文按钮优先使用 2 至 6 个字的明确动作；英文按钮优先使用动词短语。
- 标题不承担整段说明；解释文字不重复标题。
- 专有规则首次出现可配一个最短图例，后续使用统一名称。
- 同一概念全站同名，例如统一使用“捕获”和“行动”，不在玩家端混用“吃子、捕食、plies”。
- 在手机、平板和桌面视口检查最长中文、最长英文、动态数字和失败提示，不允许截断主信息或遮挡操作。

### 玩家端统一词典

| 概念 | 中文 | English | 不使用 |
|---|---|---|---|
| 狼的规则动作 | 隔空吃；首次图例解释为“狼—空点—羊同线，狼落到羊位” | leap capture；首次图例解释完整动作 | `gap-eat`、`gap-rush`、`gap-cut` |
| 目标与成绩 | 捕获、首次捕获 | capture、first capture | 英文 `eat/eats`；中文“兑现捕食” |
| 连续动作 | 连吃 | chain capture | 无解释的“连锁兑现” |
| 计数单位 | 次行动 | action / actions | 玩家端 `ply/maxPlies` |
| 非胜负终局 | 和局 | Draw | 僵局、Stalemate |
| 地图难度 | 挑战难度 | Challenge | 未说明口径的“难度/Difficulty” |
| 外观入口 | 装扮 | Skins | 把可兑换和装备页面只称“图鉴” |
| 岩石数量 | 无岩石、1 枚岩石、n 枚岩石 | No rocks, 1 rock, n rocks | `1 rocks`、`n 石` |

“三狼包围、诱导移动、链式捕捉”等策略名可以保留，但首次出现必须紧接一个可执行动作，不能只用“覆盖、兑现、建立接触、机动”等设计语言解释。

### 动态文案

- 数量、时间和货币必须在最终渲染句子中验收，不能只检查模板字段存在；英语必须处理单复数。
- 无数据是独立句式，例如“本局未捕获 / No captures”，不能把 `none` 拼进正常数字模板。
- core 错误码和异常文本先映射为本地化、可行动提示，玩家端不得直接显示内部英文。
- 分享图、无障碍名称、页面眉题和图片替代文本都属于本地化范围，不得因它们不是正文而硬编码单一语言。
- 24 关提示由一个生成入口投影；观察、目标、策略三层分别说明盘面信号、本关动作和下一步计划，并检查重复、长度和内部术语。

## 六、交互与无障碍门槛

- 主按钮在手机上至少 44px 高；图标按钮有可读名称和提示。
- 弹窗具备标题、模态语义、关闭方式和 Escape 行为；关闭后焦点不应丢失到不可见区域。
- 键盘可到达全部必要操作，焦点样式可见；状态变化使用适当的 `aria-live`，避免重复朗读。
- 终局、错误和只读状态不显示仍可改变已结束对局的控件。
- 颜色不是表达胜负、风险或选中状态的唯一方式。

## 七、证据与验收

每次涉及主流程或规则表达的改动至少完成：

1. 文案检索：排查玩家端内部术语、硬编码目标和口径不明数字；
2. 自动测试：规则、动态配置、跨季节入口、分享结果和异常状态；
3. 生产构建：玩家端与 Admin 均可构建；
4. 视口检查：桌面、平板、手机无溢出、遮挡和错误主次层级；
5. 人工体验：验证是否无需阅读代码即可说清目标、失败原因和下一步。

自动化只能确认确定性行为和布局风险。规则是否易懂、难度是否可信、文案是否自然，最终仍由真实使用与试玩反馈验证。

玩家端文案自动门禁至少覆盖统一词典禁词、岩石单复数、24 关中英三级提示和公开规则文本。新增玩家可见内部字段或自造术语时，门禁必须失败。
