# 0718 用户体验、验证与商业化专项

> 本文记录 2026-07-18 用户试玩反馈的产品判断、排查原则和首发阶段决策。它不替代代码、规则和测试的事实来源；任务状态以 [任务清单](../任务/任务清单.md) 为准，广告与渠道总原则见 [渠道能力与广告商业化总方案](../商业成功/渠道能力与广告商业化总方案.md)。

## 一、专项目标

本次反馈的核心不是逐条修饰页面，而是验证 Fangrush 是否形成了一个普通玩家可以独立理解的闭环：

```text
进入关卡 -> 看懂棋盘 -> 选择狼并行动 -> 完成/失败 -> 看懂奖励 -> 进入下一关 -> 愿意继续
```

同时验证 Admin 是否能让非程序员完成“查看、模拟、复盘、判断、记录”的工作，验证商业化是否建立在玩家已经理解并接受玩法的基础上。

用户只需要报告看到的现象或困惑；后续由开发侧主动检查代码、同类路径、真实构建和回归证据，不要求用户先提供完整错误信息。

## 二、反馈背后的产品意图

### 1. 玩家体验

- 通关后必须明确显示：完成的关卡、基础奖励、当前进度和推荐下一步。
- 玩家应该知道什么可以点击、什么只能观察、为什么某个位置不能走。
- 岩石、狼、羊的视觉含义必须自解释，尤其要避免孩子把岩石理解为“盾牌”而无法理解其规则作用。
- 移动端的结果弹窗、帮助弹窗和操作控件不能要求额外滚动或猜测关闭方式。
- 首页的 `Continue`、章节入口、重开和下一关必须反映真实存档状态。

### 2. Admin 体验

Admin 面向内容管理和验收，不要求管理员阅读代码。页面应先给出结论、风险和建议动作，再提供种子、模拟次数、胜率、棋谱等证据。

### 3. 商业化体验

商业化不是“尽可能多展示广告”，而是让玩家在不被强迫、不破坏策略节奏的情况下，愿意用时间换取明确的外观收藏进度。基础游戏、基础通关奖励和主线下一关不能被广告锁住。

## 三、玩家通关后的推荐结构

推荐顺序不是固定照搬某个平台，而是按玩家的下一步意图排序：

```text
通关结果
  -> 奖励确认
  -> Next hunt（主行动）
  -> Play again（次行动）
  -> Share result（辅助行动）
  -> Watch ad（自愿的额外奖励）
```

具体原则：

- `Next hunt` 是默认主按钮，因为完成一关后最自然的意图是继续主线。
- `Play again` 保留给想练习、刷奖励或挑战更好棋谱的玩家。
- `Share result` 是传播和复盘入口，不能因分享失败阻断继续游戏。
- `Watch ad` 不能成为默认主按钮，也不能替代基础奖励；它是“额外收益”而不是“领取奖励的门票”。
- 广告按钮并非越显眼越好。主流免费游戏通常将激励广告放在自然停顿点，以自愿奖励换取点击；强制插屏属于另一种广告形式，必须低频、可关闭并等待真实数据验证。
- 首发 H5 先使用结算后的自愿激励广告；独立站保持无广告；核心移动、捕食、连吃和教学阶段不展示广告。

当前代码已有结算激励入口，提供 30 分钟碎片双倍。需要重点验证的是玩家是否理解“未来 30 分钟双倍”以及是否有足够近的皮肤目标，而不是立刻增加广告频次。

## 四、Admin AI 的真实定位

### 1. 目的

Admin AI 不是让管理员替代人工玩完所有关卡，也不是简单显示一个“通过/失败”。它是关卡验收助手，用于：

1. 用固定种子和指定羊 AI 批量重现对局。
2. 发现狼方是否有稳定可执行的获胜路径。
3. 发现地图、开局、岩石、终局和 `maxPlies` 的异常。
4. 查看胜率、终局原因、首次捕食、行动数和关键棋谱。
5. 把可疑局面交给人工复盘，再决定规则、地图、AI 或文案是否需要调整。

### 2. 当前页面数据和自动化测试的关系

页面中的种子、模拟次数、胜率、平均 plies、终局原因和棋谱，使用的是同一套 `game-core` 规则和 AI 入口，但不等于仓库中所有测试的同一份结果。

| 内容 | 当前性质 |
|---|---|
| Admin 批量模拟 | 页面即时运行，种子、关卡、羊 AI 难度和次数由页面输入决定 |
| 固定种子平衡测试 | 仓库测试/脚本中的预设场景和断言，用于持续回归 |
| 候选验收器 | 对候选地图、开局和策略指标执行门禁，输出 accept/review/reject |
| 人工试玩 | 最终确认可理解性、操作感、教学和真实玩家体验 |

因此，“页面批量模拟全部通过”只能证明当前参数下的模拟结果满足页面或脚本门槛，不能单独证明关卡平衡或玩家体验通过。正确链路是：

```text
自动化规则测试 -> 固定种子批量模拟 -> 候选验收与棋谱 -> Admin 结论摘要 -> 人工试玩最终确认
```

### 3. Admin AI 页面应如何改进

首屏应优先展示：

- 当前关卡和 AI 难度。
- 本次模拟是否完成。
- 狼胜率、羊胜率、回合耗尽率和异常局数。
- 主要风险标签。
- 推荐动作：通过、人工复盘、修改地图/开局、检查 AI 或检查规则。

种子、模拟次数、预算、CSV、序列化数据和详细棋谱放到“展开证据”区域，并用通俗说明解释每个指标。页面还应明确“模拟不能替代人工试玩”。

## 五、季节主题、碎片和首发商业模型

当前代码中已有四个季节棋盘主题的章节解锁，以及冬季季节碎片兑换月夜棋盘；通用碎片用于兑换狼方皮肤。文案不能暗示尚未存在的完整兑换内容。

### 首发建议

- 每个季节提供一个基础季节棋盘主题，作为章节体验的一部分免费获得或随章节解锁。
- 不把玩家已经完成主线所需的核心内容锁在广告后。
- 额外的高级外观、夜间主题或特殊狼羊套装使用通用碎片兑换。
- 季节碎片只在对应季节存在真实可兑换目标时展示；如果暂时没有目标，应隐藏、标记预留或改为明确的“即将开放”，不能让玩家收集无用途货币。
- 激励广告的作用是加快外观收藏，不改变狼方战力、不跳过关卡、不直接提供必胜提示。

### 为什么不只靠增加关卡盈利

关卡是留存和内容价值，不应把每张关卡直接当成广告容器。可持续收入应分层：

```text
免费核心关卡与基础季节主题
  -> 玩家形成习惯
  -> 自愿激励广告加速碎片
  -> 高参与玩家购买去广告或少量高级外观
  -> 有真实数据后再扩展内容和渠道
```

首发不建议同时做大量货币、广告、订阅、付费关卡和复杂活动。先验证玩家是否愿意完成关卡、查看皮肤、主动看一次结算广告，再决定经济系统深度。

## 六、主动排查原则

每一个用户反馈都必须扩展为一组同类检查：

| 反馈现象 | 必查范围 |
|---|---|
| 通关后仍显示第一关 | 完成存档、主页 Continue、章节列表、下一关、重开和刷新恢复 |
| 结果弹窗在页面下方 | 胜利、失败、奖励、帮助、分享和移动端安全区域 |
| 分享微信报错 | Web Share、复制链接降级、平台能力判断、失败提示和各语言文案 |
| 岩石像盾牌 | 棋盘视觉、规则说明、首次教学、关卡提示和无障碍文本 |
| 非法位置显示方框 | 狼选择、合法落点、非法原因、鼠标、触控、键盘和焦点状态 |
| 羊图标先显示圆圈 | 图片加载、资源路径、默认皮肤、首屏 hydration 和构建产物 |
| Admin AI 看不懂 | AI、关卡、皮肤、经济、清单页面的摘要、术语和证据层级 |
| 碎片用途不清 | 获得、余额、兑换、目标差额、季节主题、广告奖励和中英文表达 |

排查顺序固定为：复现现象 -> 读取代码链路 -> 检查同类入口 -> 添加或运行回归测试 -> 修复 -> 桌面/移动端验证 -> 更新反馈记录和任务状态。

## 七、当前专项的执行顺序

1. 通关后进度、`Next hunt`、主页 `Continue`、移动端结果弹窗。
2. 分享失败降级和所有平台分享能力检查。
3. 棋盘层级、可操作对象、非法落点、加载占位和岩石含义。
4. 帮助中心、局内指导、语言入口和跨部署菜单一致性。
5. Admin AI 首屏摘要和 Admin 各页面的通俗性检查。
6. 碎片、季节主题、皮肤兑换和激励广告文案/实际行为一致性。
7. 完成自动化回归，再进入用户逐步验证 24 关。

本专项不提前启动：悔棋、高级提示、账号/云存档、App 壳、原生广告、IAP 和专业美术重制。这些仍按真实玩家数据和平台条件进入需求池。

## 八、验收标准

- 玩家不看代码、不看外部说明，也能完成一关并进入下一关。
- 通关后的主按钮符合继续主线的自然意图，广告不是主线门槛。
- 广告失败、分享失败和无填充不阻断游戏、不吞基础奖励、不重复发奖。
- 岩石、狼、羊和可行动位置的含义清楚，鼠标、触控、键盘和移动端布局一致。
- Admin 不要求管理员阅读代码；首屏能给出结论，详细数据可作为证据展开。
- Admin 模拟和仓库自动化测试使用同源规则，但结果边界和用途明确区分。
- 四季主题和碎片都有真实用途；未实现内容不以已完成能力对外承诺。
- 每个反馈项都有复现证据、归因、处理结果和回归记录。
