# Fangrush App 主线与多渠道需求总纲

> 状态：正式需求基线  
> 日期：2026-07-20  
> 覆盖：独立站 H5、Poki、CrazyGames、Android App、iOS App  
> 产品前提：本轮不增加关卡、规则、AI、多人、排行榜等核心玩法。

## 1. 本轮要解决什么

本轮不是立即开发全部渠道，而是先把五个发布端需要的非核心玩法工作规划完整，并冻结可以跨端复用的产品模型和工程边界。

目标是：同一套规则、AI、24 关、奖励与存档数据，在不同渠道只替换账号、云存档、广告、支付、生命周期和发行实现，不为任何渠道复制游戏核心。

```text
同一 game-core 与 SaveGame
        |
        +-- 独立站：自有账号 + 自有云存档
        +-- Android：自有账号 + 自有云存档 + Google 原生能力
        +-- iOS：自有账号 + 自有云存档 + Apple 原生能力
        +-- Poki：Poki 账号/云存档/广告
        +-- CrazyGames：CrazyGames 账号/数据/广告
```

## 2. 总体推荐

采用“统一内部模型、分渠道身份、分渠道存储、共用能力接口”的混合方案：

- 独立站、Android、iOS 属于自有渠道，共享 Fangrush 用户、云存档、权益和服务端 API。
- Poki、CrazyGames 默认使用各自平台账号、云存档和广告，不显示 Fangrush 的 Google/Apple 外部登录。
- 门户账号只有在平台功能已开放且确有跨渠道价值时，才通过平台 token 映射到 Fangrush 服务端；首发不把它作为前置条件。
- App 使用 Capacitor 承载现有 Web 玩家界面，继续复用 `packages/game-core`；原生能力通过插件和适配器接入。
- 自有云是独立站、Android、iOS 的进度真相；Poki/CG 平台云分别是各自渠道内的进度真相。首版不做多个云端无序双写。
- 玩家始终可以游客方式开始和离线游玩；账号用于跨设备恢复，不作为进入游戏的门槛。

这套方案是轻量单机游戏与跨平台目标之间的折中：不为门户玩家强建第二账号，也不把自有 App 的长期数据完全锁在 Google 或 Apple 单一生态中。

## 3. 已冻结决策

| ID | 决策 | 结论 |
|---|---|---|
| D-01 | 核心复用 | 五端共享 `packages/game-core`、关卡、AI、奖励模型和 `SaveGame`。 |
| D-02 | App 技术路线 | 首选 Capacitor + 独立移动壳；出现已证实的性能瓶颈后才评估重写。 |
| D-03 | 游客模式 | 所有端均可不登录开始；登录失败、云端不可用、广告无填充时核心流程仍可完成。 |
| D-04 | 自有账号范围 | 独立站、Android、iOS 统一为 Fangrush 用户；登录 provider 可以不同。 |
| D-05 | 门户账号范围 | Poki/CG 使用平台身份；不在门户包中直接放 Google、Apple、邮箱等外部登录。 |
| D-06 | 自有云范围 | 独立站、Android、iOS 使用同一 Fangrush 云存档 API。 |
| D-07 | 门户云范围 | Poki 用 Poki Cloud Gamesaves；CG 用 Data module。平台能力不可用时降级到平台允许的本地存储。 |
| D-08 | App 平台云 | PGS Saved Games、GameKit/iCloud 不是首版主存档；保留适配能力，后续按数据决定。 |
| D-09 | 广告 | 独立站固定为空实现；Poki/CG 只用平台 SDK；App 首版只接一个成熟网络，推荐 AdMob。 |
| D-10 | 支付 | Android 用 Google Play Billing，iOS 用 StoreKit；门户首发不自建 IAP。 |
| D-11 | 权益 | 去广告、付费皮肤等权益独立于普通 `SaveGame`，由商店凭证和服务端权益记录恢复。 |
| D-12 | 服务端 | 建独立 `fangrush-api`，自建 Google/Apple 验票和会话；按身份、存档、权益、购买、隐私拆模块。 |
| D-13 | 账号合并 | 不按相同邮箱自动合并账号；必须在已认证会话中显式绑定或走可审计恢复流程。 |
| D-14 | 安全存储 | Web 不长期把刷新凭证放 `localStorage`；App 凭证进入 Keychain/Keystore 类安全存储。 |
| D-15 | 共享运行时 | 保留现有总体分层，新增 `player-runtime`，消除 Admin/Mobile 对 Player 应用源码的依赖。 |
| D-16 | 数据库 | Supabase 只作 PostgreSQL、连接池、migration 和备份；不用 Supabase Auth/客户端数据库 API。 |

## 5. 不增加核心功能时，仍必须完成的工作

这些工作不改变玩法，但决定能否上线、能否恢复数据、能否通过审核：

1. 把玩家运行时产出为门户和 Capacitor 可使用的静态资源包。
2. 建 Android/iOS 原生工程、签名、构建、版本和发布轨道。
3. 处理前后台切换、暂停、静音、安全区、返回键、旋转、离线与升级。
4. 补齐存档版本、迁移、损坏恢复、游客转登录、多设备冲突和云同步。
5. 接账号、会话、账号绑定、登出、删除账号、删除/导出数据。
6. 接渠道广告生命周期；App 再接隐私同意、测试广告、失败恢复和去广告权益。
7. 接 App 内购、凭证校验、恢复购买、退款/撤销与审核沙盒。
8. 接崩溃监控和最小产品事件，建立可定位版本和设备的发布证据。
9. 完成隐私政策、Google Data safety、Apple App Privacy、年龄分级和 SDK 披露。
10. 准备图标、截图、预览、描述、审核说明、测试账号和支持页面。
11. 完成真机、弱网、离线、升级、广告失败、购买恢复和账号删除验收。
12. 完成 Poki Inspector、CrazyGames Portal、Play 内测、TestFlight 等渠道验证。

## 6. 优先级与并行关系

| 优先级 | 工作包 | 说明 |
|---|---|---|
| P0 | 公共契约与可发行运行时 | `platform-capabilities`、`player-runtime`、SaveGame 演进、静态包、渠道能力开关。所有后续工作的共同前置。 |
| P1 | 独立站账号与云存档 | 先验证 Fangrush 身份、游客合并、跨浏览器恢复和删除账号，给 App 复用服务端。 |
| P1-并行 | Poki/CG 广告与生命周期 | 由现有 H5 工具继续做；必须使用 P0 的公共广告和生命周期契约。 |
| P2 | Android App 基础版 | Capacitor 壳、离线、生命周期、Google 登录、自有云、内部测试，不要求首批广告/IAP。 |
| P3 | iOS App 基础版 | iOS 壳、Apple 登录、自有云、TestFlight；复用 Android 已稳定的公共能力。 |
| P4 | App 广告与 IAP | AdMob、UMP/ATT、去广告、恢复购买、服务端校验；在壳和存档稳定后接入。 |
| P5 | 门户账号与云存档增强 | Poki User/Cloud Gamesaves、CG Data；仅在权限、平台阶段和回访价值成立后启用。 |

P0 内部的 API/数据库设计可与运行时抽取并行；公共契约和静态构建就绪后，独立站云存档与 H5 广告再并行。各工作线不得创造第二套 `SaveGame`、广告回调语义或平台判断方式。

## 7. 本轮明确不做

- 不新增关卡、棋子、规则、AI、多人、聊天、排行榜、成就或推送运营。
- 不做订阅、Battle Pass、付费解锁主线或影响胜负的商品。
- 不为了“看起来统一”强迫 Poki/CG 玩家注册 Fangrush 账号。
- 不同时接多家 App 广告聚合平台。
- 不让客户端购买回调直接永久授予付费权益。
- 不同时把自有云、PGS、iCloud 都设为可写主存档。

## 9. 需求变更规则

- 平台政策以提交时的官方英文文档为准；本地摘要只用于规划和验收。
- 改变 D-01 至 D-16 任一决策时，必须同步更新受影响的专题文档和任务依赖。
- 新渠道只新增 adapter、构建配置和发行材料，不得修改 `game-core` 来迎合 SDK。
- 具体 SDK 版本、商店表单字段和密钥不在本文固化，由实现任务与 `distribution/` 维护。
