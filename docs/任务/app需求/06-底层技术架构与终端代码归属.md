# Fangrush 底层技术架构与终端代码归属

> 状态：开发目标架构  
> 本文回答三个问题：现有结构要不要推倒重来、底层公共能力放在哪里、每个终端的代码究竟由谁维护。

## 1. 架构结论

现有 monorepo 的总体方向正确，不需要第二次大改造：

- `packages/game-core` 已经承担规则、AI、关卡、奖励和存档数据模型，边界清楚，继续保留。
- `apps/player-web` 与 `apps/admin` 已分成独立应用，部署和公开产物门禁已建立，继续保留。
- `packages/web-assets` 已是资源唯一来源，继续保留。
- `distribution/` 已承接渠道材料和合规证据，继续保留。
- 根级 release gates 已覆盖核心测试、玩家/Admin 构建和平台边界，应该扩展而不是重写。

进入 App、账号和云存档阶段后，需要一次“第二阶段基础设施补强”，规模属于中等结构调整，不是推倒重建：

1. 新增 `packages/platform-capabilities`，承接跨 Web/App 的能力契约。
2. 新增 `packages/player-runtime`，承接真正跨 Player/Admin/Mobile 复用的玩家运行时。
3. 新增 `packages/api-contracts`，承接客户端与服务端共享的 API schema。
4. 新增 `apps/fangrush-api`，承接自建认证、云存档、权益和购买验证。
5. 新增 `apps/mobile`，一个 Capacitor 工程同时维护 Android/iOS。
6. 新增根级 `supabase/`，只维护 PostgreSQL schema、migration 和本地 seed。

不需要调整的部分：规则/AI、24 关内容、奖励设计、Admin 业务页面和现有独立站页面信息架构。

## 2. 现有结构审核

### 2.1 已经做对的部分

| 区域 | 审核结论 | 保留原因 |
|---|---|---|
| `packages/game-core` | 良好 | 无 React、Next、浏览器和平台 SDK；测试规模充分，是正确的核心唯一来源。 |
| `apps/player-web` | 良好但承担偏多 | 玩家页面、运行时、平台雏形和本地存档集中，当前 H5 可维护。 |
| `apps/admin` | 分离正确 | 独立 Next 应用、独立部署、公开产物门禁已建立。 |
| `packages/web-shared` | 起步正确 | 已集中 locale、广告和存储契约，但内容开始超出“Web shared”命名边界。 |
| `packages/web-assets` | 良好 | 正式资源只有一个源目录，Player/Admin 构建复制的是产物，不是第二套源文件。 |
| 构建/CI | 良好 | `release:check`、source boundary、artifact audit 和 target manifest 已形成可扩展基础。 |

### 2.2 进入多终端前必须解决的问题

| 问题 | 当前证据 | 风险 | 处理 |
|---|---|---|---|
| Admin 直接引用 Player 应用内部源码 | `apps/admin/next.config.ts` 和 `tsconfig.json` 为 Player 组件/lib 建了十余个 alias | Mobile 加入后会形成多个应用依赖另一个应用源码，重构和构建容易漂移 | 抽 `player-runtime`，Admin 改依赖 package |
| 平台契约放在 `web-shared` | `PlatformKind/AdsContract/StoragePort` 在 `packages/web-shared` | Mobile 被迫依赖 Web 命名和 Web-only 内容 | 迁移到 `platform-capabilities` |
| Player 页面与 Next 服务端能力耦合 | middleware、`headers()`、`cookies()`、`next/font`、SEO route handlers | 当前 `next build` 不是可直接给门户/Capacitor 的纯静态 bundle | 先做 target 静态导出样机；运行时与 Next 壳解耦 |
| 平台对象仍是同步全局单例 | `getPlatform()` 根据 env 创建 BasePlatform | SDK 初始化、登录和云能力本质异步；测试和多壳注入困难 | 改为启动时 composition root 注入 |
| Save store 直接写 localStorage | `useSaveStore.replace()` 直接调用 `persistSave()` | 无法插入 native local、CG Data 和 Fangrush Cloud 协调 | 改为 store factory + SaveCoordinator |
| Play runtime 内含页面/平台/存储依赖 | `PlayScreen` 同时处理游戏、广告、分享、存档、音频、导航 | Mobile/Admin/portal 继续 alias 会放大耦合 | 分阶段抽运行时依赖和 UI shell |
| portal 构建不是静态上传包 | `build-target.mjs` 最终仍是 `next build`，报告读取 `.next/server` | Poki/CG 上传和 Capacitor 离线包尚无真实产物 | 新增静态 bundle 和对应 audit |
| 尚无服务端/数据库迁移源 | 当前无 `apps/fangrush-api`、根级 `supabase/` | 账号、云存档和权益容易临时塞进 Player | 先建 API 和 DB 边界再做登录 UI |

## 3. 目标依赖图

```text
                         packages/game-core
                           ^          ^
                           |          |
packages/platform-capabilities     packages/api-contracts
              ^            ^          ^
              |            |          |
              +---- packages/player-runtime
                          ^   ^   ^
                          |   |   |
               apps/player-web | apps/mobile
                              |
                         apps/admin

apps/player-web ------HTTPS------+
apps/mobile ----------HTTPS------|--> apps/fangrush-api --> Supabase PostgreSQL
                                 |
Poki/CG 默认不走自有 API ---------+
仅平台明确授权映射时接入
```

依赖规则：

- `game-core` 不依赖任何其他 workspace package。
- `platform-capabilities` 只含 TypeScript 契约、通用状态和 Noop/测试实现，不依赖 SDK。
- `api-contracts` 只含请求/响应 schema、错误码和版本，不连接数据库。
- `player-runtime` 可依赖以上三个包、React 和 Zustand，不依赖 Next、Capacitor 或具体渠道 SDK。
- 应用负责选择并注入 adapter；应用之间禁止源码 import。
- `fangrush-api` 可依赖 `api-contracts`，不得依赖 `player-runtime` 或任何 UI。
- `supabase/` 只保存 SQL/config/seed，不保存业务 TypeScript。

## 5. 不同终端的代码维护位置

### 5.1 一张总表

| 终端/能力 | 主要维护位置 | 共享来源 | 禁止放入 |
|---|---|---|---|
| 核心规则、AI、24 关 | `packages/game-core` | 所有终端 | apps、SDK adapter |
| 玩家对局 UI/状态 | `packages/player-runtime` | Web、门户、App、Admin 试玩 | Next route、原生 SDK |
| 独立站 H5 | `apps/player-web/app` + `platform/standalone` | runtime/core/assets | Poki/CG/AdMob SDK |
| Poki H5 | `apps/player-web/platform/poki` + target build | runtime/core/assets | Fangrush 外部登录、CG SDK |
| CrazyGames H5 | `apps/player-web/platform/crazygames` + target build | runtime/core/assets | Fangrush 外部登录、Poki SDK |
| Android App | `apps/mobile/src/platform/android` + `apps/mobile/android` | runtime/core/API contracts | Web middleware、iOS SDK |
| iOS App | `apps/mobile/src/platform/ios` + `apps/mobile/ios` | runtime/core/API contracts | Web middleware、Android SDK |
| 自有认证/云/权益 | `apps/fangrush-api` | API contracts | Player 页面、客户端 token 信任 |
| 数据库 schema | `supabase/migrations` | API repositories | Dashboard 手工漂移、客户端 SQL |
| Admin | `apps/admin` | core + player-runtime | 玩家正式存档写入 |
| 渠道材料/证据 | `distribution/{target}` | 构建 revision | 运行时业务逻辑 |

### 5.2 独立站 H5

独立站继续由 `apps/player-web` 维护：

- Next 页面、SEO、语言路由、隐私/支持页留在 `app/`。
- Google 登录 UI 和 Fangrush API client 放 `platform/standalone` 或独立站 account feature 下。
- 本地保存使用 browser local adapter，云同步使用 Fangrush Cloud adapter。
- 广告固定选择 `NoopAdsAdapter`：不加载广告 SDK、不请求广告、不显示广告入口。
- Noop 返回 `unavailable`，业务层正常走普通奖励/继续流程。

### 5.3 Poki 与 CrazyGames

两者仍从 `apps/player-web` 构建，不复制页面：

- adapter 各自在 `platform/poki`、`platform/crazygames`。
- build target 只选择对应 adapter 和能力矩阵。
- 平台账号、云存档、广告和生命周期不进入公共 runtime。
- 若 Next 静态导出样机失败，新增 `apps/portal-web` 作为 Vite 壳；它只能消费 `player-runtime`，不能复制 Player 页面。这个分支由 BUILD-003 决策门触发，不提前创建。

### 5.4 Android 与 iOS

Android/iOS 放在同一个 `apps/mobile`，原因是：

- React 壳、页面路由、runtime、API client 和大部分生命周期协调相同。
- Capacitor 原生工程天然位于 `android/` 和 `ios/` 子目录。
- 差异通过 adapter 和原生插件隔离，不需要两个 JS 应用。

原生目录可以修改并提交，不能把它们当每次生成后丢弃的临时产物。Android Studio/Xcode 配置、entitlements、manifest、Info.plist、签名引用和原生插件都在各自子目录维护。

### 5.5 服务端与数据库

- `apps/fangrush-api` 是所有自有渠道唯一 API，不把账号 API 塞回 `apps/player-web`。
- `supabase/` 是数据库结构唯一来源；Supabase Dashboard 只做观察和紧急操作，正式变更必须回写 migration。
- 客户端从不连接 Supabase，不携带 anon key/service role，也不调用 Supabase Auth。

## 6. `player-runtime` 抽取边界

### 6.1 应迁入

- `PlayScreen`、`BoardSvg`、对局 store、active game snapshot。
- 玩家 Save store 的纯状态和 coordinator，不含具体持久化实现。
- 音频播放策略、对局反馈、分享结果格式、最小指标协议。
- Player/Admin 都使用的帮助内容和玩家文案数据。
- Admin 试玩所需的 `adminMode` 和事件回调。

### 6.2 留在 `player-web`

- Next pages/layout/metadata/middleware/route handlers。
- SEO、sitemap、robots、llms routes。
- `next/link`、`next/navigation` 具体实现。
- 独立站 Header/Footer 和网站型页面布局。
- Web cookie 语言路由、Google Web 登录 UI。
- Poki/CG SDK loader 和平台 adapter。

### 6.3 需要改造成注入的依赖

`PlayScreen` 当前直接读取平台、存档、导航、语言、音频和指标。抽取后由 `PlayerRuntimeProvider` 注入：

```ts
type PlayerRuntimeDeps = {
  target: PlatformTarget
  locale: LocalePort
  navigation: NavigationPort
  localSave: LocalSaveAdapter
  cloudSave: CloudSaveAdapter
  ads: AdsAdapter
  lifecycle: LifecycleAdapter
  audio: AudioAdapter
  metrics: MetricsAdapter
  clock: Clock
}
```

测试使用 memory/noop adapter；Admin 使用 memory save + mock ads；独立站使用 browser/Fangrush/noop；移动端使用 native/Fangrush/AdMob。

不把所有依赖做成可选值。每个 target 必须显式提供实现，即使实现是 Noop，避免运行到一半才发现 `undefined`。

## 7. 状态与存储分层

```text
game-core SaveGame update
        |
player-runtime SaveStore
        |
SaveCoordinator
   |             |
LocalSaveAdapter CloudSaveAdapter
```

- `SaveStore` 只管理内存状态和 action。
- `LocalSaveAdapter` 同步、快速、每次关键变化写入。
- `CloudSaveAdapter` 异步、可失败、带 revision。
- `SaveCoordinator` 负责 hydrate、dirty、debounce、冲突、重试和 flush。
- 购买权益由 `EntitlementStore` 独立加载，再投影到 UI，不写入 `SaveGame`。
- 当前对局 snapshot 与长期 SaveGame 继续分开。

## 8. 平台 composition root

每个应用只在一个入口组装能力：

```ts
const runtime = createPlayerRuntime({
  target,
  locale: createTargetLocale(target),
  navigation: createTargetNavigation(target),
  localSave: createTargetLocalSave(target),
  cloudSave: createTargetCloudSave(target),
  ads: createTargetAds(target),
  lifecycle: createTargetLifecycle(target),
  audio: createTargetAudio(target),
  metrics: createTargetMetrics(target),
})
```

页面、组件和 store 禁止再次读取 `NEXT_PUBLIC_PLATFORM`、`Capacitor.getPlatform()` 或 SDK 全局变量。平台判断只发生在 composition root 和构建配置。

| Target | App shell | Output | Identity | Cloud | Ads | Purchase |
|---|---|---|---|---|---|---|
| standalone | Next | Server/standalone deployment | Fangrush Google | Fangrush | **Noop** | Noop |
| poki | Next static target | Static ZIP | Poki | Poki | Poki | Noop |
| crazygames | Next static target | Static ZIP | CG | CG Data | CG | Noop |

独立站广告“保留空实现”的准确含义：契约和调用点保留，production standalone 永远绑定 Noop；以后若改变商业决策，只新增 adapter 和合规设计，不修改游戏核心和结算流程。

## 10. 环境和密钥边界

### 客户端可公开配置

- API base URL。
- Google/Apple public client ID、AdMob app/ad unit ID 等平台允许公开的标识。
- target、build revision、功能开关。

### 仅服务端/CI

- `DATABASE_URL`、migration direct URL。
- JWT/private signing keys、refresh pepper。
- Apple private key、Google/Apple 服务端验证配置。
- Play/App Store 服务账号和通知签名配置。
- Supabase database password。

### 不再需要

- 客户端 `SUPABASE_URL`、anon key。
- `SUPABASE_SERVICE_ROLE_KEY` 作为 Fangrush 运行时数据库方案。
- Supabase Auth provider 配置。

## 11. Admin 迁移策略

不能在一次提交中把 Player/Admin/平台全部重写。按以下顺序迁移：

1. 建 `player-runtime` 并先移动无 Next 依赖的 `play-store/active-game/sfx-policy`。
2. 移动 `BoardSvg`，把 locale 文案改为 props/context。
3. 移动 `PlayScreen`，注入 navigation/platform/save/audio/metrics。
4. Player 改为 package import，行为不变。
5. Admin 改为 package import，保留 `adminMode`，删除对应 webpack alias。
6. source-boundary gate 增加“任何 app 不得 import 其他 app 源码”。

每一步都要求 Player/Admin 测试和构建通过，避免一次大爆炸式迁移。

## 12. 是否需要“大调整”的最终回答

不需要改变总体分层，也不需要改写核心玩法；但如果跳过上述共享运行时抽取，后续会出现以下真实重复：

- Mobile 再复制一份 PlayScreen/store。
- Admin、Mobile 同时依赖 Player 应用内部路径。
- Web localStorage、native storage、CG Data 在 store 中堆 target 判断。
- Next 服务端能力阻挡门户静态包和 Capacitor 构建。

因此推荐的是“保留主体、补齐底座、逐步迁移”，不是“现状完全不动”，也不是“重建整个前端”。

## 13. 架构完成门槛

1. workspace 中不存在 app A import app B 源文件。
2. `player-runtime` 在无 Next 环境可以 typecheck/test/render smoke。
3. 所有 target 通过同一能力契约创建 runtime。
4. standalone production 明确绑定 Noop ads，产物无广告 SDK。
5. portal 静态产物不需要 `.next/server`。
6. mobile Web bundle 不包含 middleware、SEO route 和门户 SDK。
7. 客户端产物不含数据库连接、Supabase key 或服务端私钥。
8. Player/Admin 现有功能和 24 关核心测试无回归。

## 14. 影响与长期性复核

本次调整不是简单移动目录。候选运行时代码约 3,528 行，现有 Admin 还通过 17 个 webpack alias 和 19 个 TypeScript 路径映射依赖 Player 源码。因此影响等级定义为：

- `game-core`：低影响，规则/AI/关卡不重写。
- 现有 H5：中高结构影响、低预期行为影响。
- Admin：中等结构影响、低预期业务影响。
- 构建/CI：中高影响，需要新增静态产物、依赖边界和双端回归门禁。
- API/Mobile：新增能力，不应在启用前改变存量端行为。

`player-runtime` 不能把现有 918 行 `PlayScreen` 原样搬入。应先用特征测试冻结行为，再拆分副作用编排、展示和平台能力；迁移期保留可删除的 compatibility facade，Player/Admin 分步切换。

长期适用范围、主流方案依据、风险分级和技术债务退出条件见 [09-架构调整影响与长期技术债务评估.md](./09-架构调整影响与长期技术债务评估.md)。该评估是 P0 架构实施的放行依据。
