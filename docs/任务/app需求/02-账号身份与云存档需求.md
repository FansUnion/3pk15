# Fangrush 账号、身份与云存档需求

> 本文定义产品行为和跨端契约，不固定某个 OAuth UI 库或云厂商 SDK 版本。

## 1. 目标与边界

目标：玩家不登录也能完整游玩；登录后可在自有渠道跨浏览器、Android 和 iOS 恢复进度；门户端使用平台账号和平台云，首版不承诺与自有渠道互通。

账号系统负责“玩家是谁”，云存档负责“进度如何恢复”，权益系统负责“玩家买过什么”。三者必须关联，但不能混为一张 JSON。

本轮不建设邮箱密码、好友、社交关系、排行榜、实时经济和多人会话。

## 2. 账号模型

### 2.1 内部用户与外部身份

```text
users.id                         Fangrush 内部稳定用户
  +-- auth_identities/google     Google 身份
  +-- auth_identities/apple      Apple 身份
  +-- auth_identities/poki       可选的平台映射
  +-- auth_identities/crazygames 可选的平台映射
  +-- player_saves               自有渠道云存档
  +-- entitlements               去广告/付费皮肤等权益
  +-- purchases                  商店交易与校验状态
```

要求：

- `provider + provider_user_id` 全局唯一。
- `provider_user_id` 使用服务端验证后的稳定 subject/player ID，不使用昵称或 email。
- email 只作联系信息或辅助展示，不是账号合并主键。
- Apple 隐藏邮箱、Google 邮箱变化、门户匿名状态都不得导致新建错误映射。
- 商店购买账号、Game Center/PGS 平台身份和 Fangrush 用户是不同概念，必须显式关联。

### 2.2 账号状态

| 状态 | 含义 | 可用能力 |
|---|---|---|
| `guest` | 只有设备本地 ID 和本地存档 | 完整游戏、本地恢复 |
| `authenticated` | 已验证一个自有渠道 provider | 云同步、跨设备、账号设置 |
| `linking` | 正在绑定第二 provider | 原账号继续有效，不提前迁移 |
| `conflict` | 待处理的账号/存档冲突 | 保留两边数据，禁止静默覆盖 |
| `deletion_pending` | 删除申请已确认 | 禁止新云写入；按政策完成删除 |
| `deleted` | 身份和非保留数据已删除 | 回到游客状态 |

登录、云同步或删除失败都不得影响本地游客继续游玩。

## 3. 各端登录需求

### 3.1 独立站 H5

首期：Google 登录 + 游客模式。

- 登录入口放在设置、云存档或已有进度的自然节点，不首屏强制弹出。
- 使用 Google 当前推荐的 Web 身份流程；服务端验证签名、issuer、audience、expiry、nonce/state。
- 登录成功后创建 Fangrush 会话，再进行游客存档与云存档合并。
- 后续为与 iOS 用户形成完整跨端入口，可增加 Sign in with Apple Web。

### 3.2 Android App

- 游戏内主账号使用 Google 身份能力换取 Fangrush 会话。
- Play Games Services v2 平台认证可用于未来成就、平台资料或 Saved Games，但不直接等同 Fangrush 主账号。
- Google provider 验证后映射同一 `auth_identities/google`，使 Android 与独立站可共享用户。
- 登录按钮不可成为首局前置；无 Google 服务、取消登录、弱网时继续本地模式。

### 3.3 iOS App

- 首版提供 Sign in with Apple 和游客模式。
- 若 iOS 同时提供 Google 社交登录，必须同时提供满足 Apple 审核规则的等价登录选项；产品上直接保留 Sign in with Apple。
- Apple identity token/authorization code 由服务端验证并映射到 `auth_identities/apple`。
- 用户首次提供的 name/email 可能以后不再返回，服务端只能在首次授权时按最小必要原则保存。

### 3.4 Poki

- 只使用 Poki User Accounts 登录 UI，不显示 Fangrush Google/Apple/邮箱登录。
- `login()` 只能由用户操作触发；登录成功可能整页刷新，初始化后必须重新查询用户状态。
- 默认由 Cloud Gamesaves 透明同步平台存档。
- 若后续接自有服务端，只把短时 Poki token 发到服务端；服务端调用官方验证端点换取稳定、游戏专属 `user_id`。
- 未获得 User Accounts/API key 能力时，账号 UI 隐藏并回退本地存档。

### 3.5 CrazyGames

- 使用 CrazyGames User；游客可玩。
- 首选 Data module 处理游客本地和登录用户云同步。
- 若已有自有后端并需要映射，使用 User token/官方账户链接流程；不自绘绕过平台的授权弹窗。
- 不在 CG 包中提供独立 Google、Facebook、Apple、邮箱登录主路径。

## 4. 会话与凭证安全

### 4.1 Web

推荐：短期访问 token 只放内存；刷新凭证使用 `Secure`、`HttpOnly`、合理 `SameSite` 与限定 path/domain 的 cookie。若部署拓扑必须跨站，再单独设计 CORS、CSRF、cookie 和浏览器第三方存储策略，不把长期 refresh token 放入 `localStorage` 规避问题。

要求：

- 全链路 HTTPS。
- provider token 只用于换取 Fangrush 会话，不写日志、不进入 URL、不长期保存。
- 刷新 token 轮换并支持撤销；检测重放时撤销会话族。
- 访问 token 短期有效；服务端按 `sub` 取用户，不相信客户端传入 `userId`。
- 认证接口限流；关键失败写安全审计，不记录完整 token。

### 4.2 App

- refresh/session 凭证存 Keychain/Keystore 类安全存储，不存普通 Web Storage。
- WebView JavaScript 只能通过受限桥接请求会话能力，不暴露原生商店密钥或长期凭证。
- App 删除、账号删除、显式登出时清理相应本地凭证。
- 调试日志、崩溃报告和分析事件必须过滤 token、购买凭证、email 和完整存档。

### 4.3 对旧 `P-ACCOUNT.md` 的修正

旧文档可继续作为 StartUpSense 代码地图，但正式实现按本文修正：

- 禁止按相同 email 自动绑定已有用户。
- 不把长期有效访问/刷新凭证放 `localStorage`。
- 不默认信任通用 Google access token；采用当前官方身份流程并在服务端完整验证。
- 不让 portal 构建包含独立站登录入口或自有 provider 配置。
- 付费权益不进入普通 SaveGame 合并。

## 5. 账号生命周期需求

### 5.1 注册/首次登录

1. 玩家以游客状态已有本地进度。
2. 用户主动选择 provider。
3. 客户端完成 provider 授权，服务端验证凭证。
4. 服务端按 provider subject 查找或创建 `users.id`。
5. 客户端取得 Fangrush 会话。
6. 拉取云存档，按第 8 节合并，写回本地和云端。
7. UI 显示同步完成；失败时本地进度保留并允许重试。

### 5.2 登录已有账号

- 云端不存在：把经过迁移和校验的本地存档作为首个云版本。
- 本地不存在：恢复云端到本地。
- 两边都存在：执行确定性合并，不静默整包覆盖。
- 云端 schema 高于客户端支持版本：停止上传，提示升级 App/刷新页面，不降级覆盖。

### 5.3 绑定第二 provider

- 只有已登录用户可发起。
- 服务端对当前会话和新 provider 都进行验证。
- 新 provider 未占用：绑定到当前 `users.id`。
- 新 provider 已绑定当前用户：幂等成功。
- 新 provider 已绑定另一用户：进入冲突流程，保留两账号，不自动合并。

### 5.4 登出

- 撤销当前会话并清除凭证。
- 本地存档不删除，玩家继续以游客方式玩。
- 登出后的新本地进度不自动写入旧账号；下次登录时重新合并。
- Poki/CG 的平台登出行为由 SDK 管理，本游戏监听状态并切换对应存档域。

### 5.5 删除账号

- 独立站提供 Web 删除入口；Android 同时提供 App 内入口和 Web 入口；iOS 提供 App 内发起入口。
- 删除前明确展示影响，进行重新认证或等价确认。
- 删除身份映射、云存档、非法律保留的权益关联和个人资料；交易法定留存字段最小化并去关联化。
- 撤销 Sign in with Apple 等 provider token（适用时），撤销所有 Fangrush 会话。
- 完成后客户端回到游客状态；不得自动重新创建并恢复已删除云数据。
- 若门户用户从未映射自有服务端，账号删除由门户负责；若存在映射，需提供删除该映射和自有数据的入口。

## 6. 云存档数据契约

`SaveGame` 继续是游戏进度内容；同步元数据放在外层，避免污染规则核心：

```ts
type CloudSaveEnvelope = {
  schemaVersion: number
  revision: number
  updatedAt: string
  updatedByDeviceId: string
  checksum: string
  save: SaveGame
}
```

要求：

- `revision` 由服务端单调递增。
- 客户端写入携带已知 `baseRevision`，服务端用乐观并发控制检测冲突。
- `checksum` 用于发现传输/存储损坏，不作为安全签名替代品。
- 单个存档设置远低于门户 1MB 限制的产品预算，建议 JSON 未压缩不超过 128KB。
- 每次读取先走 `migrate()`，迁移失败保留原始副本和诊断信息，不直接重置云端。
- 云端至少保留最近若干历史 revision 供客服恢复；保留数量和期限由运维策略确定。

## 7. 保存时机与离线策略

本地保存优先，云同步异步：

- 完成关卡、领取奖励、解锁/装备皮肤、任务领取、设置变化时立即写本地。
- 云写入 debounce，并在 App 进入后台、页面隐藏、正常退出前尽力 flush；不能依赖退出事件一定成功。
- 离线时记录待同步 revision/脏标记，不阻塞游戏。
- 恢复网络后先拉云、比较 revision，再上传；禁止盲目覆盖。
- 同一字段变化的重复请求必须幂等。
- 广告回调、购买回调和存档写入失败不能把基础进度回滚。

CrazyGames 使用 Data module 时，读写统一经过 Data module，不再旁路写第二份普通 localStorage 主存档；Poki 使用其透明 Cloud Gamesaves 时，遵守平台同步键和忽略键规则。

## 8. 冲突合并规则

合并函数必须是纯函数、交换规则明确、同输入同输出，并有 fixture 单测。

| 数据 | 合并规则 |
|---|---|
| `clearedLevels` | 合法关卡 ID 并集 |
| `unlockedChapters` | 根据合并后的通关重新计算，不直接信任客户端数组 |
| 免费 `unlockedSkinIds` | 合法免费解锁并集；付费皮肤由 entitlement 注入 |
| `fragments` | 不相加；首版取经过上限校验的较大值，未来可升级为幂等奖励账本 |
| `equipped` | 对应皮肤有效且已拥有时，按字段更新时间取新值 |
| `guide` | 完成布尔值取 OR；提示次数/失败次数按语义取最大或最新周期值 |
| `settings` | 按字段更新时间取新值；设备专属设置可声明不同步 |
| `quests` | 周期 key 相同才合并；进度取最大，claimed 并集；周期不同按当前有效周期处理 |
| `lastPlayedLevelId` | 取最近有效记录，不影响解锁真相 |
| `entitlements` | 不从 SaveGame 合并；查询服务端权益后投影到客户端 |

如果未来加入可购买货币、排行榜或竞争数据，现有轻量合并不再满足安全要求，必须改为服务端权威事件与防重放账本。

## 9. 自有服务端最小 API

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/v1/auth/google/exchange` | 验证 Google 身份并建立 Fangrush 会话 |
| `POST` | `/v1/auth/apple/exchange` | 验证 Apple 身份并建立 Fangrush 会话 |
| `POST` | `/v1/auth/refresh` | 轮换会话 |
| `POST` | `/v1/auth/logout` | 撤销当前会话 |
| `GET` | `/v1/me` | 当前用户、provider 与同步能力 |
| `POST` | `/v1/me/identities/link/*` | 显式绑定第二 provider |
| `DELETE` | `/v1/me/identities/{provider}` | 在保留至少一种恢复方式时解绑 |
| `GET` | `/v1/save` | 读取云存档 envelope |
| `PUT` | `/v1/save` | 基于 `baseRevision` 写入；冲突返回当前云版本 |
| `GET` | `/v1/entitlements` | 获取有效权益 |
| `POST` | `/v1/privacy/export` | 请求导出数据 |
| `POST` | `/v1/privacy/delete` | 发起账号删除 |

购买校验 API 见 App 文档。接口统一带版本、请求 ID、限流、结构化错误和可审计日志。

## 10. 最小数据表

| 表 | 核心字段 |
|---|---|
| `users` | `id`, `status`, `created_at`, `deleted_at` |
| `auth_identities` | `user_id`, `provider`, `provider_user_id`, 最小 profile，唯一约束 |
| `sessions` | token hash/family、device、expiry、revoked_at |
| `player_saves` | `user_id`, `revision`, `schema_version`, `save_json`, checksum、timestamps |
| `save_revisions` | 历史 revision、原因、保留期限 |
| `entitlements` | key、source、status、有效期、更新时间 |
| `purchases` | platform、product、transaction、original transaction、verification status |
| `privacy_requests` | 类型、状态、确认和完成时间 |
| `audit_events` | 安全/购买/恢复事件，不存敏感原文 |

门户平台若只使用平台云，不需要写入这些表。只有启用平台 token 映射时才新增对应 `auth_identities`。

## 11. 验收场景

### 11.1 自有渠道

1. 游客可完成全部 24 关并在重启后本地恢复。
2. 浏览器 A 登录并上传后，浏览器 B 同 provider 登录可恢复。
3. Android 与独立站使用同一 Google 身份可落到同一 Fangrush 用户。
4. iOS Apple 用户可显式绑定 Google，绑定前后数据不丢失。
5. 两设备离线产生不同进度后上线，领域合并结果符合表格且 revision 连续。
6. 旧 schema 自动迁移；新 schema 遇旧客户端不会被旧版覆盖。
7. 云端 401、403、409、429、500、超时、弱网时本地仍可玩并可重试。
8. 登出后本地仍可玩，新进度不会静默写入旧账号。
9. 删除账号后所有会话失效，云存档不可恢复，用户回到游客状态。
10. 日志、分析和崩溃报告中无 token、购买凭证和完整个人数据。

### 11.2 门户渠道

1. 门户包不存在 Fangrush Google/Apple 登录入口和相关密钥。
2. 未登录平台账号时可玩并本地保存。
3. 登录平台账号后按 SDK 规则迁移/加载，不覆盖已有较新进度。
4. 平台账号能力未开放、用户拒绝、AdBlock、私密模式时安全降级。
5. Poki 存档和 CG Data 都低于平台限制，并通过平台测试工具验证。

## 12. 完成定义

账号与云存档模块只有在以下条件同时满足时才算完成：

- 产品流程、API、数据库迁移、provider 配置和安全审计齐全。
- 游客、首次登录、重复登录、绑定、冲突、登出、删除全部有自动测试或可重复验收证据。
- 至少完成两个独立浏览器和 Android/iOS 各一台真机的跨端恢复。
- portal 构建审计证明不存在外部登录和自有会话泄漏。
- 隐私政策、数据披露和删除入口与真实实现一致。
