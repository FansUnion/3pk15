# Fangrush 多渠道技术架构、实施计划与验收

> 本文是技术执行总览。开发级代码归属、数据库/API 设计和逐项任务依赖分别以 `06`、`07`、`08` 为准；它不要求本轮立即实现代码。

## 1. 架构原则

1. 核心规则只有一份，任何 SDK 都不能进入 `packages/game-core`。
2. `SaveGame` 内容只有一份，存储位置由 provider 决定。
3. 本地保存、云保存、账号、广告、支付、生命周期分别建契约，不做一个万能平台对象。
4. SDK adapter 可以不同，产品状态和错误语义必须相同。
5. 编译目标显式声明能力，不通过运行时猜 URL 或只隐藏 UI。
6. 无账号、无网络、无广告、无支付时核心游戏仍完整。
7. 服务端不信任客户端 user ID、付费权益和商店回调。
8. 每个发布包有可机器检查的内容边界和可人工复验的平台证据。

## 2. 推荐仓库结构

```text
packages/game-core
  rules / ai / levels / rewards / SaveGame / migrations / merge pure functions

packages/platform-capabilities
  identity / local-storage / cloud-save / ads / purchase / lifecycle contracts
  shared result types, error codes and target capability matrix

packages/api-contracts
  versioned auth / save / entitlement / purchase / privacy request-response schemas

packages/player-runtime
  shell-neutral React gameplay UI, stores and save coordination

packages/web-shared
  Web-only shared UI contracts, locale and styles

packages/web-assets
  shared visual/audio assets and asset gates

apps/player-web
  standalone Next application
  static portal targets: poki / crazygames
  Web target adapters

apps/mobile
  Capacitor shell
  Android and iOS native projects
  native bridge adapters

apps/fangrush-api
  auth / users / saves / entitlements / purchases / privacy

supabase/
  PostgreSQL config / migrations / local seed; no Supabase Auth dependency

apps/admin
  existing diagnostics and QA; never shipped to players

distribution/
  platform submissions, compliance, assets and evidence
```

### 2.1 为什么新增 `platform-capabilities`

当前身份、存储和广告契约部分放在 `web-shared` 与 `player-web/lib`，未来 `apps/mobile` 若直接依赖它们会形成“移动端依赖 Web 应用实现”的反向边界。进入 App 主线后，抽出只含 TypeScript 类型和纯契约的小包已有实际价值。

这个包不包含 Poki、CG、AdMob、StoreKit 或 Google SDK 代码，也不保存密钥。

### 2.2 `player-runtime` 调整结论

当前 Admin 已通过十余个 alias 直接消费 Player 的组件和 lib；Mobile 加入后，共享运行时已经达到独立 package 的真实门槛。P0 抽取 `packages/player-runtime`，由 Player、Admin、Mobile 共同消费，禁止应用之间继续源码 import。

Next portal 静态导出仍需单独做样机。若抽取 runtime 后双模式依然困难，再增加轻量 portal 壳，而不是撤销 runtime 或复制页面。

## 3. 能力契约

### 3.1 目标能力

```ts
type PlatformTarget =
  | 'standalone'
  | 'poki'
  | 'crazygames'
  | 'android'
  | 'ios'

type TargetCapabilities = {
  identity: 'fangrush-google' | 'fangrush-apple' | 'poki' | 'crazygames' | 'none'
  localStorage: 'browser' | 'crazygames-data' | 'native'
  cloudSave: 'fangrush' | 'poki' | 'crazygames' | 'none'
  ads: 'admob' | 'poki' | 'crazygames' | 'none'
  purchases: 'google-play' | 'app-store' | 'none'
  lifecycle: 'browser' | 'poki' | 'crazygames' | 'native'
}
```

能力由构建目标生成只读配置，并在 CI 检查非法组合，例如 Poki + AdMob、CG + Google 外部登录、standalone + portal ads。

### 3.2 拆分接口

```ts
interface IdentityAdapter {
  getState(): Promise<IdentityState>
  signIn(): Promise<AuthResult>
  signOut(): Promise<void>
  getProviderToken?(): Promise<string | null>
}

interface LocalSaveAdapter {
  load(): SaveGame | null
  save(value: SaveGame): void
  remove(): void
}

interface CloudSaveAdapter {
  getAvailability(): Promise<CapabilityState>
  load(): Promise<CloudSaveEnvelope | null>
  save(input: CloudSaveWrite): Promise<CloudSaveResult>
  flush?(): Promise<void>
}

interface AdsAdapter {
  getAvailability(kind: 'interstitial' | 'rewarded'): Promise<CapabilityState>
  showInterstitial(input: AdRequest): Promise<AdResult>
  showRewarded(input: RewardedAdRequest): Promise<AdResult>
}

interface PurchaseAdapter {
  listProducts(): Promise<Product[]>
  purchase(productId: string): Promise<PurchaseResult>
  restore(): Promise<PurchaseResult[]>
  finish(transactionId: string): Promise<void>
}

interface LifecycleAdapter {
  loadingStarted?(): void
  loadingFinished(): void
  gameplayStart(): void
  gameplayStop(): void
  subscribe(event: LifecycleEvent, handler: () => void): Unsubscribe
}
```

本地存储与云存储必须拆开：当前 `StoragePort` 是同步 key/value API，适合 browser localStorage 和 CG Data；Fangrush Cloud 是异步、带 revision 的远端资源，不能硬塞进同步接口。

## 4. 各目标 adapter

| Target | Identity | Local | Cloud | Ads | Purchase | Lifecycle |
|---|---|---|---|---|---|---|
| standalone | Fangrush Google | Browser | Fangrush API | Noop | None | Browser |
| poki | Poki User | Browser/Poki keys | Poki Cloud Gamesaves | Poki SDK | None | Poki SDK |
| crazygames | CG User | CG Data | CG Data transparent sync | CG SDK | None | CG SDK |
| android | Fangrush Google | Native | Fangrush API | AdMob | Play Billing | Capacitor/native |
| ios | Fangrush Apple (+Google optional) | Native | Fangrush API | AdMob | StoreKit | Capacitor/native |

Mock adapter 只存在于开发/Admin/自动测试目标；生产 target 配置为真实 provider 但 SDK 缺失时，构建直接失败，不能静默发布 unavailable。

## 5. 玩家状态与数据流

### 5.1 游客

```text
游戏事件 -> SaveGame pure update -> LocalSaveAdapter.save
                                   -> 标记 dirty（若目标支持云）
```

### 5.2 自有账号登录

```text
provider credential
  -> fangrush-api 服务端验票
  -> Fangrush session
  -> GET cloud envelope
  -> migrate(local/cloud)
  -> deterministic merge
  -> local save
  -> PUT cloud with baseRevision
```

### 5.3 日常自有云同步

```text
local immediate save
  -> debounce queue
  -> PUT baseRevision
       success: update revision, clear dirty
       conflict: fetch server -> merge -> retry once
       offline/5xx: keep local + dirty, exponential retry
       incompatible schema: stop writes + require upgrade
```

### 5.4 门户存档

- Poki adapter 保持官方 Cloud Gamesaves 的透明键同步语义，不再向 Fangrush API 双写。
- CG 构建把 `LocalSaveAdapter` 指向 Data module；SDK 负责游客 localStorage 与登录云端切换。
- 若未来启用门户到 Fangrush API 映射，先另写 ADR 定义唯一 primary，不能直接给现有 adapter 加第二个无序写目标。

## 6. 服务端技术方案

### 6.1 选型

推荐独立 `apps/fangrush-api`，使用 Next Route Handlers 的 Node runtime、`pg` 和 Supabase 托管 PostgreSQL。Supabase 只提供数据库、连接池、migration 和备份，不使用 Supabase Auth、客户端 Data API 或 service-role DAO。选择依据是：

- 能稳定部署独立 API 域名。
- 支持事务、数据库迁移、后台任务/商店通知和结构化日志。
- provider 私钥、数据库连接、商店凭证只在服务端。
- Android/iOS/Web 共享相同版本化 API。

不推荐把长期 API 只嵌在独立站页面应用中，因为移动 App 生命周期更长，站点重构不应迫使 API URL 和会话语义一起变化。

### 6.2 模块

| 模块 | 职责 |
|---|---|
| `auth` | Google/Apple/门户 token 验证、会话签发/轮换/撤销 |
| `users` | 用户状态、provider 绑定、支持恢复 |
| `saves` | revision、schema、校验、历史和冲突 |
| `entitlements` | 去广告/付费皮肤有效状态 |
| `purchases` | Play/App Store 校验、幂等、退款/撤销 |
| `privacy` | 导出、删除、法定保留和审计 |
| `operations` | 健康检查、限流、日志、告警、备份恢复 |

### 6.3 部署与安全

- dev/staging/prod 数据库和 OAuth/商店配置分离。
- 数据库迁移向前兼容，生产变更可回滚代码但不破坏新 schema。
- 每日自动备份并定期做恢复演练；仅“有备份”不算完成。
- API 有请求 ID、速率限制、超时、输入大小限制和统一错误码。
- 审计管理员访问、账号合并、购买修复和存档回滚。
- 最小权限数据库连接；客户端永远拿不到数据库密码或商店服务端密钥。

## 7. 当前代码的具体演进

### 7.1 `packages/game-core`

当前 `SaveGame.schemaVersion` 为 1，`migrate()` 对未知版本回默认存档。进入云同步前必须：

- 增加可测试的逐版本迁移链，不直接把未知云存档重置并上传。
- 新增确定性 `mergeSaveGames`，覆盖关卡、皮肤、碎片、任务、设置和装备。
- 明确付费权益不进入普通存档。
- 为 schema、迁移、合并、重复应用和损坏输入增加 fixture 测试。
- 必要的字段更新时间可以放同步 envelope/metadata，不让规则函数依赖系统时钟。

### 7.2 `packages/web-shared`

- 把 `AdsContract`、`StoragePort`、`PlatformKind` 等平台契约迁到 `platform-capabilities`。
- `web-shared` 保留 locale、Web UI 和 Web-only 契约。
- 迁移过程保持现有 import 可过渡，完成后清除双定义。

### 7.3 `apps/player-web/lib/platform.ts`

当前 `GamePlatform` 聚合 ads/lifecycle，可作为 facade，但具体 adapter 必须组合注入：

- 不再由 `getPlatform()` 只读环境变量后创建空 BasePlatform。
- 为 Poki/CG 初始化异步状态，SDK 未准备时 UI 等待或安全降级。
- lifecycle 和 ads 不在多个 React 组件重复触发。
- 增加生产 target 缺 adapter 的启动/构建失败证据。

### 7.4 `apps/player-web/lib/storage.ts`

- 保留 browser local adapter 和异常保护。
- 把 save orchestration 与 raw key/value 分开。
- CG target 注入 Data adapter；不在函数内部写 `if (crazygames)`。
- 自有云同步由独立 async service 协调，不改变 `loadSave()` 的同步首屏能力。

### 7.5 `apps/player-web/scripts/build-target.mjs`

当前脚本只改变环境变量并执行普通 `next build`。P0 必须补：

- 静态 portal/app bundle 产物。
- target manifest 和 SDK/能力声明。
- source map、Admin、auth、API、密钥、外部域名审计。
- 包体/文件数/入口/相对路径检查。
- 在无 Next server 环境启动并跑 smoke test。

## 8. CI 与证据体系

### 8.1 每次变更

- `game-core` 规则、SaveGame、迁移、合并测试。
- platform contracts 类型与 adapter contract tests。
- standalone/poki/crazygames 静态或服务端构建。
- target 边界、外部请求、敏感字符串和产物审计。
- 玩家核心 Playwright smoke：开始、行动、结算、刷新恢复。

### 8.2 合并到发布分支前

- 独立站 auth/cloud 集成测试，使用隔离测试用户和数据库。
- Android/iOS 原生构建、签名检查和真机 smoke。
- 广告测试 provider 全结果矩阵。
- 商店 sandbox 购买、恢复、退款/撤销测试。
- 隐私披露与 SDK inventory 自动/人工对账。

### 8.3 平台外部证据

- Poki Inspector 报告/截图和事件日志。
- CrazyGames Portal Basic/Full 验证记录。
- Google Play 内部/封闭测试版本与 pre-launch report。
- TestFlight build、崩溃和 App Review notes。
- 生产 API 备份恢复演练和账号删除记录。

外部平台“提交成功”不等于功能完成；必须保留可复验的版本号、构建 revision 和结果。

## 10. 两条工作线如何避免互相踩踏

| 公共资产 | 基础/App 主线负责 | H5 广告执行线负责 |
|---|---|---|
| `AdsContract`/结果语义 | 定义、测试夹具、幂等规则 | Poki/CG 真实 adapter |
| lifecycle | 公共状态机和调用点 | 平台事件映射与 Inspector/Portal 验证 |
| target config | 类型、非法组合和构建门禁 | 填各平台能力与 SDK 配置 |
| SaveGame | schema、迁移、合并 | 不修改；只接目标 StorageAdapter |
| static bundle | 构建管线和审计 | 验证平台上传与 SDK 加载 |
| UI | 公共广告入口/暂停状态 | 不复制页面；只返回 adapter 结果 |

接口变更先改 contract test，再同步两个实现；不得靠口头约定各自解释 `cancelled/unfilled/cooldown`。

## 11. 风险与控制

| 风险 | 后果 | 控制 |
|---|---|---|
| Next 不能稳定静态导出当前路由 | 门户/App 无可打包运行时 | P0 做样机；必要时抽 player-runtime |
| 旧存档合并设计过于乐观 | 多设备丢进度/重复资产 | revision、字段级合并、fixture、历史版本 |
| 直接复制 StartUpSense auth | 带入不合适的邮箱合并/会话风险 | 只复用经安全审查的模块和范式 |
| App 是低质量网页壳 | 审核拒绝、体验差 | 本地打包、离线、原生生命周期、安全区、真机 QA |
| 多云双写 | 覆盖或循环同步 | 每 target 唯一 primary，新增映射先写 ADR |
| 门户外部请求/登录违规 | 拒审或下架 | compile-time 关闭、产物域名审计、平台 token 流程 |
| 广告/购买重复回调 | 重复发奖/权益错误 | idempotency key、服务端唯一约束、状态机测试 |
| 商店政策变化 | 延期或拒审 | 提交前按官方英文文档重新核对，不固化 SDK 版本 |
| 凭证进入 WebView/日志 | 账号和交易安全事故 | secure storage、桥接最小化、日志脱敏、密钥扫描 |

## 12. 决策门与止损

- 静态运行时样机不通过：停止 App/portal 壳扩展，先解决运行时边界。
- 独立站云同步在冲突/迁移测试未通过：不得复制到 App。
- H5 数据未达到既定投入门槛：可完成架构和内测包，但不扩大商店商业化投入。
- Android 真机显示 WebView 性能不足：先定位资源/渲染；证据确认是路线瓶颈后才评估重写。
- 广告显著降低完成率/回访：降低频率或关闭插屏，不用更多广告抵消留存问题。
- 门户账号/云能力未开放：保持本地/平台默认路径，不自建外部登录绕过。

## 13. 总体验收清单

1. 五个 target 的能力矩阵可由代码读取并由 CI 检查。
2. 五端使用同一 `game-core`、SaveGame schema 和迁移测试。
3. Poki/CG/Capacitor bundle 无服务器依赖且无目标外 SDK/密钥。
4. 独立站、Android、iOS 账号落到同一用户模型并可跨端恢复。
5. Poki/CG 在各自平台账号/云不可用时仍可本地完整游玩。
6. 广告所有结果都恢复游戏；激励和购买都幂等。
7. 付费权益来自验证记录，不来自客户端 SaveGame。
8. 账号删除、隐私披露、Data safety、App Privacy 与实现一致。
9. 真机、平台工具、商店沙盒和备份恢复都有版本化证据。
10. 任一 provider 可通过配置关闭并安全降级，不修改核心规则。
