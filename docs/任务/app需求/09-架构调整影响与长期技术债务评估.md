# Fangrush 架构调整影响与长期技术债务评估

> 状态：架构实施前评审结论  
> 评估范围：现有独立站 H5、Poki/CrazyGames 构建方向、Admin、未来 Android/iOS、自建 API 与 Supabase PostgreSQL。  
> 本文用于回答：改动有多大、是否值得做、能否长期迭代、哪些债务必须现在控制。

## 1. 最终结论

本次方案可以作为长期目标架构，建议继续执行，但要把它定义为“中等规模、分阶段、行为保持型的架构迁移”，不能当成简单目录整理。

结论分四层：

1. **现有总体分层不推倒**：`game-core`、Player/Admin 独立应用、资源唯一来源和发布门禁都应保留。
2. **存量 H5 结构影响中高，产品行为影响应为低**：需要迁移约 3,528 行候选运行时代码并解除平台、存储、导航和语言依赖，但页面信息架构、玩法和关卡不重写。
3. **Admin 结构影响中等，业务影响应为低**：当前 17 个 webpack alias、19 个 tsconfig Player 路径映射需要退出；关卡编辑、AI 工具和试玩行为保留。
4. **目标架构符合主流工程实践**：它是 monorepo 内的模块化单体、端口/适配器、显式 composition root 和共享运行时，不是微服务化，也不是为架构而架构。

长期可支持的产品范围：单机核心玩法、多渠道发行、账号与云存档、广告/IAP、皮肤/任务/关卡内容增长、运营后台。实时联机、服务器权威战斗、开放 UGC/聊天不在本架构承诺内，若以后确定进入这些方向，需要单独扩展服务端领域，而不是继续向当前 runtime 塞逻辑。

## 2. 评估依据

### 2.1 当前代码量与耦合证据

本次不是全量前端重写。初步候选抽取范围共约 3,528 行：

| 区域 | 当前行数 | 主要问题 |
|---|---:|---|
| `PlayScreen.tsx` | 918 | 同时处理对局、奖励、广告、分享、存档、音频、指标、导航和 AdminMode |
| `BoardSvg.tsx` | 450 | 可复用，但文案/交互上下文仍来自 Player |
| play/save/active-game/storage | 562 | store 与 browser storage、全局状态耦合 |
| platform/ads/audio/metrics 等 | 660 | 平台全局对象和具体运行环境边界不清 |
| 玩家文案、帮助与 guidance | 938 | Player/Admin 共享，但需要分清运行时文案与 Web 路由文案 |

这些数字是“需要逐项判定、拆分或迁移的候选范围”，不是一次性机械搬家的工作量。最终进入 `player-runtime` 的内容必须满足至少两个实际消费者且不含具体应用壳依赖。

Admin 当前通过 17 个 webpack alias 和 19 个 TypeScript 路径映射指向 `apps/player-web`。这证明共享 package 已有真实需求，也说明直接删除 alias 会产生集中回归风险。

### 2.2 方案依据

- Capacitor 官方将其定位为可加入现有现代 JavaScript 项目的跨 iOS/Android 原生运行时，符合当前 Web-first 团队复用 React 运行时的选择：[Capacitor 文档](https://capacitorjs.com/docs)。
- Supabase 提供完整 PostgreSQL、直接连接与连接池，支持服务端用标准 PostgreSQL client 访问；本项目只采用该数据库能力，不把 Auth/Data API 放进客户端：[Supabase 数据库概览](https://supabase.com/docs/guides/database/overview)、[连接方式](https://supabase.com/docs/guides/database/connecting-to-postgres)。
- Supabase 官方 migration 工作流以 `supabase/migrations` 为结构来源，符合 SQL-first 和可重放要求：[数据库迁移](https://supabase.com/docs/guides/deployment/database-migrations)。
- 自建会话不是低成本能力。OWASP 明确指出认证、会话和访问控制的绑定复杂且容易导致会话劫持，因此它是本方案最大的主动技术责任：[Session Management Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Session_Management_Cheat_Sheet.html)、[Authentication Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Authentication_Cheat_Sheet.html)。

## 3. 对存量 H5 的影响

### 3.1 影响等级

| 范围 | 影响 | 是否改用户行为 | 说明 |
|---|---|---|---|
| 24 关规则、AI、胜负判定 | 低 | 否 | 继续由 `game-core` 提供，只补 SaveGame migration/merge |
| 对局 UI 与状态 | 中高 | 否 | 从 Player 内部实现变成 `player-runtime` 公共实现 |
| 本地存档 | 中 | 否 | 从直接 localStorage 改为 `LocalSaveAdapter + SaveCoordinator` |
| 页面、SEO、语言路由 | 低 | 否 | 继续留在 Next `apps/player-web` |
| 平台初始化与广告 | 中 | 有意区分渠道 | standalone 绑定 Noop；Poki/CG 各自 adapter |
| 构建发布 | 中高 | 否 | 现有 portal 产物仍依赖 `.next/server`，必须验证纯静态 ZIP |
| 账号与云存档 | 新增 | 是 | 只为独立站/App 接 Fangrush API；门户按平台政策处理 |

### 3.2 最大回归面

迁移最容易破坏的不是棋盘规则，而是边缘时序：

- 首次 hydrate 与页面首帧之间是否覆盖本地存档。
- 广告开始、暂停、失败、奖励和恢复回调是否只结算一次。
- App/网页切后台时当前对局与长期存档是否及时 flush。
- AdminMode 是否仍使用 memory/mock adapter，且不写正式玩家存档。
- locale、返回路径、下一关跳转和分享链接是否仍由应用壳控制。
- 静态渠道包是否意外带入独立站登录、服务端路由或错误平台 SDK。

因此 H5 的验收不能只依赖 unit test，必须保留迁移前后的页面截图、关键流程 E2E、24 关 smoke 和产物审计。

### 3.3 不应发生的变化

- 不重做 UI，不同时改关卡和奖励数值。
- 不改变现有存档字段语义；演进只能通过版本化 migration。
- 不把 Next 页面全部搬进 package。
- 不为 Poki/CG 各复制一份 Player。
- 不在公共组件内新增 `if (target === ...)` 分支。

## 4. 对 Admin 的影响

### 4.1 影响等级

Admin 不需要重写页面和工具，但必须改变依赖方式：

| 区域 | 影响 | 处理 |
|---|---|---|
| Admin 页面/权限壳 | 低 | 保留在 `apps/admin` |
| 关卡与 AI 工具 | 低 | 继续直接消费 `game-core` |
| Admin 试玩 | 中高 | 从 Player alias 改消费 `player-runtime` 公共 API |
| mock 广告、memory save、localeOverride | 中 | 变成 Admin composition root 注入的 adapter |
| webpack/tsconfig alias | 中 | 逐项退出，最后由 source-boundary CI 禁止回流 |
| Admin 构建部署 | 低到中 | package build 成为新前置，部署形态不变 |

### 4.2 Admin 专项控制线

- `player-runtime` 不得知道“Admin 应用”，只提供 `readOnly/persistence/telemetry/navigation` 等通用能力和试玩事件。
- `adminMode` 可以作为迁移期兼容参数，但不能继续扩散成几十个条件分支；完成迁移后应由能力组合表达差异。
- Admin 只能通过 package public exports 使用运行时，不得引用 package 内部文件。
- Admin 的 memory save 和 mock ads 必须有独立 contract tests，防止误写真实云存档或触发正式 SDK。
- alias 逐项移除；在 Player/Admin 双构建都通过之前保留可回退的兼容 facade。

## 5. 是否支持长期迭代

### 5.1 能稳定支持的变化

| 未来变化 | 目标架构如何承接 |
|---|---|
| 新增关卡、AI、规则修正 | `game-core` 内容和版本化测试 |
| 新增 H5 渠道 | 新 adapter、composition root、target manifest 和发行证据 |
| Android/iOS 差异 | `apps/mobile/android`、`ios` 与平台 adapter |
| 更换广告/IAP SDK | 替换 adapter，不改 `game-core/player-runtime` |
| 新增登录提供方 | API identity provider + client adapter，不改变内部 user ID |
| 云存档字段演进 | SaveGame migration、merge policy、API schema version |
| 新增皮肤/任务/权益 | 游戏存档与服务端 entitlement 分离演进 |
| Supabase 迁出 | 标准 PostgreSQL migration/repository；应用不直接依赖 Supabase SDK |

### 5.2 明确不适用的边界

以下需求出现时，需要新一轮架构评审：

- 实时 PVP、房间、匹配、观战。
- 服务端权威战斗或强反作弊经济系统。
- 大规模 UGC、聊天、内容审核和社交图谱。
- 需要独立扩缩容的高吞吐领域服务。

这不是当前方案的缺陷。以目前游戏规模提前建设微服务、实时同步和复杂事件平台，反而会制造更大的运维和一致性债务。

## 6. 主流性判断

| 选择 | 判断 | 理由 |
|---|---|---|
| monorepo + workspace packages | 主流且适合 | 多应用共享 core/runtime/contracts，原子变更和统一 CI 有明显收益 |
| 模块化单体 API | 主流且适合 | 当前团队和业务规模不需要微服务；事务、部署、观测更简单 |
| ports/adapters + composition root | 主流且必要 | 平台 SDK、存储、广告、身份天然存在多实现 |
| React runtime + 薄 Web/Mobile 壳 | 适合当前项目 | 最大化复用现有玩法 UI，原生差异留在 Capacitor adapter |
| SQL-first PostgreSQL migration | 主流且可迁移 | schema 可审查、可重放，不绑定 ORM 或 Supabase 专有 API |
| 自建 Google/Apple 身份映射与 session | 可行但风险高于托管 Auth | 满足“不依赖 Supabase Auth”，同时承担完整安全与运维责任 |
| 每个渠道独立复制应用 | 不推荐 | 会造成修复、存档和 UI 长期漂移 |
| 现在拆微服务 | 不推荐 | 没有独立团队、吞吐或部署边界支撑其成本 |

所以，本方案不是激进架构，而是偏保守的模块化单体。偏离最常见小团队方案的地方只有“自建 session”，这一点必须按安全关键模块管理。

## 7. 技术债务清单

### 7.1 必须在 P0 控制

| 风险 | 等级 | 会形成的债务 | 当前措施 | 放行条件 |
|---|---|---|---|---|
| `player-runtime` 变成新的大杂烩 | 高 | 所有端都依赖一个无法修改的巨型包 | public API、分 gameplay/progression/runtime、依赖门禁 | 不含 Next/Capacitor/SDK；无 target 条件散落 |
| `PlayScreen` 918 行原样搬家 | 高 | 只是把坏边界换了目录 | 先拆 orchestration、展示、overlay 与 side effects，再迁移 | 特征测试覆盖关键状态；依赖通过 provider/props |
| AdminMode 条件继续扩散 | 高 | 公共 UI 被 Admin 特例污染 | 通用能力配置 + 事件回调；兼容参数限期删除/收敛 | Admin 无正式 persistence/ads；分支不再增长 |
| 静态 Next target 条件爆炸 | 中高 | standalone/portal 相互拖累 | `BUILD-003/004` 决策门 | 失败时启用薄 `portal-web` 壳，不复制 runtime |
| 迁移期同时开发新功能 | 高 | 无法区分回归来源 | 行为冻结、逐任务提交、双构建回退点 | H5/Admin 基线全绿后才进入依赖功能 |
| app-to-app 源码引用回流 | 高 | Mobile/Admin 再次依赖 Player 内部 | source-boundary CI | 任意 `apps/A -> apps/B` import 直接失败 |

### 7.2 P1 及以后持续控制

| 风险 | 等级 | 处理原则 |
|---|---|---|
| 自建 session 安全责任 | 高 | 只做 Google/Apple OIDC，不做密码；服务端验签；opaque refresh token 哈希存储、轮换、复用检测、吊销、审计 |
| Next + Vite 两套构建链 | 中 | 两端壳保持薄；锁定版本；公共逻辑只进 package；分别做产物审计 |
| SQL 与 TypeScript 类型漂移 | 中 | Zod contract + repository output validation + migration integration tests；达到规模阈值后再评估生成类型 |
| 云存档冲突逻辑复杂化 | 中高 | 合并规则在 `game-core` 单一实现；revision/requestId/幂等和故障注入测试 |
| package 继续无限拆分 | 中 | 新 package 必须有两个真实消费者或独立发布/安全边界，否则留在现有模块 |
| Supabase 运营依赖 | 低到中 | 只依赖标准 PostgreSQL；定期 `pg_dump`/恢复演练；记录连接池和备份策略 |

## 8. 调整后的迁移策略

### 阶段 0：建立基线

1. 冻结架构迁移期间的玩法/UI 功能变更。
2. 保存 Player/Admin 当前 unit、build、24 关 smoke、关键页面截图和存档 fixtures。
3. 建立 `PlayScreen` 关键状态特征测试：首次进入、续局、胜负、奖励、广告失败/重复回调、语言、AdminMode。

### 阶段 1：先建立契约，不搬 UI

1. 创建 `platform-capabilities` 和 `player-runtime` 最小 public API。
2. 建立依赖边界测试和 Noop/memory adapter。
3. 定义临时 compatibility facade 的入口、启用时机和删除条件；实际在消费者切换前启用。

### 阶段 2：拆职责并逐项迁移

1. 先迁移 pure store、active-game、音频策略和指标协议。
2. 再迁移 `BoardSvg`。
3. 将 `PlayScreen` 的副作用编排与展示组件分开，再迁移；不做视觉重设计。
4. 每一步同时运行 Player 和 Admin 构建，出现回归可回到上一独立提交。

### 阶段 3：切换消费者并封边界

1. Player 先切 package，保持兼容 facade。
2. Admin 再切 package，逐项删除 alias。
3. 双端基线通过后删除 facade，启用 app-to-app import CI 硬门禁。
4. 再进入 portal 静态导出、SaveCoordinator、API 和 Mobile。

### 阶段 4：技术债务放行审计

P0 结束前必须单独评审：公共 package exports、循环依赖、runtime bundle、target 条件、AdminMode 分支、静态产物、存档兼容。未通过时不得以“后续再整理”为理由进入 App 业务开发。

## 9. 长期架构控制线

以下规则应进入 CI 或代码评审模板，而不是只留在文档：

1. 应用不能 import 其他应用源码。
2. `game-core` 不能依赖 React、Web、Native、SDK 或 API。
3. `player-runtime` 不能依赖 Next、Capacitor、渠道 SDK、数据库 client。
4. 平台识别只允许在各 target composition root 和构建配置中发生。
5. 所有必需能力显式注入；不允许业务运行中靠 `undefined` 判断平台能力。
6. 客户端不能直连 Supabase，不能包含数据库凭据。
7. SaveGame 变更必须带 migration、merge 和旧 fixture 测试。
8. API schema 变更必须先改 `api-contracts`，并通过兼容性测试。
9. 新 package、新服务和新状态系统必须写明消费者、责任边界和退出条件。
10. 每个发布 target 都必须有 capability manifest、产物审计和最小运行 smoke。

## 10. 是否会留下很多技术债务

按原方案直接整体搬迁，会留下较多债务，尤其是巨型 `player-runtime`、巨型 `PlayScreen`、AdminMode 扩散和双构建条件分支。

按本文调整后的顺序执行，预期债务是可控的：

- 架构债务低到中：模块边界和依赖方向明确，可由 CI 约束。
- 迁移债务中：需要兼容 facade 和双轨构建，但有明确删除任务。
- 安全债务中高：来自自建 session，是已知且长期存在的主动责任。
- 运维债务中：增加 API、数据库 migration、备份恢复和移动商店流水线。
- 厂商锁定债务低：客户端不依赖 Supabase，数据库使用标准 PostgreSQL。

因此准确结论不是“没有技术债务”，而是“主要债务已知、可测、可设置退出门槛，且比复制三套前端和多套存档逻辑的长期代价小”。

## 11. 架构批准条件

建议在以下条件下批准进入代码实施：

1. 接受 P0 是行为保持型迁移，期间不混入新玩法/UI 改版。
2. 接受先拆 `PlayScreen` 职责，再完成 package 迁移。
3. 接受 `BUILD-004` 是真实决策门：Next 静态方案不干净时使用薄 Vite portal 壳。
4. 接受自建认证属于安全关键模块，必须配套轮换、吊销、审计和安全测试。
5. P0 技术债务审计未通过时，不提前启动依赖该底座的 App 业务功能。

满足以上条件后，本架构足以支撑 Fangrush 当前可预见的多年多端迭代，不需要再进行一次总体推倒式改造。
