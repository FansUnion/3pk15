# Fangrush 多渠道方案选型与主流实践

> 本文中的“主流”是同类项目常见的工程模式，不代表精确市场占有率。平台硬性边界以文末官方资料为准。

## 1. 先按游戏类型判断

游戏开发者通常不是先决定登录 SDK，而是先看产品属于哪一类：

| 类型 | 常见做法 |
|---|---|
| 单平台、轻量、纯单机 | 游客即玩，本地存档；可选平台云，不建自有账号。 |
| 单平台、有付费恢复 | 游客即玩 + 商店购买恢复；平台身份或轻量后端。 |
| 多端但没有实时服务 | 自有用户 ID + 多身份 provider + 自有云存档；平台能力做适配。 |
| 长线运营/多人/经济系统 | 自有账号与服务端权威数据，平台账号只做登录入口和渠道身份。 |
| H5 门户游戏 | 使用门户广告、生命周期、账号和数据模块；尽量不引入外部登录。 |

Fangrush 的玩法复杂度属于轻量单机，但目标覆盖自有 Web、Android、iOS 和两个门户，因此不能完全照“单平台纯本地”做。推荐选择第三类的轻量版本：自有渠道统一，门户渠道平台化，不建设重型实时游戏后端。

## 2. App 客户端技术怎么选

| 方案 | 其他开发者何时选择 | 优点 | 代价 | Fangrush 判断 |
|---|---|---|---|---|
| Capacitor + Web UI | 已有成熟 Web 游戏、交互和性能可由 WebView 承担 | 最大复用 React/TS；原生能力可插件化 | 需处理 WebView 生命周期、静态资源和插件质量 | **推荐** |
| React Native | App 是主产品，UI 需要大量原生交互 | 原生控件生态成熟 | 现有 DOM/SVG UI 不能直接复用，维护第二套页面 | 暂不选 |
| Flutter | 团队熟悉 Dart、希望统一原生渲染 | 跨端一致、性能稳定 | 游戏 UI 和工程几乎重写 | 暂不选 |
| Unity/Godot | 3D、物理、动画和游戏引擎工具链是核心 | 游戏平台生态成熟 | 当前规则与 Web 产品需迁移，投入远超收益 | 不选 |
| Android/iOS 原生 | 极致平台体验或重原生功能 | 控制最强 | 两套客户端和第三套 UI | 不选 |

推荐依据：当前产品是回合制 SVG/React 游戏，性能压力低，仓库已有清晰的纯 `game-core`。Capacitor 官方支持 Android、iOS 和 Web，并允许 Java/Kotlin、Swift/Objective-C 插件桥接，符合现有资产结构。

约束：先做真机性能样机，验证启动时间、棋盘输入、内存、后台恢复和低端 Android；验证失败才重新评估技术路线，而不是凭感觉重写。

## 3. 玩家是否必须登录

| 方案 | 常见场景 | 问题 |
|---|---|---|
| 启动即强制登录 | 多人、强经济、监管账户 | 轻量游戏首局流失高；登录服务故障会阻断游戏 |
| 游客先玩，关键时机提示登录 | 绝大多数轻量/休闲游戏 | 要实现游客存档合并 |
| 永远不登录 | 一次性或单设备游戏 | 无法跨设备恢复，客服恢复能力弱 |

**推荐：游客先玩，账号可选。** 独立站可在设置、通关后或玩家已有有效进度时提示“保存到云端”；App 同样不在首屏拦截。Poki/CG 的登录只能由用户操作触发，并遵守平台 UI。

理由：账号的价值是恢复而不是准入。对 24 关轻量游戏，先让玩家证明愿意玩，比先证明愿意注册更重要。

## 4. App 与独立站账号是否统一

### 方案 A：每端各自账号

实现最直观，但 Android、iOS、Web 的进度和权益彼此割裂，后续迁移成本最高。只适合明确永不跨端的小项目。

### 方案 B：只使用 Google/Apple 平台账号

同生态体验自然，但 Android 与 iOS、App 与 Web 不自然互通。商店购买身份也不等于游戏账号身份。

### 方案 C：统一 Fangrush 用户，外部账号作为身份 provider

内部只有一个 `users.id`，Google、Apple 等记录在 `auth_identities`。这是跨平台游戏的常见结构。

**推荐方案 C。** “统一账号”指统一内部用户，不是五端显示同一个登录按钮。独立站和 Android 可用 Google，iOS 用 Apple；同一玩家通过明确绑定流程把多个 provider 绑定到同一个 Fangrush 用户。

## 5. Poki/CG 是否接入自有账号

| 方案 | 可行性 | 评价 |
|---|---|---|
| 门户中直接显示 Fangrush Google/Apple 登录 | 受外部请求、个人数据和平台账号规则限制 | **不采用** |
| 只用门户账号与门户云 | 平台支持，玩家心智一致 | **首发推荐** |
| 用门户 token 映射自有后台用户 | Poki、CG 都有受控的 token/user 接口 | 后续可选，需权限、服务端验票和明确跨端价值 |

Poki 当前提供 User Accounts、短时 token 验证和透明 Cloud Gamesaves；CrazyGames 提供 User、Data，并允许有自有后端的游戏通过平台账号链接。两者都不等于允许在门户包里自由放外部登录。

**推荐：首发平台闭环，后续再映射。** 这能先获得平台云恢复而不承担跨门户账号合并、隐私和客服复杂度。

## 6. 云存档选哪种

| 方案 | 主流使用者 | 优点 | 缺点 |
|---|---|---|---|
| 仅本地 | 轻量单机首版 | 最快、离线自然 | 清缓存/换机丢失 |
| 仅平台云 | 单生态游戏 | 平台登录和恢复自然 | Android/iOS/Web 互不相通 |
| 仅自有云 | 多端游戏 | 统一、可运营、可客服恢复 | 需要后端、安全与长期运维 |
| 自有云 + 平台云双写 | 大型成熟团队 | 可兼容生态能力 | 冲突源多、测试矩阵大、容易覆盖进度 |
| 按渠道选择 provider | 多渠道 Web/移动游戏 | 内部模型统一，渠道政策清晰 | 不承诺所有渠道互通 |

**推荐：按渠道选择 provider。**

- 独立站、Android、iOS：本地优先响应，Fangrush Cloud 为登录用户的跨设备恢复真相。
- Poki：Poki Cloud Gamesaves。
- CrazyGames：Data module；它同时代理游客 localStorage 和登录用户云同步。
- 首版不让 PGS/iCloud 与 Fangrush Cloud 同时作为可写主库。

这是“一个存档模型、多个存储实现”，不是“五套存档业务”。

## 7. 自有后端怎么选

| 方案 | 行业使用情况 | 适合度 |
|---|---|---|
| Firebase Auth/Firestore | 移动休闲游戏常见，Google 生态完整 | 能用，但与现有 Supabase/Postgres 和 StartUpSense 代码重复 |
| Supabase Auth + Postgres | 小团队和 Web 产品常见，开发快 | 若从零开始很合适；会替换已有自定义认证思路 |
| 薄 API + Supabase Postgres | 需要控制数据模型、已有认证/API 代码 | **最适合当前仓库** |
| PlayFab | 经济、排行榜、运营活动较多的游戏 | 当前过重 |
| Nakama/自建完整游戏后端 | 多人、实时和复杂社交 | 当前明显过重 |
| 每个平台单独后端 | 临时项目 | 长期重复，拒绝采用 |

**推荐：独立 `fangrush-api` + Supabase 托管 Postgres，复用 StartUpSense 中经过审查的通用认证代码。**

“复用”不等于整段复制：只复用 provider 验票、JWT/session、DAO 和数据库连接范式；重新定义 Fangrush 的用户、存档、权益、购买和隐私请求。正式实施前必须修正旧草案中的邮箱自动合并、Web token 长期存放和客户端直接授予权益等风险。

## 8. 各端登录 provider 怎么选

| 端 | 可选方案 | 主流且推荐的首版 |
|---|---|---|
| 独立站 | Google、Apple、邮箱验证码、匿名 | Google + 游客；Apple 随 iOS 统一账号阶段补充 |
| Android | Google Credential Manager、游戏自有密码、PGS 平台身份 | Google 游戏内账号 + PGS 平台身份分层；游客可玩 |
| iOS | Sign in with Apple、Google、自有密码、Game Center | Sign in with Apple + 游客；若提供 Google，同屏提供 Apple 合规选项 |
| Poki | Poki User、外部登录 | Poki User |
| CrazyGames | CrazyGames User、外部登录 | CrazyGames User |

Google Play Games Services v2 已把平台认证与游戏内账号分开：PGS Player ID 主要服务成就、排行榜等平台功能，游戏的主账号应独立管理。Apple 审核规则要求使用第三方社交登录建立主账号时，同时提供满足隐私条件的等价登录；实践中 iOS 应提供 Sign in with Apple。

## 9. 多 provider 账号怎么合并

| 方案 | 风险 | 结论 |
|---|---|---|
| 相同 email 自动合并 | Apple 隐藏邮箱、邮箱变更、provider 未验证、账号接管 | 禁止 |
| 后台人工改用户 ID | 可救急但不可规模化、审计差 | 只作支持工具 |
| 用户登录 A 后显式绑定 B | 双方凭证都可验证，意图明确 | **推荐** |
| 一次性迁移码/二维码 | 跨设备登录入口不对称时实用 | 后置备选 |

推荐流程：用户先登录现有 Fangrush 账号，再点“绑定另一种登录”，服务端验证第二 provider 后建立唯一映射；若第二 provider 已属于其他用户，停止并进入冲突恢复，不自动覆盖。

## 10. 存档冲突怎么处理

| 方案 | 常见用途 | Fangrush 判断 |
|---|---|---|
| 整包最后写入覆盖 | 存档简单且单设备 | 多设备离线时容易丢进度 |
| 用户手选本地/云端 | 大型存档槽、玩家能理解差异 | 对轻量游戏打断感强 |
| 字段级领域合并 | 解锁、关卡等单调增长数据 | **主要方案** |
| 服务端权威事件流 | 强经济/竞技游戏 | 当前过重 |

**推荐混合规则：**

- 通关、章节、免费解锁：集合并集。
- 单调累计且不涉及支付的进度：按领域规则取最大或按幂等事件累计。
- 设置、装备、最后游玩：带字段时间戳的最后写入优先。
- 任务：按周期 key 校验后合并进度与已领取状态。
- 付费权益：完全不从普通存档合并，由购买/权益服务恢复。
- 碎片等可消费资产：首版可接受“客户端进度 + 服务端校验上限”的轻量方案，但不能简单把两端余额相加；若未来有竞争或高价值经济，再升级为服务端事件账本。

## 11. App 广告怎么选

| 方案 | 常见选择 | Fangrush 判断 |
|---|---|---|
| 单一成熟广告网络 | 小规模首发 | **推荐 AdMob**，集成和运营成本最低 |
| 一开始接 mediation | 已有较大流量、专人优化填充和竞价 | 当前过早 |
| 自建 Web 广告塞进 App WebView | 审核、体验、合规和计量问题多 | 不采用 |
| 买断无广告 | 内容量足够、用户愿意预付 | 当前 24 关不作为主方案 |

App 首版建议只做自愿激励广告和完整对局间低频插屏；首局、教学、连续失败、棋子移动和 AI 行动中不得展示。广告失败必须恢复输入与音频，且不吞基础奖励。

iOS/Android 均需要广告隐私同意流程；iOS 如请求跟踪权限还需 ATT。拒绝跟踪不能阻止游戏或非个性化广告路径。

## 12. IAP 与权益怎么选

| 方案 | 评价 |
|---|---|
| 只依赖客户端 purchase callback | 容易重复授予、伪造、退款后不撤销，不采用 |
| 只调用商店恢复购买，不建权益表 | 去广告单商品可勉强工作，但跨 Android/iOS 不统一 |
| 商店购买 + 服务端校验 + entitlement | 跨端和客服恢复最清晰，**推荐** |

首版商品只建议：非消耗型“去广告”和少量不影响胜负的皮肤。Android 使用 Google Play Billing，iOS 使用 StoreKit。购买、恢复、重复回调、退款/撤销、待处理交易、审核沙盒都必须验收。

Poki 不复用 App IAP；CrazyGames IAP 受平台开放和支付方案约束，未获邀请前不进入首发范围。

## 13. 监控与分析怎么选

| 方案 | 优点 | 代价 | 推荐 |
|---|---|---|---|
| 只看商店/门户后台 | 零开发 | 无法定位自有客户端错误和跨端漏斗 | 不足 |
| Firebase Analytics + Crashlytics | 移动游戏常见，Android/iOS 成熟 | WebView JS 堆栈和隐私披露需额外处理 | App 首选基线 |
| Sentry | JS + 原生错误链路更统一 | 另一个供应商，产品分析较弱 | JS 错误覆盖不足时补充 |
| 自建事件库 | 数据控制强 | 运维、合规和分析成本高 | 当前不做全量 |

推荐首版：门户优先平台统计；App 使用 Crashlytics + 最小 Analytics 事件；服务端保留账号、存档和购买审计。只采集能回答首局完成、关卡完成、重试、回访、广告和购买问题的数据。

## 14. 首发范围怎么取舍

### 方案 1：五端同时做全功能

看似完整，实际会把平台权限、服务端、两套原生工程、广告和支付同时放到关键路径，风险最高。

### 方案 2：各端都只做本地存档

最快，但与本轮“账号和云存档完整规划”的目标冲突，且后补时迁移成本高。

### 方案 3：公共契约先行、按渠道分阶段落地

**推荐。** 先把 SaveGame、provider、静态运行时和服务端契约做对；独立站验证账号云存档；Android、iOS 复用；门户账号云存档在广告接入后按平台阶段启用。
