# 玩家全旅程体验专项审计

> 实施状态（2026-07-19）：AI 可完成的代码、自动化、480 局新基线和文档收口已经完成。重复局面调整为第 3 次预警、第 5 次强预警、第 6 次结束；连吃控制改为视口底部临时操作条并在短屏缩小棋盘；列表和直接 URL 共用同一关卡解锁规则；棋盘键盘焦点改为圆形；岩石轮廓与裂纹已增强；狼/羊移动、捕食、连吃、新增威胁、胜负、僵局、解锁与装备均有区分反馈；帮助图例复用真实棋盘；机器文件移入折叠技术区；新增品牌化 404/错误恢复页；不支持广告时不展示激励入口。剩余工作全部是用户真实设备、听感、辨识、审美与关卡体感验收。

> 长期检查要求已提炼到[产品质量与玩家全旅程检查清单](../产品核心/07-产品质量与玩家全旅程检查清单.md)。本文件完成后只作为本批证据归档，不继续承担长期规范职责。

> 状态：已完成四遍审查，方案已经确认并进入开发与回归。本文只维护本次发现、实施状态和证据；长期标准由产品核心规范承接。

## 一、最终判断

这次问题不是六个孤立缺陷，而是三个系统性问题同时出现：

1. **规则反馈来得太晚。** 重复局面直接在第三次结束，没有提前告诉玩家正在走向无效循环；连吃决策只有棋盘下方按钮，小屏玩家也可能在做出关键选择时看不到。
2. **已有能力没有形成可感知体验。** 声音代码区分移动、捕获、连吃、胜负、AI和威胁，但“威胁”只是轻微合成提示音，不是真正可辨认的羊叫；页面有装扮、任务和帮助，局内导航却没有把它们组织起来。
3. **玩家信息与站点维护信息混在一起。** 页脚把 Sitemap、`llms.txt`、`llms-full.txt` 与玩法、隐私、设置并列；广告入口不先判断当前发行渠道是否真的可用；部分页面仍显示无标签数字、重复入口或技术性表达。

因此推荐做一次“玩家全旅程收口”，而不是继续逐条打补丁。目标是让玩家在每个阶段都能回答：现在在哪、要做什么、为什么发生、下一步去哪。

## 二、六个反馈的核实与推荐

### 1. 三次重复是否太少

**当前事实**：核心规则按完整战略局面计数，初始出现算第一次；现在第 3 次出现预警、第 5 次强预警、第 6 次记录 `repetition` 和局。它不要求行动连续相同，而是该完整局面在本局累计出现六次。行动上限仍是另一道兜底。新规则的 480 局结果见[六次重复规则基线](../产品核心/验证记录/2026-07-19-六次重复规则480局基线.md)。

**判断**：对竞技棋类，三次重复是常见裁定；对狼方单人闯关，它过于突然。玩家在第一次循环时通常还没意识到路线无效，第三次直接终局更像系统惩罚，不像教学反馈。

**推荐规则**：

- 同一战略局面第 3 次出现：不中止，显示一次明确但不遮挡棋盘的“路线正在重复”提示；
- 第 5 次出现：升级提示，明确“再回到同一局面将结束本局”；
- 第 6 次出现：判定和局，挑战未完成；
- 行动上限继续保留，胜利和困狼仍优先于重复；
- 提示只在局面真正回到相同状态时出现，不按玩家“看起来走了相似一步”模糊判断。

这会给玩家一次理解和改线机会，同时仍能阻止无限往返。它不是纯文案修改：必须同步核心规则、存档、回放、Admin、模拟器、固定种子矩阵和24关重复率。正式术语继续用“和局”，提示阶段不使用“失败”。

### 2. 小屏时“结束连吃”不明显

**当前事实**：连吃面板位于棋盘之后、底部 Hint/Help/More 之前。棋盘较高或浏览器可视区域较小时，玩家完成第一次捕获后，按钮可能落在首屏下方。

**判断**：这是关键决策控件的布局缺陷。它不是“偶尔出现所以可以放低”，恰恰因为只在连吃时出现，玩家更难形成固定寻找位置。

**推荐布局**：

- 手机和平板：连吃发生时，用贴近屏幕底部安全区的临时行动栏替换普通 `提示 / 帮助 / 更多` 导航；主信息说明“可继续点红圈”，明确按钮为“结束本轮连吃”；
- 桌面宽屏：可在棋盘右侧显示同一行动栏，但右侧只作为宽屏增强，不作为唯一入口；
- 不把按钮覆盖在棋盘节点上，避免遮住下一次可捕获目标；
- 不同时在棋盘下、右侧和底部放三个按钮，保持一个主入口；
- 按钮文字改为“结束连吃（本轮已捕获 n 只）”，避免 `n/5` 暗示玩家必须追满5次。

### 3. 威胁与其他音效是否实现

**2026-07-19 收口结论**：已完成。14 类事件均有独立 WAV，威胁使用移动前后集合差分，只在新形成立即捕食线时播放；单狼新近受困增加独立声音、棋盘标记和文字反馈；设置页可逐项试听并恢复被浏览器暂停的声音。详细证据见[正式声音与受困反馈收口](../产品核心/验证记录/2026-07-19-正式声音与受困反馈收口.md)。

仍需用户做的只有真实设备主观听感，不再属于确定性开发缺口。

### 4. Sitemap 与 LLM 文件位置

**当前事实**：三项机器入口直接与“怎么玩、猎场、隐私、设置”并列在每个玩家页面页脚。

**判断**：不合适。它们对搜索引擎和机器访问有价值，但不是玩家任务。公开端点可以继续存在，不需要成为全站主导航。

**推荐方案**：玩家页脚只保留玩法、关卡、装扮、任务、设置、隐私和语言；Sitemap、`llms.txt`、`llms-full.txt` 从可见主导航移除。若项目所有者仍希望偶尔点击查看，可在隐私页底部增加折叠的“站点与机器信息”，或直接通过固定URL访问。

### 5. 全站文案与流程是否已经用户友好

结论是：**规则准确性已经明显改善，但完整旅程尚未收口。** 当前没有发现新的规则级虚假描述，仍存在以下体验问题。

| 级别 | 环节 | 问题 | 影响 | 推荐 |
|---|---|---|---|---|
| P0 | 结算广告 | 独立站或广告不可用渠道仍可能显示“看广告”，点击后才提示不可用 | 玩家以为奖励被吊起后落空 | 渲染前读取渠道能力；不可用时不显示，Mock只在开发/Admin出现 |
| P0 | 重复终局 | 第三次重复直接结束，没有预警 | 玩家不知道哪一步开始进入死循环 | 3次提示、5次强提醒、6次和局 |
| P1 | 连吃 | 关键“结束连吃”可能落在小屏首屏下方 | 玩家误以为只能继续或不知道如何交回羊方 | 临时底部行动栏，宽屏右侧增强 |
| 已完成 | 声音 | 原威胁音难辨认且缺单狼受困反馈 | 已补14类正式样本、威胁差分、受困反馈、试听与恢复 | 仅保留真实设备听感复核 |
| P1 | 局内导航 | 更多菜单只有重开、静音、报告、语言和退出 | 游戏像孤立棋盘，装扮、任务、关卡体系不可达 | 菜单按“本局 / 游戏内容 / 设置与反馈”分组 |
| P1 | 首页回访 | “开始冒险”始终进入春日01，真正继续入口视觉较弱 | 回访玩家容易误点重开第一关 | 有进度时“继续当前关”成为主按钮，新开局降级 |
| P1 | 四季总览 | 顶部只显示无标签碎片数字 | 玩家不知道数字用途 | 显示“通用碎片 n”并可进入装扮 |
| P1 | 中文季节卡 | 卡片主标题直接显示 `spring/summer/...` | 中文流程中突然出现英文信息 | 使用春日、夏日、秋日、冬日正式名称 |
| P1 | 隐私页 | 中文页仍硬编码 `Analytics / Google Privacy Policy` | 中英混排且不解释为何跳转 | 本地化标题和链接说明，只在实际启用统计时显示 |
| P2 | 玩法页 | 返回首页按钮写“关闭”；完整玩法与快速提示内容重复 | 导航语义不准，维护两套说明 | 改“返回首页”；设置页只链接完整玩法，删除重复快速提示 |
| P2 | 关卡说明 | 标题下再次显示 `Fangrush`，不提供进度或挑战难度 | 品牌重复，占用首屏但缺少决策信息 | 显示“第x/24关、挑战难度、岩石数量” |
| P2 | 任务 | `0/1 · +碎片` 能用但缺少任务如何累计的即时解释 | 新玩家不确定是否需要手动领取 | 首次进入增加一句规则；可领取任务优先排序 |
| P2 | 装扮 | 装扮页功能完整，但玩家只能从首页/玩法页找到 | 奖励与使用场景脱节 | 通关奖励卡和局内菜单提供明确入口 |
| P2 | 问题报告 | “报告本局问题”与普通玩家命令并列 | 普通玩家不清楚发给谁，正式上线后显得开发态 | 改“反馈本局问题”，放入反馈分组；平台无反馈渠道时隐藏 |
| P2 | 终局详情 | 广告、分享、关卡列表、报告都挤在折叠详情里 | 产品主次有了，但低频动作仍杂乱 | 分成“战绩”“奖励”“反馈”，按渠道显示能力 |
| P2 | 页脚 | 缺装扮和任务，却突出机器入口 | 人类导航优先级反转 | 增加装扮/任务，移除机器入口 |

### 6. 游戏过程中的菜单是否完整

**结论：对局工具基本完整，游戏级导航不完整。** 当前“更多”菜单适合处理本局，却没有关卡列表、四季、装扮、任务和设置。玩家无法从局内感知完整的奖励与长期目标闭环。

推荐菜单结构：

1. **本局**：重新开始、声音开关、返回关卡列表；
2. **游戏内容**：四季关卡、装扮、任务；进入这些页面时保留当前活动对局，可从首页或设置继续；
3. **设置与反馈**：玩法说明、语言、设置、反馈本局问题；
4. **离开本局**：单独置底并使用危险样式，不与普通功能混排。

不建议把所有入口长期铺在局内底栏。底栏继续只放高频的提示、帮助、更多；完整能力放进分组菜单。

### 7. 全部声音是否只有威胁音有问题

不是。初始诊断还发现羊步、和局、解锁、装备、浏览器恢复和单狼受困缺口；这些确定性问题现已统一完成。现行事件矩阵和放行标准只维护在[视听资源与场景反馈验收标准](../产品核心/06-视听资源与场景反馈验收标准.md)，本专项不再复制易过期的资源清单。

### 8. 前后台关卡文案是否统一

**结论：核心事实大多同源，但还不是完整单一来源。**

当前主链路：

```text
game-core LevelConfig
  ├─ 玩家关卡列表、说明页、局内标题
  ├─ Admin 关卡列表、详情和试玩
  ├─ 24关设计卡的运行事实投影
  └─ SEO/公开关卡页面
```

同源部分包括：关卡ID、中英文名称、中英文短说明、季节、序号、岩石、开局、AI、目标、行动上限、挑战难度、策略标签、中文设计元数据和奖励配置。Admin 确实直接读取 `level.nameZh / blurbZh / difficulty / ai`，没有再维护一份关卡表。

仍有四个缺口：

1. 英文教学说明由 `LEVEL_TEACHING_EN` 独立映射，中文教学说明在 `LevelConfig.teachingPoint`，结构分裂，新增或改关时可能只改一边；当前 `spring-01` 仍出现不够自然的 `gap capture`。
2. 设计卡契约目前只验证中文名称、中文短说明、AI、目标和行动上限；没有验证英文文案、挑战难度、岩石/开局摘要、策略、教学、风险、奖励和生产状态。
3. 玩家端通用UI文案集中在双语消息树，Admin 产品解释多数直接写在组件里。两者不应强求字面相同，但“规则事实、指标定义、终局名称”必须使用同一选择器或契约。
4. Admin关卡详情的“前台说明”链接使用相对 `/hunt/{id}`。两应用分域部署后会打开 `admin.fangrush.com/hunt/...`，不是 `fangrush.com/hunt/...`，属于确定性错误。

推荐收口：

- 把中英文名称、短说明和教学点组成同一个本地化关卡内容结构，避免两个独立字段集合；
- 提供 `getLevelPlayerContent(level, locale)` 和 `getLevelProductContent(level)` 两种明确投影；玩家与Admin可以表达层级不同，但不重复定义事实；
- 扩展24关契约，验证所有玩家可见关卡字段和设计卡投影；坐标仍只由代码维护，不复制到文档；
- Admin跨应用链接统一通过 `NEXT_PUBLIC_PLAYER_SITE_URL` 或共享站点配置生成绝对地址；
- 通用UI不要求前后台逐字相同：玩家使用操作语言，Admin使用产品验证语言，但术语表和规则结果必须一致。

### 9. 第三轮补充发现

上一版覆盖了主要可见页面，但未把“页面不存在、加载失败、直接深链、存档异常、生命周期和所有功能状态”逐项展开。第三轮新增以下问题：

| 级别 | 缺口 | 当前事实 | 推荐 |
|---|---|---|---|
| P0 | Admin前台说明跨域链接 | 相对路径落到Admin域名 | 使用玩家站绝对URL并加部署契约测试 |
| P1 | 玩家404 | 没有玩家品牌化 `not-found.tsx` | 提供返回首页、四季和继续游戏入口 |
| P1 | 页面错误边界 | 没有玩家 `error.tsx/global-error` | 给出重试、返回和进度安全说明 |
| P1 | 加载状态 | 没有全局或关键页面 `loading.tsx` | 只为真实等待提供稳定骨架，不伪装为错误 |
| P1 | 锁定关卡深链 | 列表有解锁状态，但直接URL边界未形成产品契约 | 明确“允许预览但不能开局”或“Admin专享绕过”，并测试 |
| P1 | 恢复/损坏存档提示 | 底层会降级，但玩家不知道进度为何变化 | 只在真正恢复失败时显示可行动说明 |
| P1 | 后台/前台生命周期 | 有可见性与平台暂停逻辑，但未纳入完整人工矩阵 | 覆盖切后台、锁屏、恢复、静音与AI回合 |
| P1 | 浏览器音频授权 | 有诊断计数，没有玩家可见恢复入口 | 播放被阻止时提供“点按恢复声音”，不弹技术错误 |
| P2 | 分享全部分支 | 分享、复制、下载、取消均有代码，缺完整页面级验收 | 在桌面、移动和不支持Web Share时分别验证 |
| P2 | 广告六种结果 | Mock有成功、失败、取消、不可用、无填充、冷却 | 玩家文案目前大多合并为一个失败提示，应按可行动性分组 |
| P2 | 装扮完整状态 | 预览、余额不足、可兑换、装备均存在 | 增加解锁/装备反馈及局内返回路径验收 |
| P2 | 任务周期边界 | 日/周刷新和防重复领取有实现 | 增加跨日期、时区、离线恢复和已完成排序体验检查 |
| P2 | 字体缩放/横屏/安全区 | 常规响应式存在，未见专项覆盖 | 加入200%字体、窄横屏、刘海安全区和键盘焦点检查 |
| P2 | 减少动态效果 | CSS关闭主要动画 | 验证关闭后仍能看到捕获、选中和终局状态 |

这意味着最终“全面审查”不能只按URL计数，还要按页面、用户状态、对局状态、发行渠道和设备环境的组合验收。

### 10. 锁定关卡仍可点击并跳回四季

**当前事实**：春季第一关开放；后续关卡行仍渲染为链接，关卡名和“说明”都可点击，但未解锁时目标统一是 `/chapters`。生产页面已确认可访问，用户观察与本地实现一致。

**判断**：这是明确的交互缺陷。卡片表现为“这里有一个可执行目标”，点击却跳到不解决问题的上级页面；玩家不知道是未解锁、链接错误还是进度丢失。锁定状态和直接深链也不一致：列表表现为不能玩，但 `/play/{level}` 的产品边界尚未明确。

**推荐方案**：

- 锁定关卡整行改为不可导航状态，显示锁图标和“通关上一关解锁”；
- 点击锁定行时只显示就地解释，不跳转，不把“说明”伪装成可用链接；
- 已开放关卡的主动作明确为“开始/继续”，说明作为次入口；已通关显示完成状态；
- 玩家端直接访问未解锁 `/play/{id}` 时显示品牌化锁定页，提供“继续当前可玩关”和“返回本季”，不能绕过主线；
- `/hunt/{id}` 可以允许预览，但开始按钮必须遵守解锁；是否允许预看后续策略由产品决定，首发推荐只显示名称和解锁条件，避免剧透；
- Admin继续允许无条件试玩全部24关，玩家与Admin入口由部署边界测试区分。

### 11. 帮助页内容与图例审计

**当前结构**：规则概览五段 → 操作说明 → 两张图例 → 六项策略 → 本地进度 → 三个入口 → FAQ。内容覆盖较多，但对第一次访问者偏长，而且“最先需要学的动作”和“以后查阅的完整规则”没有分层。

主流策略游戏的帮助通常分为：快速开始、目标与失败、操作与界面、特殊规则、进阶策略、进度/设置/FAQ。它们不会要求玩家在开局前读完整手册，而是用短教程完成第一步，把完整规则留作随时查阅。

当前两张图的规则坐标基本正确：

- 普通移动：狼与绿色相邻空点正交相邻，符合规则；
- 隔空吃：狼—中间空点—红圈羊同线，符合规则；
- 岩石没有占据上述合法落点。

但图例仍不合格：

1. 只有动作前，没有动作后，玩家看不到狼落到羊位、羊被移除；
2. 没有箭头或步骤编号，红色横线的含义需要猜；
3. 岩石用CSS形状，棋盘是简化网格，却声称使用与实战相同资源；
4. 中文消息树还保留未渲染的 `W/S/#/.` ASCII图例，属于过时死文案；
5. 没有展示继续连吃与主动结束的差别；
6. 没有展示羊只能横移/向狼方前进、岩石如何阻挡、三狼全部被困为何失败；
7. 图例与正式规则没有共享状态生成或合法行动测试，未来规则变化可能漂移。

**需要更多图片，但不是简单增加数量。** 推荐重做为4组高信息图例：

1. `移动与隔空吃`：各自使用操作前→操作后；
2. `连吃决策`：继续捕获与结束连吃两个结果；
3. `羊与岩石`：羊允许方向、禁止后退、岩石不可落脚；
4. `终局与循环`：三狼被困、路线重复预警和行动上限的最短说明。

第一屏只展示快速开始和第1组图，其他内容按“完整规则、策略、进度与FAQ”分组或折叠。长期帮助结构和图例标准已写入[玩家教学与策略系统](../产品核心/03-玩家教学与策略系统.md)。

### 12. 狼行动后的正方形边框

**截图现象**：`spring-03` 已捕获1只、顶部显示狼回合，底线狼外侧保留黑色圆角方框。玩家无法判断它表示“当前选中”“刚刚移动”“必须继续连吃”还是浏览器焦点。

**代码事实**：正式产品选中态在 `BoardSvg` 中定义为金色圆形光环、轻微放大和阴影，没有定义黑色正方形框。普通移动后会清除选中，羊回应完成后也不会自动重新选中上一只狼；连续捕获时则会强制保留当前狼。SVG棋子使用可点击 `<g>`，现有全局焦点样式没有明确处理SVG交互元素，因此截图方框更可能是浏览器焦点轮廓或SVG图像边界反馈。仅凭截图无法确认当时 `state.chain` 是否存在，精确归因仍需本局状态或浏览器DOM焦点证据。

**产品判断**：当前表现不符合用户习惯。方框本身不是绝对错误，许多棋盘游戏会用方格表示最后一步或选中格；问题是 Fangrush 的棋子位于交点、正式选中态是圆形、方框没有图例，而且行动后仍保留，造成四种语义竞争。

**推荐行为**：

1. 普通移动：动作动画结束后清除选择；羊回应结束进入下一狼回合时，没有狼默认被选中。
2. 连续捕获：同一狼保持金色圆形光环，红圈显示下一目标，固定行动栏明确“继续连吃 / 结束连吃”；不显示黑色方框。
3. 上一步提示：如需要，用短暂起点/终点痕迹并在约1秒后淡出，不与选中态共用样式。
4. 键盘焦点：为棋子建立正式可聚焦按钮语义和主题焦点环；鼠标/触控不显示默认浏览器焦点框，键盘用户仍能看见焦点。
5. 终局、AI回合和不可操作状态：清除选择、合法点和焦点假象。
6. 增加普通移动、捕获后无连吃、捕获后可连吃、羊回应后、键盘/触控、终局六组视觉状态测试。

该问题归入第一批“关键对局反馈”，并由[视听资源与场景反馈验收标准](../产品核心/06-视听资源与场景反馈验收标准.md)长期约束。
