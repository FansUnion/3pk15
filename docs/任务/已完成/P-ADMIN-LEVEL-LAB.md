# P-ADMIN-LEVEL-LAB：24 关生产与试玩工作台

## 1. 目标

把 Admin 建设为围绕 24 张主线地图的生产、试玩、诊断和反馈工作台。普通玩家继续使用逐关解锁与奖励流程；管理员可以查看全部关卡、绕过解锁直接试玩，并记录用户试玩结论。管理员试玩必须复用正式关卡初始化、规则、羊 AI、终局和重开链路，只绕过解锁与奖励，不维护另一套游戏规则。

当前不引入数据库。反馈先保存在浏览器 `localStorage`，支持 JSON 导入导出；出现多人协作、跨设备同步、完整棋谱长期存储或线上关卡版本发布需求时再迁移数据库。

## 2. 信息源边界

- `packages/game-core/src/content/levels.ts`：运行配置和前台结构化内容的机器事实来源。
- `docs/产品核心/24关设计卡/<levelId>.md`：单关完整产品档案，保存名称含义、设计理念、配置解释、策略、模拟证据、调优历史和用户试玩结论。
- 自动化测试：验证关卡数量、字段完整性、初始化、重开、序列化、规则和确定性模拟口径。
- Admin：读取并展示上述信息，记录试玩反馈；不得复制或重写规则。

Markdown 不直接驱动运行配置。通过结构化字段、自动化检查和 Admin 缺失提示控制代码与档案漂移。

## 3. 阶段一：结构化关卡元数据

为每关维护并在 Admin 展示：

- 关卡 ID、季节、序号、中文名称。
- 名称含义、设计理念、前台中文简介和中文教学说明。
- 玩家体验目标、狼方策略、羊方防守特点。
- 岩石、狼/羊开局、羊 AI、操作难度、目标吃子数、`maxPlies` 和预期节奏。
- 生产状态、风险标签和最后验证版本。

验收条件：24 关结构化内容完整；缺失字段由测试或 Admin 明确提示；运行配置与前台展示使用同一 `LevelConfig` 数据。

## 4. 阶段二：`/admin/levels` 关卡总控台

- 一屏查看全部 24 关，不受普通玩家解锁状态限制。
- 按季节、羊 AI、操作难度、试玩状态和风险标签筛选。
- 每关显示稳定尺寸的 6x6 棋盘缩略图，包含狼、羊和岩石。
- 展示名称、名称含义、设计理念、前台文案、教学说明、配置和风险。
- 支持查看详情、直接试玩、上一关、下一关、重开和打开单关文档。
- 汇总生产验收、未试玩、试玩通过和待修订数量。

验收条件：管理员可在两个操作内打开任意关卡；24 张缩略图与真实 opening/rocks 一致；筛选与状态汇总正确。

## 5. 阶段三：Admin 无解锁试玩

建议入口：`/admin/play/[levelId]`。

- 复用正式 `createLevelInitialState`、移动/跳吃/连吃、羊 AI、终局、重复和 `maxPlies` 逻辑。
- 不读取普通玩家解锁条件，不发放奖励，不修改普通玩家进度和任务。
- 提供关卡快速选择、上一关、下一关、重开、返回总控台和查看说明。
- 显示本局 plies、吃子数、终局原因和尝试次数。
- 结束后可记录通过、待修订、难度评分和备注。

验收条件：任意关卡可直接试玩；初始化与正式游戏一致；Admin 试玩不会污染玩家存档或奖励。

## 6. 阶段四：本地试玩记录

```ts
type LevelReview = {
  levelId: string
  levelVersion: string
  status: 'unreviewed' | 'passed' | 'needs_changes'
  difficultyRating?: 1 | 2 | 3 | 4 | 5
  attempts: number
  result?: 'wolf' | 'sheep' | 'draw'
  terminalReason?: string
  plies?: number
  notes: string
  reviewedAt: string
}
```

- 自动累计尝试次数并保存最近终局摘要。
- 保存状态、主观难度和备注。
- 支持导出/导入全部反馈 JSON。
- 支持清空单关反馈；清空全部反馈需要二次确认。
- 关卡配置哈希变化时标记历史反馈“需复核”，不静默沿用旧结论。

验收条件：刷新后记录仍存在；导入导出往返不丢字段；错误 JSON 不覆盖已有记录。

## 7. 阶段五：`/admin/ai` 诊断增强

- 选择关卡、羊 AI、狼策略、种子、节点预算和局数。
- 显示胜负、终局原因、plies、吃子数、首次捕食和异常标签。
- 支持单局播放、暂停、逐步前进、调速和完整行动序列。
- 支持按困狼、重复、`maxPlies` 和异常终局筛选。
- 支持从模拟局面由管理员接管狼方。
- 导出复现参数、配置摘要和棋谱；后续再增加调优前后 A/B 对照。

验收条件：同配置、策略、预算和种子结果可复现；人工接管继续使用正式规则；导出数据足以在本地复跑。

## 8. 文档回填与上线闭环

- Admin JSON 是试玩采集物，不自动改生产配置或 Markdown。
- 每轮试玩后，将结论回填对应单关设计卡的“用户试玩记录”。
- 总表维护生产验收、用户试玩确认、待修订和高风险数量。
- 待修订关卡每次只改一个主要变量，并重跑固定种子矩阵、核心回归和 Web 构建。
- 关卡稳定后补英文名称解释、教学说明和前台文案。

## 9. 数据库触发条件

满足任一条件后单独立项数据库迁移：

- 多人或跨设备提交试玩反馈。
- 长期保存完整棋谱和历史关卡版本。
- 统计正式玩家通关漏斗与留存。
- 在线编辑、审核和发布关卡。
- 区分开发、测试、生产配置并支持回滚。

建议表：`level_versions`、`play_sessions`、`play_events`、`level_reviews`、`simulation_runs`。

## 10. 执行顺序

1. `T-026`：统一结构化元数据与一致性检查。
2. `T-027`：完成 `/admin/levels` 总控台。
3. `T-028`：完成 Admin 无解锁试玩和本地反馈。
4. `T-029`：增强 `/admin/ai` 棋谱诊断与人工接管。
5. 用 Admin 完成 24 关用户试玩确认，问题关卡进入 `T-004` 单关修订。
6. 执行 `T-016` 上线回归。

## 11. 当前非目标

- 不建设在线关卡编辑器。
- 不让 Admin 直接修改生产配置。
- 不引入数据库、账号同步或多人审核流。
- 不让管理员试玩使用简化规则或独立 AI。
- 不在地图仍可能调整时优先补齐全部英文内容。
