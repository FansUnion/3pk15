## 一、最终判断

1. **和局作为防无限对局的规则是正常且必要的，但当前玩家表达不合格。** 系统存在“三次重复局面”和“达到行动上限”两种和局；两者都不通关、不发通关奖励、不解锁下一关，实际等同“挑战未完成”。当前结算副标题却一律写成“达到步数上限”，重复和局会被错误解释。
2. **截图中的和局更可能是三次重复，但未确认。** summer-06 的既有固定种子基线出现过重复和局、没有出现行动上限和局；如果本局明显没有达到300次单方行动，重复是更合理的推断。缺少本局存档、结果详情或棋谱，不能把推断写成事实。
3. **关闭结算后仍显示“狼回合”、旧选中框，容易让终局看起来还能继续。** 这不改变规则结果，但属于终局只读状态表达错误；稳定结果条还没有显示精确终局原因。
4. **“被威胁的羊没有逃”不一定是 AI 错误。** 羊方目标是让三狼被困并阻止狼累计捕获8只，不是保证每一只羊都活着。牺牲一只羊若能封锁狼、破坏连吃或换取更强站位，可以是合理防守。
5. **但当前 AI 不能充分证明这是合理牺牲。** normal 只比较羊走一步后的静态总分，虽然惩罚狼的直接捕食机会，却不追踪“哪只羊被救、是否新增送子、下一次最长连吃”。summer-01、summer-06 使用 normal，因此确实可能出现短视送子；summer-02 使用 hard，但 hard 只在 normal 的即时最优候选内做有限前瞻，也不能保证每次都符合人的直觉。
6. **声音实现已经区分普通移动与捕食，但“每次都能听见”尚未被证明。** 普通移动使用 `step.wav`，捕食使用 `capture.wav`，连吃使用 `chain.wav`；玩家等待多久再点击不会改变动作类型，代码仍会尝试播放捕食声。浏览器音频上下文、静音状态、首次异步加载或资源失败仍可能造成听不到。
7. **“狼产生威胁时羊叫”有体验价值，但必须克制。** 推荐只在一次狼方普通移动新产生至少一个立即可捕食目标时播放一次短促羊叫；选择棋子、已经存在的威胁、捕食本身和连续吃期间不重复播放，避免声音嘈杂或泄露过多计算答案。
8. **生产现场目前不能被严格复现，这是核实 AI 反馈的首要阻塞。** 正式对局初始随机种子来自 `Date.now()`，活动存档只保存棋盘、不保存 AI 种子；恢复后会换一个新种子。Admin 可以输入种子，但玩家现场没有可导出的对应种子和动作历史。
9. **Hard 线上结果还受设备时间影响。** 正式 Hard 默认同时限制4000节点和12ms；相同种子在不同设备负载下可能搜索到不同深度，甚至降级到 normal。固定节点的测试可复现，带毫秒预算的生产选择不保证跨设备完全相同。
10. **现有 AI 测试只能证明一个简单局面会避开直接送子，不能覆盖用户反馈。** 已有测试验证 normal/hard 在“安全动作风险为0、危险动作风险大于0”的构造局面选择安全动作；没有验证多羊、多狼、岩石、已有威胁、最长连吃、战术牺牲和并列选择。

### 2.1 和局与优先级

当前规则顺序为：

1. 狼累计捕获达到关卡目标：狼胜；
2. 三狼全部无合法行动：羊胜；
3. 达到 `maxPlies`：和局；
4. 若前面均未终局，再记录完整局面；同一局面第三次出现：和局。

一次捕食若同时达到目标，立即狼胜。一次羊移动若同时让三狼无路且达到行动上限，先判羊胜。行动上限优先于该动作造成的第三次重复。该顺序符合“明确胜负目标优先，安全截断最后处理”的原则，建议保留。

当前和局不会调用通关奖励和解锁逻辑，主操作是“再来一局”。因此不存在玩家靠和局推进关卡的问题。

### 2.2 重复局面存在一个定义缺陷

重复键目前包含每枚棋子的内部 ID、坐标、岩石、行动方和连吃状态。行动方与连吃状态必须保留，岩石虽固定也可保留；但羊在规则上是同质棋子，内部 ID 不应改变战略局面。

例如两只羊交换位置后，玩家看到的占位和所有合法行动完全相同，当前实现仍可能把它们视为不同局面。这会延后或漏掉重复判定，让无意义往返继续更久。

推荐把重复身份改为：排序后的狼坐标 + 排序后的羊坐标 + 岩石 + 行动方 + 连吃狼坐标/次数。狼在连续吃状态下需要知道是哪只狼继续行动，但普通状态也可以按占位处理。修改时必须处理旧存档中的历史重复键，最稳妥的做法是版本迁移时重建当前局面计数，不混用新旧键。

此外 `BoardState` 目前只保存 `draw`，没有保存 `repetition` 或 `maxPlies`。玩家、Admin试玩、候选模拟和批量模拟各自用相似但独立的函数反推终局原因。这正是玩家副标题与真实原因脱节的架构来源。推荐由核心规则在终局发生时一次性写入 `terminalReason`，其他入口只读取，不再复制推断顺序。

### 2.3 羊 AI 为什么不一定救被攻击的羊

normal 的评价包含：剩余羊数、狼行动空间、羊群聚集、推进、包围、当前狼跳吃数量和羊机动性。羊完成移动后，剩余羊数暂时没变；它主要通过“减少狼的直接跳吃数量”等间接指标判断安全。

因此会有四类外观相似、性质不同的行为：

| 类型 | 行为 | 产品判断 |
|---|---|---|
| 成功脱险 | 移走被威胁羊，并减少狼的可捕食机会 | 合理防守 |
| 战术牺牲 | 不救或主动暴露一羊，但立即困住狼、切断连吃或显著压缩狼机动 | 可以接受，需有证据 |
| 等价选择 | 多个动作留下相同风险和防守价值，固定随机从并列项中选择 | 可接受，但不应频繁显得愚蠢 |
| 明显送子 | 存在能减少即时损失且不损害封锁的走法，AI 却新增捕食或长连吃机会 | AI 缺陷 |

当前 Admin 没有直接显示这四类归因，所以仅凭肉眼无法区分“有意牺牲”和“短视送子”。

### 2.4 当前声音链路

- 点选狼：程序合成轻提示音；
- 普通移动：`step.wav`，狼和羊共用；
- 第一次捕食：`capture.wav`；
- 后续连续捕食：`chain.wav`；
- 威胁形成：当前没有专用声音；
- 静音：不会播放任何上述声音。

捕食动作被规则接受后会产生 `jump` 反馈，界面 effect 据此调用声音。等待很久再完成捕食不应使声音失效，因此用户反馈更像播放可靠性问题，而不是设计成“超时不播放”。目前缺少真实浏览器事件日志和音频播放成功回执，结论为“代码每次尝试，实际是否每次成功未确认”。

本轮已读取音频文件本体：`capture.wav` 是有效的 PCM 16-bit、22050Hz、单声道文件，时长0.16秒，平均音量约 -19.6dB、峰值约 -6dB；其余移动、连吃、胜负文件也均可正常解析。因此“文件损坏、空音频或音量接近零”已排除。

更具体的可靠性风险是：首次播放样本时先异步请求和解码，之后才尝试恢复 AudioContext；浏览器的用户手势授权可能已经过期。恢复调用没有等待结果、没有捕获“被自动播放策略阻止”，也没有播放成功回执。200ms 棋盘反馈结束不会取消已经开始的异步函数，但无法保证异步完成后浏览器允许出声。

### 2.5 正式对局与 Admin 复现链路不等价

正式对局在初始化时用当前时间生成种子，每次羊行动后递增；AI 实际收到的种子还混入已捕获数和当前棋子数。这个过程没有写进活动存档、对局指标或分享结果。

因此当前存在三种不同含义的“同一局”：

- 同一关重新开始：开局和地图相同，AI 随机序列不同；
- 恢复同一活动存档：棋盘相同，后续 AI 随机序列可能已经改变；
- Admin 输入一个种子：可以复现 Admin 自己的算法路径，但不一定是生产玩家当时的种子。

这不一定让普通玩家感知为 bug，却会让“羊为什么走这一步”的质量问题无法证伪或复现。第三批必须先建立生产复现契约：对局 ID、初始 seed、逐次 AI seed、难度、Hard 节点/时间预算、规则版本、关卡版本、动作序列和终局原因一起导出。活动恢复必须继续使用原 seed 进度。

## 三、产品方案

### 3.1 和局改为“僵局，挑战未完成”

不删除重复规则，也不贸然调整各关 `maxPlies`。玩家结算按真实原因分别表达：

- 历史实现为三次重复结束；2026-07-19 起正式规则改为第 3 次预警、第 5 次强预警、第 6 次结束，现行说明以核心规则和玩家全旅程专项为准。
- 行动上限：`僵局 · 本局已达到行动上限`；建议：`更早建立接触，避免长期没有进展的调位。`

两者主按钮都是“重新挑战”，不发奖励、不推进关卡。Admin 和棋谱继续使用精确技术原因 `repetition`、`maxPlies`。

关闭结算查看终局棋盘后：

- 顶部行动方改为“对局已结束”，不再显示狼回合/羊回合；
- 清除选中框和合法行动高亮；
- 底部稳定显示“僵局 + 具体原因 + 查看结果 + 重新挑战”；
- 本局详情显示行动数、首次捕食、捕获数和终局原因。

### 3.2 羊 AI 建立“送子判定器”

不采用“被威胁的羊必须逃跑”的硬规则。该规则会让 AI 可预测，也会禁止合理牺牲。推荐对羊方每个候选动作计算以下可解释指标：

1. 动作后狼方立即可捕食数量；
2. 动作后狼方一次完整回合最多可捕获数量；
3. 原被威胁羊是否脱险；
4. 新增暴露羊数量；
5. 狼行动空间变化；
6. 是否立即困住三狼；
7. 是否切断或制造连续捕食；
8. 动作后最坏狼回应的终局和评分。

判定原则不采用单一加权阈值，而采用“支配关系”：若动作 A 相比合法动作 B，会带来更多即时捕食、更长最坏连吃或更接近狼方目标，同时在困狼结果、狼机动压缩、羊机动和后续评分上均不更好，则 A 是被 B 严格支配的明显失误。normal/hard 应排除被支配动作；easy 可保留一定概率。这样不会误杀“牺牲一羊但立即困狼”的合理策略。

只有不存在安全支配动作时，才比较加权战略价值。所有“战术牺牲”必须能给出至少一项补偿证据：立即困狼、显著减少狼总行动、切断更长连吃、把狼推入不可退出区域，或前瞻后明显提高羊方结果。不能只因为总评分略高就称为合理牺牲。

Admin AI 页面增加面向产品人员的解释，不展示函数名：

- `这只羊为什么没逃？`
- `移动前可被捕获 X 只，移动后 Y 只`
- `最坏情况下狼下一回合可连续捕获 Z 只`
- `选择理由：脱险 / 封锁 / 战术牺牲 / 并列随机 / 疑似送子`
- 可展开比较前三个候选动作，并一键导出当前局面、关卡、AI档位和随机种子。

此外必须显示 Hard 本次是否完成前瞻、访问节点数、耗时以及是否降级，但翻译成产品语言，例如“完整比较 / 预算不足，使用普通防守”，避免只有开发者能理解。

### 3.3 三关专项复现

对 summer-01、summer-02、summer-06 各运行固定局面检查：

- summer-01：normal 是否在有无损脱险动作时忽略受威胁羊；
- summer-02：hard 是否因为候选过滤、12ms预算或短视评价选择不逃；
- summer-06：normal 是否主动把羊移入新攻击线，以及该动作是否换来真实封锁收益。

没有用户当时的精确局面和种子，无法复现原动作；先用自动扫描寻找同类反例。扫描不能只从关卡开局随机跑，应收集每次羊行动的全部候选及支配关系。若扫描未发现，再由用户下一次遇到时从游戏直接导出生产复现包，而不是要求用户描述全部坐标。

### 3.4 声音分层

建议保留并强化四层反馈：

| 事件 | 声音 | 目的 |
|---|---|---|
| 狼/羊普通移动 | 短、轻的落点声 | 确认动作完成 |
| 狼首次捕食 | 独立捕食声 | 确认关键收益，不与移动混淆 |
| 连续捕食 | 随连吃次数略增强 | 强化连击节奏 |
| 狼移动后新形成直接威胁 | 一次短羊叫 | 提醒局势变化，不代替高亮和思考 |

实现可靠性方案：首次用户点击时同步解锁 AudioContext，并在后台预加载；动作发生时已加载则播放样本，未加载则立即播放合成降级音，不等待网络请求后才决定是否反馈；等待 `resume()` 结果后再启动样本。记录开发模式下的 `requested / buffer / fallback / muted / blocked / played` 状态。威胁羊叫每个狼方动作最多一次，并设置短冷却。

胜利捕食还需要避免“捕食声”和“胜利声”在同一时刻互相覆盖：先播放捕食确认，再在短延迟后播放胜利音；普通捕食不增加额外延迟。

## 四、全面检查范围

本批不能只修截图和三个关卡，还应覆盖：

### 规则与终局

- 重复键采用战略等价局面而非棋子内部身份；
- 胜利、困狼、行动上限、重复的优先级测试；
- 羊方跳过行动是否正确进入重复和行动上限；
- 存档恢复、重开、回放、Admin、候选模拟的终局原因一致；
- 所有关卡的重复率、行动上限率和长尾重新生成，不能沿用规则键修改前的矩阵。

### AI

- easy/normal/hard 的允许失误边界；
- 立即送子、最长连吃、困狼收益和合理牺牲；
- Hard 预算耗尽与降级选择是否可见；
- 固定种子下正式游戏与 Admin 选择一致；
- 活动恢复前后 AI seed 进度与下一动作一致；
- 固定节点基线与12ms生产模式分开报告，不把时间预算波动解释成策略随机；
- AI 调整后重跑24关矩阵，避免“更会保羊”导致狼方胜线整体消失。

### 界面与声音

- 两种僵局的标题、副标题、建议和只读棋盘；
- 终局关闭后不显示行动方、选中态和可操作暗示；
- 捕食、连吃、威胁、普通移动、静音、切后台再回来、首次加载失败；
- 手机浏览器和嵌入式 H5 环境的自动播放限制；
- 视觉高亮仍是主要信息源，声音关闭后玩法完整。

## 五、执行顺序

### 第一批：可复现与终局可信度

1. 建立生产复现契约，保存并导出初始 seed、seed 进度、动作序列、AI预算和版本；恢复对局继续原随机序列；
2. Hard 自动验收使用固定节点作为可复现基线，生产毫秒预算结果同时记录是否降级，不能与固定节点数据混称同一结果；
3. 修复重复局面战略身份定义并补旧活动存档迁移测试；
4. 终局状态保存明确的 `terminalReason`，界面、存档和回放不再事后猜测；
5. 分开表达重复僵局和行动上限僵局；
6. 修复终局只读棋盘的行动方、选择框和稳定结果条；
7. 重跑固定种子终局分布。

### 第二批：羊 AI 送子诊断

1. 建立候选动作风险指标、支配关系和反例扫描；
2. 扫描 summer-01、summer-02、summer-06 及24关；
3. 在 Admin 提供通俗原因和候选比较；
4. 只根据反例调整评价或候选护栏，不规定“受威胁羊必须逃”；
5. 重跑平衡矩阵和关键棋谱。

### 第三批：声音可靠性与威胁反馈

1. 先修复每次移动、捕食和连吃的可靠播放；
2. 增加开发诊断状态和自动测试；
3. 再加入限频的“新威胁羊叫”；
4. 在手机、桌面、静音和切后台恢复场景人工试听。

## 六、验收标准

- 玩家看到任何终局，都能在首屏说清“谁达成了什么、为什么结束、下一步做什么”。
- 相同战略局面第三次出现必定结束，不因同质羊的内部 ID 不同而漏判。
- 任意生产对局可通过一个复现包在 Admin 重放到反馈动作；活动恢复不会改变后续 AI 随机序列。
- normal/hard 不选择被其他合法动作严格支配的送子动作；合理牺牲有可查看的封锁或机动收益证据。
- summer-01、summer-02、summer-06 都保留至少一条“合理不逃”和一条“疑似送子已阻止”的固定种子/局面测试。
- 每次被规则接受的移动和捕食都有声音请求；样本不可用时立即降级，静音和平台限制除外。
- 音频文件继续通过格式、时长和音量门禁；胜利捕食的两个声音按顺序播放，不互相覆盖。
- 威胁声只在新威胁形成时出现，每次狼行动最多一次；关闭声音不影响理解和操作。
- AI 或重复规则修改后，24关自动矩阵、代表棋谱和Admin摘要全部更新；真实玩家难度仍由后续人工试玩确认。

## 七、需要用户确认的产品选择

推荐默认采用以下三项，确认后再开发：

1. 玩家端把两种 draw 统一称为“僵局，挑战未完成”，但明确显示“达到重复上限”或“达到行动上限”；重复次数以后续正式规则为准；
2. 允许有证据的羊方战术牺牲，normal/hard 禁止无补偿明显送子；
3. 增加“狼移动后新形成威胁”的短羊叫，每次动作最多一次，可随总音效关闭。
