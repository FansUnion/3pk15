# CI 与发布门禁专项（待确认）

> 实施状态（2026-07-18）：已完成。CI 使用冻结锁文件、并发取消、超时和官方当前 Actions 版本；命令定义集中在 `release:check`，GitHub 按 gate ID 展示具体失败阶段。根因是 `web-shared` 只指向未纳入 Git 的 `dist`，本地残留产物掩盖问题；工作区导出已改为 `src/index.ts`。本地移走 dist 的复测与 GitHub Linux 完整门禁均通过，CI #34 全部步骤为绿色。

## 一、问题与目标

用户反馈：每次提交 GitHub 后 CI 总是报错，需要确认 `.github/workflows/ci.yml` 是否有价值、为何失败，以及是否存在同类工程问题。

目标不是让红灯消失，而是建立可信的发布门禁：红灯代表确定问题，绿灯代表玩家端、Admin、共享规则和部署边界达到约定标准；本地检查、GitHub Actions 和两个 Vercel 项目不再各用一套标准。

## 二、已确认事实

### 1. 最近失败位置

- GitHub `CI #30`（提交 `eecf9db`）状态为 Failure，失败步骤为 `Run pnpm test:web`。
- 从下线旧 `apps/web` 的提交 `4afc67c` 开始，到当前连续 7 次均失败在同一步；此前 `45f93a1` 成功。
- `4afc67c` 首次把旧 Web 包的测试迁入 `apps/player-web` 与 `apps/admin`，并首次让 CI 执行 `pnpm test:web`。
- GitHub 同时显示 Actions 内置 Node.js 20 弃用警告；这是 warning，不是本次退出码 1 的直接原因。

### 2. 本地复现结果

在 Windows 本地执行：

```text
pnpm test:web
CI=true pnpm test:web
```

均通过：玩家端 6 个测试文件、19 个测试；Admin 1 个测试文件、2 个测试。由此可确认测试并非在所有环境稳定失败，问题与 GitHub 的 Linux/干净安装环境、资源限制或未显示的具体测试错误有关。

### 3. 根因确认

分阶段门禁将失败定位到 `apps/player-web/lib/active-game.test.ts`。该实现引用 `@wolf-sheep/web-shared`，而包清单此前只声明 `dist/index.js` 和 `dist/index.d.ts`；`dist` 不受 Git 管理。Windows 本地存在旧构建产物，因此测试通过；GitHub 干净检出没有 dist，因此模块解析失败。

修复后，`web-shared` 与 `game-core` 一致，通过 package exports 将工作区类型和默认入口指向 `src/index.ts`。临时移走本地 dist 后再次运行 active-game 测试通过，证明测试不再依赖残留构建产物。

## 三、`ci.yml` 是否有价值

有价值，而且应保留。Vercel 的两个构建分别证明玩家项目或 Admin 项目能部署，但不能替代：

- 核心规则测试和平衡回归。
- 玩家端与 Admin 单元测试。
- 玩家包不包含 Admin、密钥和非目标平台能力。
- 资源引用、皮肤资源和部署契约。
- 两个应用共同依赖的代码是否一致。

但当前 CI 没有充分发挥价值：

- `test:web` 把两个应用合成一步，失败后不能快速定位。
- CI 没有执行后来新增的架构边界、部署契约和产物审计。
- CI 与 `pnpm release:check` 维护两份不同清单，已经漂移。
- `pnpm install` 没显式写 `--frozen-lockfile`。
- 所有文档提交也触发完整构建，没有按改动范围区分成本。
- Actions 版本产生运行时弃用警告，持续制造噪音。

## 四、结构性风险

### 1. 发布门禁存在双重真相

`scripts/release-check.mjs` 已包含 14 项检查，包括应用测试、平台边界、部署契约、资源、两个生产构建和产物审计；CI 只执行其中一部分。以后新增检查时很可能只改一处。

推荐发布门禁只有一个代码事实源：CI 调用 `pnpm release:check`，或让 CI 各步骤调用同一组可复用脚本；不要手工复制命令列表。

### 2. Admin 仍直接依赖玩家端源码

`apps/admin/tsconfig.json` 和 `apps/admin/next.config.ts` 通过相对路径/别名直接读取多个 `apps/player-web` 组件和库。当前边界检查只禁止玩家端引用 Admin，没有禁止 Admin 引用玩家端。

这不等于当前 CI 根因，但带来真实风险：

- Admin 构建依赖另一个应用的目录结构。
- 路径解析、测试配置和 Linux 大小写差异更难控制。
- 玩家改界面可能意外破坏 Admin。
- “公共能力属于哪里”再次变得不清楚。

长期方案是把真正共享的 UI、存储、对局适配和文案能力迁入明确共享包；Admin 依赖共享包，不依赖玩家应用源码。

### 3. 自动门禁偏重技术存在性

当前门禁可验证文件、构建和部分逻辑，却不能发现：

- 页面标题、翻译和 SEO 文案漂移。
- 帮助图例与真实规则不一致。
- 手机布局、溢出、键盘焦点和颜色依赖。
- 皮肤虽能加载但在实战尺寸不可读。

建议形成“自动门禁 + 截图/可访问性检查 + 人工体验验收”三级质量模型。

## 五、推荐修复与收口方案

### 第一阶段：定位当前红灯

1. 将 `test:web` 拆成 `player tests` 和 `admin tests` 两个命名步骤。
2. 开启详细 reporter，失败时保留完整日志；必要时上传测试报告。
3. 在 GitHub Linux 环境固定 Node、pnpm、时区和 CI 环境变量。
4. 使用 `pnpm install --frozen-lockfile`，排除锁文件被隐式修正。
5. 重跑取得具体失败文件后做最小修复，不通过跳过测试或 `continue-on-error` 消除红灯。

### 第二阶段：统一发布门禁

1. 明确 `release:check` 为发布检查唯一清单。
2. CI 调用同一清单，并让失败阶段名称可读。
3. 补入核心 lint/typecheck、皮肤检查，以及现有部署和产物边界检查。
4. 更新官方 Actions 主版本，消除 Node 运行时弃用警告。
5. 设置合理超时和 concurrency，取消同一分支被新提交取代的旧运行。

### 第三阶段：提高效率

- 仅文档变化时运行轻量文档检查，不重复完整生产构建。
- 玩家端变化重点验证玩家构建；Admin 变化重点验证 Admin；核心/共享包变化验证两端。
- `main` 合并或发布前仍执行完整 `release:check`，路径过滤不能降低最终发布标准。

### 第四阶段：架构边界

- 盘点 Admin 对玩家源码的直接引用。
- 按“游戏规则 / Web 通用能力 / 玩家应用 / Admin 应用”重新归属。
- 迁移真正共享代码后，增加双向应用边界检查，禁止应用互相导入源码。

## 六、验收标准

- GitHub 红灯能直接看到是核心、玩家、Admin、构建、资源还是产物边界失败。
- 同一提交本地 `release:check` 与 GitHub 完整门禁结论一致。
- CI 使用锁定依赖，不因干净安装生成锁文件变化。
- 连续提交不会并行浪费构建资源，文档改动不会无意义执行全部构建。
- Vercel 成功不被误当成全部质量通过；CI 成功也不替代真实设备和视觉验收。
- Admin 与玩家应用最终不再直接引用对方源码。

## 七、推荐决策

保留并重构 `ci.yml`。先拆分测试步骤取得真实失败日志，再统一到 `release:check`，最后处理 Admin 对玩家源码的直接依赖。当前不能在没有完整失败日志的情况下声称已确认具体断言根因。
