# 全球化创业思路对照（A / B / C / D）

> 状态：备选归档（2026-07-12）  
> 用途：对照全球独立开发者常见玩法；**不替代**主工作流  
> 主真相源：[`../创业思路.md`](../创业思路.md)（多源可部署底座 · 标准化选品管道）

---

## 与主路线的关系

当前采用的主流程是：**不知道先验需求时，从多源可部署底座（GitHub OSS、Vercel 模板、官方 starter、同类 boilerplate 等）发现候选 → 技术门禁（含冒烟部署）→ 需求验证 → 再二开上线**（可逐步自动化）。

| 对照 | 说明 |
|------|------|
| 与 A / C / D | 那些路线要求先想清需求/人群/先卖；你当前痛点是「不知道什么需求好」，故不作为默认起点 |
| 与 B | 最接近；但主文档强调的是**可重复管道与自动化**，底座不限开源，亦含 Vercel 模板等 |
| 本文角色 | 备选叙事库：日后若主流程验证失败或策略切换，可从这里启用另一套顺序 |

## 总览对照

| 思路 | 一句话 | 默认顺序 | 「需求未知」时是否好用 |
|------|--------|----------|------------------------|
| A | 需求先行 · SEO 微 SaaS | 验需求 → 再做/再借开源或模板 | 差：要求先知道搜什么 |
| B | 开源托管变现 | 找可托管仓库 → 验「自建痛」与付费 → 上线 | 较好：仓库可当需求线索 |
| C | 垂直切口 | 定 ICP → 挖痛点 → 拼技术 | 差：要求先锁定人群 |
| D | 先卖再做 | 落地页/意向 → 再兑现 MVP | 中：可测假说，但仍需先有假说 |
| **主流程** | 多源底座管道 | 发现底座 → Tech → Demand → 上线 | **最好**：把「未知需求」交给管道筛 |

---

## 思路 A：需求先行 · SEO 微 SaaS

**一句话**：先用关键词与竞品证明「有人搜、能排名、有人付钱」，再决定用开源还是模板快速做出来。

| 维度 | 内容 |
|------|------|
| 顺序 | 验需求（Trends / KD / 竞品）→ 定切口 → 找开源或模板加速 → 上线 |
| 卖什么 | 一个很窄的英文搜索意图（长尾词）对应的结果 |
| 获客 | SEO 为主；Product Hunt / 社区为辅 |
| 开源角色 | 加速器，不是起点 |
| 适合谁 | 能接受先调研再动手；怕「做完没人要」 |
| 风险 | 调研慢；纯 SEO 见效慢；需求未知时难启动 |

**全球化参照**：Indie Hackers / Pieter Levels 系「先找有搜索量的小问题，再最快上线」。

---

## 思路 B：开源托管变现（Managed OSS）

**一句话**：挑「能自建但麻烦」的成熟开源，提供一键托管 + 支付 + 英文体验 + 支持，收订阅费。

| 维度 | 内容 |
|------|------|
| 顺序 | 找可托管优质仓库 → 确认自建痛与付费竞品 → 轻量二开上线 |
| 卖什么 | 省事与稳定（便利税），不是「功能一定更强」 |
| 获客 | SEO（hosted / alternative / self-host）+ 对比文 + 社区 |
| 开源角色 | 产品本体；你卖运维与体验层 |
| 适合谁 | 想少研发、快上线；与当前技术栈偏好一致 |
| 风险 | License / 品牌合规要严；易被说成「套壳」，须讲清价值主张 |

**全球化参照**：Managed open source、self-hosted 工具的 hosted 竞品（便利订阅）。

---

## 思路 C：垂直切口 · 开源只是零件

**一句话**：先锁定一个海外垂直人群，再拼开源/API 做成专用工具。

| 维度 | 内容 |
|------|------|
| 顺序 | 定 ICP → 挖痛点 → 拼技术栈（可含开源）→ 上线 |
| 卖什么 | 对某一类人「够专」 |
| 获客 | 垂直社区 + 内容 + SEO |
| 开源角色 | 零件；不追求整个产品 fork 自单一仓库 |
| 适合谁 | 已有某垂直认知（如跨境、创作者、开发者工具链） |
| 风险 | 要先选对人；二开可能多于纯托管 |

**全球化参照**：Vertical micro-SaaS，先人群后功能。

---

## 思路 D：先卖再做（Landing → Waitlist → 兑现）

**一句话**：先做英文落地页与定价，收集邮箱/意向甚至预售；信号够了再 fork 开源上线。

| 维度 | 内容 |
|------|------|
| 顺序 | 落地页与报价 → 流量测试 → 有信号再二开部署 |
| 卖什么 | 承诺的结果；MVP 用开源快速兑现 |
| 获客 | 冷启动（小预算广告、社区、SEO 草稿页） |
| 开源角色 | 验证通过后的兑现手段 |
| 适合谁 | 极度厌恶无效研发；能接受「验证失败则不上线」 |
| 风险 | 仍需先有产品假说；文案与获客能力要求更高 |

**全球化参照**：Validate-first / smoke test landing page。

---

## 何时考虑切换主路线

| 信号 | 可考虑 |
|------|--------|
| 管道筛出大量「技术通但需求全挂」 | 临时加入 A 的关键词池，或 D 的落地页预检 |
| 你已锁定明确人群 | 切换或叠加 C |
| 某候选已过 Tech+Demand，想降研发风险 | 上线前用 D 做小流量意向测试 |
| 候选天然是「自建痛 + 已有 hosted 竞品」 | 叙事上明确采用 B 的「便利税」表述 |

切换时：**先改 [`../创业思路.md`](../创业思路.md) 的工作流顺序与门禁**，再在本文变更记录中记一笔。
