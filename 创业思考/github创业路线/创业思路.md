# 可部署底座选品工作流：多源管道

> 状态：主真相源（2026-07-12）  
> 角色：个人开发者 / 一人公司 · Web 出海 · 新手友好  
> 备选叙事对照：[`备选/全球化创业思路对照-ABCD.md`](./备选/全球化创业思路对照-ABCD.md)  
> 范围：通用选品管道（GEO、OpenRoute 等单产品另案维护）

---

## 1. 一句话定位

在**尚不清楚什么需求最好**时，从**可快速部署的底座**发现候选（GitHub 开源、**Vercel 模板**、官方 starter、同类 boilerplate 等）→ **技术门禁**（含冒烟部署）确认能低成本落地 → **需求门禁**验证市场信号 → 通过后才轻量二开上线。

原则：**来源不重要，统一过同一套门禁**。管道可逐步自动化；人工只做拍板、合规与推广。

---

## 2. 问题定义

| 现状 | 对策 |
|------|------|
| 不知道什么需求好 | 不从「先想一个灵感」开始，从**可落地的底座池**开始 |
| 怕做完没人要 | 技术通过后必须过需求验证，不过则淘汰，不加大自研 |
| 时间精力有限 | 固定门禁、时间盒与白名单；能自动的步骤写成可脚本化规格 |
| 要出海、要低成本 | Web 优先；Vercel 生态；中低竞争切口（KD 原则，阈值待标定） |

**主顺序（已拍板）**：`发现底座 → 技术可落地 → 需求可行 → 二开上线`。  
（若日后改为「需求先行」等，见备选 ABCD，并回写本节。）

---

## 3. 硬边界

### 3.1 做（In Scope）

| 项 | 说明 |
|----|------|
| 多源底座先行 | GitHub OSS、Vercel Templates、官方/生态 starter、同类可一键上 Vercel 的模板均可作为入口 |
| Web 优先 | 工具 / SaaS 优先；游戏与重客户端为低优先 |
| Vercel 友好 | 优先 Next.js、Supabase 及常见 Vercel 生态；一键 Deploy 模板加分 |
| 轻量二开 | 仅白名单内改造（见第 7 节） |
| 需求门禁 | Trends / KD / 流量与竞品等交叉验证；须先有一句话假说 |
| 英文出海 | 面向海外用户推广与变现 |

### 3.2 不做（Out of Scope）

| 项 | 原因 |
|----|------|
| 从零重写 | 违背少研发 |
| 超出二开白名单的大改 | 一人公司不可持续 |
| 未过 Tech 就做 Demand 深研 | 技术落不下则调研浪费 |
| 未过 Demand 就正式推广投入 | 降低无效获客 |
| 未冒烟部署就标 tech_pass | 纸面「能部署」不可信 |
| 未写一句话假说就做 Demand | 模板壳无法抽取有效关键词 |
| 高 KD 红海硬刚 | 新手获客成本不可控 |

## 4. 候选卡（最小字段）

每条候选入库时必须填齐；跨源去重后合并为一条。

| 字段 | 说明 |
|------|------|
| `source_type` | `github_oss` / `vercel_template` / `official_starter` / `other_boilerplate` |
| `name` / `url` | 名称与主入口链接 |
| `one_liner_hypothesis` | 一句话功能/付费假说（Demand 前必填；模板壳尤其重要） |
| `tech_tags` | 如 Next.js、Supabase、Stripe… |
| `license_or_terms` | OSS License，或模板商用/再分发条款摘要 |
| `deploy_hint` | 一键 Deploy、vercel.json、部署文档链接等 |
| `dedupe_key` | 同源不同入口（如 GitHub ↔ 同一 Vercel 模板）合并用 |
| `status` | `discovered` → `tech_pass` / `tech_fail` → `demand_pass` / `demand_fail` → `queued` → `shipped` / `killed` |
| `fail_reason` | 失败时必填（超时、部署挂、无需求信号等） |

---

## 5. 标准工作流（六段门禁）

```text
Discover → Tech 门禁 → Demand 门禁 → Score 排序 → MVP 二开上线 → 复盘止损
```

任一门禁不通过：**不进入下一阶段**；记 `fail_reason`，回到 Discover 取下一条。

### 5.1 Discover（发现底座池）

| 项 | 内容 |
|----|------|
| 输入 | 多源搜索/目录 + 收录模板（见下来源白名单） |
| 动作 | 按来源拉取或手工导入；填候选卡；按 `dedupe_key` 去重 |
| 通过标准 | 候选卡必填字段完整（假说可在进 Demand 前补全，但建议收录时就写草稿） |
| 可否自动化 | **是**（GitHub API、模板目录/清单导入、定时任务）；目录站可半自动 |

**来源白名单**：

1. GitHub 开源仓库  
2. **Vercel Templates / Marketplace 模板**  
3. Next.js / Supabase 等官方或生态 starter  
4. 其他同类：可一键部署到 Vercel（或导出同构 Next 项目）的 boilerplate / 商业模板  

**质量信号（按源）**：

| 来源 | 质量信号 |
|------|----------|
| `github_oss` | 星数（阈值待标定）、近 N 个月提交、文档、License |
| `vercel_template` / `official_starter` | 官方或维护方更新、一键 Deploy 可用、说明完整（无 Star 不扣分） |
| `other_boilerplate` | 维护活跃、商用条款清晰、与 Vercel 兼容 |

**品类信号**：工具、dashboard、CMS、analytics、AI chat 等 Web 向；游戏默认降权。

### 5.2 Tech 门禁（技术可落地）

| 项 | 内容 |
|----|------|
| 输入 | 单条候选卡 |
| 动作 | 静态可部署检查 + **冒烟部署** + License/条款 + 二开是否落白名单 |
| 通过标准 | **冒烟部署成功**（本地或 Vercel 上可打开首页/主流程）；二开落白名单；条款允许目标商业形态；且在时间盒内完成 |
| 可否自动化 | 静态检查大部分可自动；冒烟部署可半自动；条款灰色地带人工 |

**硬门禁：冒烟部署**

- 必须真实跑通一次（`vercel deploy` / 一键 Deploy / 本地 `dev` 能打开主界面）  
- **禁止**仅凭「看起来像 Next.js」就标 `tech_pass`  
- 时间盒内卡壳无法跑通 → `tech_fail`

**按 `source_type` 的条款检查**：

| 类型 | 看什么 |
|------|--------|
| `github_oss` | License 是否允许商用托管与品牌化再分发；AGPL 等标风险 |
| 模板 / starter / boilerplate | 商用条款、是否禁止转售「未改模板」、是否允许去掉原品牌 |

**加分 / 降权**：

- 加分：Next.js、已有 Vercel 一键 Deploy、Supabase/Auth 现成、英文说明  
- 降权：重型 Docker/K8s、长期自建 GPU/队列、无文档、禁止商用或转售  

不通过 → `tech_fail`。

### 5.3 Demand 门禁（需求可行）

| 项 | 内容 |
|----|------|
| 输入 | `tech_pass` 候选 + **已填写的一句话假说** + 从描述/竞品名抽出的关键词 |
| 动作 | Trends、KD、Ahrefs/SimilarWeb、付费竞品与社区信号交叉检查 |
| 通过标准 | 至少 1 个正信号（搜索意图 / 付费竞品 / 强社区讨论）；KD 非明显红海（阈值待标定） |
| 可否自动化 | **大部分是**；叙事是否站得住半自动/人工 |

**关键词来源**：README / 模板商店描述 / 竞品名 / **一句话假说**。  
模板若只是 UI 壳：**必须先写假说再查**；无假说不得进入 Demand。

**必做信号**：Google Trends；KD；Ahrefs / SimilarWeb 等。  
**辅助信号**：Reddit / Indie Hackers / Product Hunt；付费竞品；Issue/评论中的痛点。

不通过 → `demand_fail`（技术通也不能正式推广投入）。

### 5.4 Score（评分排序）

| 项 | 内容 |
|----|------|
| 输入 | `tech_pass` + `demand_pass` |
| 动作 | 多维打分 + 一票否决 → 「可动手队列」 |
| 通过标准 | 进入队列前 N 名（N 待标定）；无否决项 → `queued` |
| 可否自动化 | **是**（规则引擎）；权重表见后续「筛选评分表」 |

| 维度 | 看什么 |
|------|--------|
| 需求匹配 | 假说是否清晰；是否像愿为省事/结果付费 |
| 市场信号 | 搜索、KD、付费竞品 |
| 底座质量 | 按 `source_type`：OSS 看活跃/文档；模板看维护/一键 Deploy/条款 |
| 可部署性 | Vercel 友好与冒烟结果 |
| 二开工作量 | 是否落在白名单 |
| 运维成本 | 一人能否扛住 |
| 变现路径 | 订阅 / 买断 / 增值；海外收款可行性 |

**一票否决**：冒烟失败或无法低成本部署、二开接近重做、条款不合规、KD 明显过高且无切口、Demand 无正信号。

### 5.5 MVP（轻量二开上线）

| 项 | 内容 |
|----|------|
| 输入 | 队列头部 `queued` 候选 |
| 动作 | 仅白名单改造 → Vercel 部署 → 域名 → 英文最小可推广页 |
| 通过标准 | 对外可访问；支付或等价变现路径可测（若本轮需要）；卖点一句话说清；**已品牌化**（禁止未改模板皮肤直接当成品推广） |
| 可否自动化 | 部署部分可辅助；品牌文案与合规 **人工** |

### 5.6 复盘止损

| 项 | 内容 |
|----|------|
| 输入 | 上线后可观测信号（访问、注册、意向、付费等） |
| 动作 | 加深 / 维持观察 / 止损换下一个（`killed`） |
| 通过标准 | 见第 8 节；未达则换候选，不默认加自研 |
| 节奏 | **同时在办候选 ≤ 3**；每周至少推进 1 条过 Tech，或止损 1 条 |
| 可否自动化 | 数据采集可自动；决策 **人工** |

## 6. 可行性护栏（防空转）

| 护栏 | 规则 |
|------|------|
| Discover 时间盒 | 单条收录 ≤ 15 分钟 |
| Tech 时间盒 | ≤ 4 小时（含冒烟）；超时记 `tech_fail` 并换下一条 |
| Demand 时间盒 | ≤ 半天；超时记 `demand_fail` 或降为待补信号，不阻塞整条管道 |
| Demand 最低通过 | ≥ 1 个正信号，且 KD 非明显红海 |
| 禁止项 | 未冒烟部署标 `tech_pass`；未写一句话假说做 Demand；同时在办 > 3 |

未在时间盒内完成 = **失败并记录原因**，不是「改天再磨」。

---

## 7. 二开白名单

**允许（默认）**：

- 支付（如 Stripe）与基础套餐门禁  
- 品牌化（名称、域名、Logo、主题）— **模板类必做**  
- i18n / 英文默认体验  
- 落地页与基础 SEO 元信息  
- 隐私政策、服务条款、Key/数据安全说明等基础合规文案  

**默认淘汰（超出即 `scope_fail`）**：

- 重做核心业务逻辑或换架构  
- 自建重量级基础设施（非 Vercel 友好）  
- 为过 Demand 而堆大量新功能  

---

## 8. 成功标准与止损

| 标准 | 含义 |
|------|------|
| 技术通 | 冒烟部署通过且能稳定对外访问 |
| 需求信号通 | Demand 门禁通过（原则级；数字阈值待标定） |
| 成本可控 | 时间与金钱在一人公司可接受范围（含时间盒） |
| 初步市场信号 | 上线后出现可观测兴趣（访问/注册/意向/付费任一），用于「是否加深」 |

未达到：缩小切口、换关键词、或 **换队列下一条**；禁止用「再开发三个月」作为默认反应。

## 9. 可自动化规格（本阶段只定规格，不写代码）

目标：把 Discover → Tech → Demand → Score 做成可定时跑的管道；人工消费「可动手队列」。

| 阶段 | 建议自动化方式 | 人工保留 |
|------|----------------|----------|
| Discover | GitHub API + **Vercel/官方模板目录或手工清单导入** + 去重入库 | 调整来源与查询集 |
| Tech | 静态仓检 + 触发预览部署/冒烟脚本 | 条款灰色地带、架构争议 |
| Demand | 假说/描述关键词抽取 + Trends/KD/竞品 API | 叙事是否站得住 |
| Score | 权重表打分与否决 | 最终是否开工 |
| MVP / 推广 | CI 部署辅助 | 品牌、合规、投放与内容 |

后续可单开文档：「自动化脚本 / Agent 执行规格」（输入 schema、输出字段、失败重试）。

## 10. 品类与技术栈偏好（管道默认过滤器）

| 偏好 | 说明 |
|------|------|
| 品类 | Web 工具 / SaaS 优先；垂直服务次之；游戏低优先 |
| 前端/全栈 | 最好 Next.js |
| 数据/后端 | 最好 Supabase 或同类 serverless 友好方案 |
| 托管 | Vercel；**一键模板加分**；技术栈常见、可一人维护 |
| 上线闭环 | 底座可部署 + 白名单改造（含品牌化）+ 1 个域名 → 可开始推广 |
